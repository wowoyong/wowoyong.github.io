## 핵심 요약

데이터베이스 학습의 중심은 데이터를 안전하게 저장하고, 필요한 순간에 빠르게 꺼내 쓰는 구조를 이해하는 데 있습니다. 관계형 모델, 정규화, 인덱스, 트랜잭션은 학부 과정에서 가장 자주 평가되는 핵심 축이며 서로 강하게 연결됩니다. 실제 적용에서는 이론만 아는 것보다 조회 패턴을 기준으로 스키마를 조정하고, 병목을 측정해 개선하는 해석 능력이 더 중요합니다. 또한 단일 서버 중심 설계에서 복제·분산 구조로 넘어갈 때 일관성과 지연시간의 교환관계를 읽는 관점이 필요합니다. 최근 흐름은 인공지능 연계 검색, 실시간 분석, 자동 튜닝처럼 운영 자동화와 데이터 활용 범위를 넓히는 방향으로 발전하고 있습니다.

## 관계형 모델과 정규화 설계 포인트

- 관계형 모델은 행과 열로 데이터를 표현해 조인으로 의미를 연결하며, 예를 들어 학생 표와 수강 표를 분리하면 중복 입력을 줄일 수 있습니다.
- 제일정규형은 한 칸에 한 값만 두는 원칙으로, 전화번호 여러 개를 한 칸에 넣는 방식보다 검색과 수정이 훨씬 단순해집니다.
- 제이정규형은 부분 함수 종속을 제거해 복합 기본키 테이블의 중복을 줄이며, 과목명 반복 저장 같은 문제를 예방합니다.
- 제삼정규형은 이행 함수 종속을 제거해 변경 이상을 줄이고, 학과명 변경 시 여러 표를 동시에 고치는 위험을 낮춥니다.
- 정규화는 쓰기 무결성에 강하지만 조회 조인이 늘 수 있어, 통계 화면처럼 읽기 위주 업무에서는 일부 반정규화를 비교 검토합니다.
- 실무에서는 정규형을 출발점으로 잡고 실제 질의 로그를 본 뒤 컬럼 통합·분리를 결정하는 단계적 접근이 안정적입니다.
- 정규화 수준은 정답이 하나가 아니라 업무 특성에 따라 달라지므로, 갱신 빈도와 조회 지연시간을 함께 기준으로 잡아야 합니다.

## 인덱스 구조와 질의 성능 해석

- 인덱스는 책의 색인처럼 원하는 행 위치를 빠르게 찾게 하며, 주문일 조건 검색에서 전수 검사보다 응답시간을 크게 줄일 수 있습니다.
- 선두 컬럼 규칙은 복합 인덱스 활용의 핵심으로, 예를 들어 고객번호·주문일 인덱스는 주문일 단독 조건보다 고객번호 포함 조건에 유리합니다.
- 선택도가 낮은 컬럼 인덱스는 효과가 작아, 성별처럼 값 종류가 적은 컬럼은 다른 컬럼과 결합할 때 성능 이점이 커집니다.
- 커버링 인덱스는 필요한 컬럼을 인덱스에 포함해 본문 접근을 줄이며, 목록 화면 조회에서 입출력 횟수를 줄이는 데 실용적입니다.
- 인덱스가 많아지면 조회는 빨라질 수 있지만 삽입·수정·삭제 비용이 증가하므로 쓰기 비중 높은 서비스는 균형 설계가 필요합니다.
- 실행 계획 분석은 느린 질의 개선의 출발점이며, 예상 행 수와 실제 행 수 차이를 보면 통계 정보 갱신 필요성을 판단할 수 있습니다.
- 실무 최적화는 감으로 하지 않고 느린 질의 목록, 입출력 지표, 잠금 대기 시간을 함께 보며 원인을 분리하는 방식이 효과적입니다.

## 트랜잭션과 동시성 제어 실전 이해

- 트랜잭션은 모두 성공하거나 모두 취소되어야 하는 작업 묶음으로, 계좌 이체에서 출금만 반영되는 오류를 막습니다.
- 원자성·일관성·격리성·지속성 원칙은 장애 상황 신뢰성을 보장하며, 특히 결제 시스템에서는 지속성 확인이 핵심입니다.
- 격리 수준이 높을수록 정합성은 좋아지지만 동시 처리량이 줄 수 있어, 조회 중심 업무와 갱신 중심 업무를 구분해 선택합니다.
- 더티 리드·반복 불가능 읽기·유령 읽기는 격리 수준 비교의 대표 사례로, 시험에서는 현상과 해결 수준을 함께 묻는 경우가 많습니다.
- 잠금 기반 제어는 이해가 직관적이지만 대기가 길어질 수 있고, 다중 버전 기반 제어는 읽기 성능에 강한 대신 저장공간 부담이 늘 수 있습니다.
- 데드락은 서로 잠금 해제를 기다리는 상태로, 잠금 순서 통일과 짧은 트랜잭션 유지가 실무 예방책으로 자주 쓰입니다.
- 실무 해석에서는 오류 재시도 정책까지 트랜잭션 설계에 포함해야 하며, 특히 분산 환경에서는 부분 실패를 기본 가정으로 둡니다.

## 데이터 모델링과 스키마 진화 전략

- 개념·논리·물리 모델을 구분하면 요구사항 변화 대응이 쉬워지며, 먼저 업무 용어를 정리한 뒤 표 구조로 내리는 흐름이 안정적입니다.
- 기본키 선택은 성능과 유지보수에 직접 영향이 있어, 자연키와 대리키를 비교해 변경 가능성이 높은 값은 기본키로 피하는 편이 안전합니다.
- 참조 무결성 제약은 데이터 고아 문제를 막지만 삭제 제약이 생길 수 있어, 업무 규칙에 맞는 삭제 정책을 함께 설계해야 합니다.
- 스키마 변경은 무중단 배포 관점이 중요하며, 컬럼 추가 후 이중 쓰기와 점진 전환을 적용하면 서비스 중단 위험을 줄일 수 있습니다.
- 시간 이력 모델은 감사·분석에 유리해, 가격 변동 추적처럼 과거 상태 복원이 필요한 업무에서 큰 가치를 냅니다.
- 명명 규칙 통일은 협업 비용을 줄이는 실무 요소로, 테이블·컬럼 접두 규칙만 맞춰도 질의 가독성이 크게 좋아집니다.
- 모델링 품질은 문서보다 검증 시나리오로 확인해야 하며, 실제 조회·집계 질의를 미리 작성해보면 구조 결함을 빨리 발견할 수 있습니다.

## 복제·분할·샤딩으로 확장하기

- 복제는 같은 데이터를 여러 서버에 두는 방식으로, 읽기 부하 분산과 장애 복구 시간을 줄이는 데 실무적으로 널리 쓰입니다.
- 동기 복제는 일관성에 강하지만 지연시간이 늘 수 있고, 비동기 복제는 성능이 좋지만 최신 데이터 지연을 감수해야 합니다.
- 수평 분할은 행 기준으로 데이터를 나누어 저장해 대용량 처리에 유리하며, 사용자 번호 범위 분할이 대표 예시입니다.
- 샤딩 키 선택이 잘못되면 특정 샤드만 과부하되는 쏠림이 생겨, 균등 분포와 질의 패턴을 함께 고려해야 합니다.
- 전역 조인은 분산 환경에서 비용이 커서, 서비스 경계별 데이터 소유 원칙을 정하면 복잡도를 줄일 수 있습니다.
- 장애 대비 설계에서는 자동 장애 전환만 믿지 말고 복구 리허설을 주기적으로 수행해야 실제 사고 대응 속도가 올라갑니다.
- 확장 전략은 처음부터 과설계하기보다 병목 지표가 확인될 때 단계적으로 적용하는 편이 비용 대비 효과가 큽니다.

## 관계형과 비관계형 선택 기준

- 관계형은 강한 정합성과 복잡한 조인에 유리해 금융·회계처럼 규칙이 엄격한 도메인에서 안정성이 높습니다.
- 문서형 저장소는 유연한 구조 변경에 강해 콘텐츠 메타데이터처럼 속성이 자주 바뀌는 업무에서 개발 속도를 높입니다.
- 키값 저장소는 단순 조회 지연시간이 매우 짧아 세션 저장이나 캐시 계층에서 실무 효율이 큽니다.
- 열 지향 계열은 대규모 집계 분석에 유리해 로그 분석·통계 대시보드처럼 읽기 집약 업무에서 장점이 뚜렷합니다.
- 그래프 계열은 관계 탐색이 핵심인 추천·사기 탐지에서 강점을 보이며, 다단계 연결 질의가 많은 경우 비교 우위가 큽니다.
- 하나의 저장소로 모든 문제를 해결하려 하기보다 업무별 다중 저장소 전략을 쓰면 성능과 개발 생산성을 함께 잡기 쉽습니다.
- 선택 기준은 유행보다 일관성 요구, 조회 패턴, 운영 인력 역량을 우선으로 두는 것이 장기 유지보수에 유리합니다.

## 최신 흐름

- 생성형 인공지능 연계 검색을 위해 벡터 기반 검색 기능을 기존 데이터베이스와 함께 운용하는 사례가 빠르게 늘고 있습니다.
- 실시간 분석 요구 증가로 행 단위 처리와 열 단위 집계를 결합한 혼합 처리 구조 채택이 확대되고 있습니다.
- 자동 인덱스 추천과 자동 질의 최적화 기능이 고도화되어 초급 운영자도 기본 성능을 안정적으로 확보하기 쉬워졌습니다.
- 무중단 스키마 변경 도구가 성숙해지며 대규모 서비스에서도 배포 중 데이터 구조 전환이 표준 절차로 자리잡고 있습니다.
- 클라우드 관리형 데이터베이스 중심 운영이 강화되면서 백업·복구·모니터링 자동화가 기본 기대치가 되었습니다.
- 비용 최적화 측면에서 저장계층 분리와 수명주기 정책을 통한 차등 저장 전략이 실무 핵심 과제로 부상했습니다.
- 개인정보 보호 규제 대응을 위해 마스킹·암호화·접근통제 감사 기능을 데이터 계층에서 기본 제공하는 흐름이 강해졌습니다.
- 다중 지역 배포 환경에서 지연시간을 줄이기 위한 지역 근접 읽기와 충돌 해결 정책 설계가 중요도가 커졌습니다.
- 관찰 가능성 강화 흐름으로 질의 추적, 잠금 대기 시각화, 원인 분석 자동 보고 기능이 운영 표준에 가까워지고 있습니다.
- 데이터 품질 관리가 데이터베이스 운영과 통합되며 중복·결측·이상치 탐지를 배치가 아닌 상시 점검으로 전환하는 추세입니다.
