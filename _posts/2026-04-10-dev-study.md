## 핵심 요약

운영체제는 하드웨어와 응용 프로그램 사이에서 자원을 배분하고 실행을 조정하는 핵심 소프트웨어다. 학부 수준에서는 프로세스와 스레드, 중앙처리장치 스케줄링, 메모리 관리, 파일 시스템, 입출력, 동기화와 교착상태를 함께 이해해야 전체 그림을 잡을 수 있다. 실제 환경에서는 응답 시간이 중요한 모바일 앱, 처리량이 중요한 서버, 안정성이 중요한 임베디드 장치가 서로 다른 운영체제 설계를 요구한다는 점이 중요하다. 예를 들어 가상 메모리와 캐시는 프로그램 실행을 쉽게 해 주지만, 문맥 교환과 잠금 경쟁은 성능 저하의 원인이 되기도 한다. 최근에는 메모리 안전성 강화, 경량 가상화, 기밀 컴퓨팅, 이기종 메모리 지원처럼 보안과 성능을 함께 다루는 방향으로 운영체제가 발전하고 있다.

## 프로세스와 스레드의 실행 단위 차이 이해하기

- 프로세스는 독립된 주소 공간을 가진 실행 단위이고, 스레드는 같은 프로세스 안에서 자원을 공유하는 실행 흐름이다. 그래서 웹브라우저는 탭을 프로세스로 나누고, 탭 내부 작업은 여러 스레드로 처리하는 경우가 많다.
- 프로세스 제어 블록은 프로세스 상태, 프로그램 카운터, 레지스터 값 같은 실행 정보를 저장하며, 운영체제는 이를 바탕으로 중단된 작업을 다시 이어서 실행한다.
- 스레드는 생성 비용과 문맥 교환 비용이 비교적 작아 채팅 앱에서 화면 처리와 네트워크 수신을 동시에 수행하기에 적합하지만, 메모리를 공유하므로 경쟁 상태가 생기기 쉽다.
- 프로세스 상태는 생성, 준비, 실행, 대기, 종료로 나뉜다. 예를 들어 파일 읽기를 요청한 프로그램이 입출력 완료를 기다릴 때 대기 상태로 이동한다고 이해하면 쉽다.
- 문맥 교환은 현재 작업의 레지스터와 상태를 저장하고 다른 작업으로 전환하는 과정이다. 멀티태스킹을 가능하게 하지만, 너무 자주 일어나면 실제 계산보다 전환 비용이 더 커질 수 있다.
- 사용자 수준 스레드는 전환이 빠르지만 커널이 직접 스케줄링하지 못하는 한계가 있고, 커널 수준 스레드는 관리 부담이 크지만 다중 코어 활용과 차단 처리에 유리하다.
- 단일 스레드 프로그램은 디버깅이 단순한 대신 응답성이 떨어질 수 있고, 다중 스레드 프로그램은 반응성은 좋지만 동기화 실수가 버그로 이어지기 쉬워 실무에서는 설계 기준이 중요하다.

## 중앙처리장치 스케줄링과 응답 시간의 관계 파악하기

- 스케줄러는 어떤 프로세스에 중앙처리장치를 할당할지 결정하며, 목표는 공정성, 처리량, 응답 시간, 대기 시간 사이에서 균형을 맞추는 것이다.
- 선입선출 방식은 구현이 단순하지만 긴 작업 하나가 짧은 작업들을 오래 막을 수 있어, 과제 채점 서버처럼 작업 길이가 제각각인 환경에서는 비효율적일 수 있다.
- 최단 작업 우선 방식은 평균 대기 시간을 줄이는 데 강점이 있지만 실행 시간을 미리 알기 어렵고, 긴 작업이 계속 밀릴 수 있어 실제 적용에는 제약이 있다.
- 라운드 로빈 방식은 일정한 시간 조각마다 작업을 번갈아 실행해 대화형 시스템에 적합하며, 스마트폰 화면 반응성이 중요한 이유를 설명할 때 자주 활용된다.
- 우선순위 스케줄링은 중요한 작업을 먼저 처리하지만 낮은 우선순위 작업이 굶주릴 수 있어, 오래 기다린 작업의 우선순위를 높이는 에이징 같은 보완책이 필요하다.
- 다단계 피드백 큐는 짧고 상호작용이 많은 작업을 우대하고 긴 작업은 뒤로 보내는데, 데스크톱 운영체제가 사용자 체감 반응성을 높이는 대표적인 방식으로 볼 수 있다.
- 실시간 시스템에서는 평균 성능보다 마감 시간 준수가 중요하므로, 일반 데스크톱 스케줄링과 달리 예측 가능성이 핵심 평가 기준이 된다.

## 가상 메모리와 페이지 교체를 실제 성능과 연결하기

- 가상 메모리는 각 프로세스에 연속적이고 큰 메모리가 있는 것처럼 보이게 하는 기법으로, 실제 물리 메모리보다 큰 프로그램도 실행할 수 있게 해 준다.
- 페이지는 가상 메모리를 일정한 크기로 나눈 단위이며, 필요한 페이지만 적재하는 요구 페이징 덕분에 대형 프로그램도 처음부터 전부 메모리에 올릴 필요가 없다.
- 페이지 부재는 필요한 페이지가 메모리에 없을 때 발생하는데, 처음 앱을 실행할 때 로딩이 느려지는 현상을 이 개념으로 설명할 수 있다.
- 페이지 테이블은 가상 주소를 물리 주소로 바꾸는 자료구조이고, 변환 색인 버퍼는 자주 쓰는 변환 결과를 저장해 주소 변환 비용을 줄인다.
- 선입선출 페이지 교체 방식은 오래된 페이지를 내보내지만 자주 쓰는 페이지를 잘못 제거할 수 있고, 최근에 사용하지 않은 페이지를 고르는 방식은 실제 접근 패턴을 더 잘 반영한다.
- 지역성은 프로그램이 가까운 시간과 공간의 데이터를 반복해서 참조하는 성질로, 반복문에서 배열에 순차적으로 접근할 때 캐시와 페이지 성능이 좋아지는 이유가 된다.
- 과도한 페이지 부재로 대부분의 시간을 교체에 쓰는 스래싱이 발생하면 중앙처리장치 사용률이 오히려 떨어지므로, 실무에서는 다중 프로그램 정도를 제한하거나 작업 집합을 관리한다.

## 동기화와 교착상태를 안전하게 다루는 방법

- 동기화는 여러 실행 흐름이 공유 자원을 올바른 순서로 사용하도록 만드는 기법이며, 은행 계좌 잔액을 동시에 갱신할 때 값이 어긋나지 않게 하는 상황을 떠올리면 이해하기 쉽다.
- 임계 구역은 한 번에 하나의 실행 흐름만 들어가야 하는 코드 구간으로, 재고 수량 감소나 방문자 수 증가처럼 읽기와 쓰기가 함께 일어나는 부분이 대표적이다.
- 뮤텍스는 상호 배제를 위한 잠금이고 세마포어는 허용 가능한 접근 수를 조절하는 도구이므로, 프린터 한 대를 보호할 때는 뮤텍스가, 연결 수를 제한할 때는 세마포어가 더 자연스럽다.
- 모니터는 잠금과 조건 변수를 구조적으로 묶어 주므로, 단순한 잠금을 무분별하게 사용하는 코드보다 생산자 소비자 문제를 더 읽기 쉽게 구현할 수 있다.
- 교착상태는 서로가 가진 자원을 기다리며 영원히 멈추는 상태인데, 두 스레드가 잠금 두 개를 서로 반대 순서로 잡을 때 흔히 발생한다.
- 교착상태의 네 가지 조건은 상호 배제, 점유와 대기, 비선점, 환형 대기이며, 잠금 순서를 통일하면 적어도 환형 대기를 끊는 효과를 얻을 수 있다.
- 실무에서는 완전한 예방보다 탐지와 복구, 시간 제한, 잠금 순서 규칙 같은 현실적인 전략을 많이 사용하며, 데이터베이스 트랜잭션과 운영체제 잠금 관리는 비슷한 사고방식을 공유한다.

## 파일 시스템 구조와 저장장치 추상화 이해하기

- 파일 시스템은 데이터를 파일과 디렉터리 형태로 조직해 주는 계층으로, 사용자는 블록 장치의 실제 배치를 몰라도 문서와 사진을 다룰 수 있다.
- 아이노드 같은 메타데이터 구조는 파일 크기, 권한, 위치 정보를 저장하며, 파일 이름과 실제 데이터 위치를 분리해 관리 효율을 높인다.
- 연속 할당은 순차 읽기 성능이 좋지만 외부 단편화에 약하고, 연결 할당은 확장에 유리하지만 임의 접근이 느려 저장 방식 선택의 장단점을 잘 보여 준다.
- 저널링 파일 시스템은 변경 내용을 먼저 기록해 두어 갑작스러운 전원 차단 뒤에도 복구 가능성을 높이는데, 노트북이 강제로 종료된 뒤 손상이 줄어드는 이유와 연결된다.
- 버퍼 캐시는 디스크 접근을 줄여 성능을 높이며, 같은 파일을 반복해서 열 때 두 번째가 훨씬 빠른 현상은 운영체제가 데이터를 메모리에 남겨 두기 때문이다.
- 디스크 스케줄링은 헤드 이동을 줄이기 위한 전통적인 기법이 중요했지만, 반도체 저장장치에서는 지연 특성이 달라져 운영체제의 최적화 지점도 함께 바뀐다.
- 파일 권한과 접근 제어는 다중 사용자 환경의 기본 보안 장치로, 서버에서 로그 파일을 누구나 수정할 수 있게 두면 운영 문제와 보안 문제가 동시에 생길 수 있다.

## 시스템 호출, 입출력, 사용자 모드와 커널 모드 연결하기

- 시스템 호출은 사용자 프로그램이 운영체제 기능을 요청하는 공식 통로로, 파일 열기나 프로세스 생성 같은 작업은 직접 하드웨어를 다루지 않고 이 경로를 거친다.
- 사용자 모드와 커널 모드의 분리는 안전성을 위한 장치이며, 일반 프로그램이 마음대로 메모리 보호 설정을 바꾸지 못하게 막아 시스템 전체를 보호한다.
- 트랩은 예외나 시스템 호출로 인해 커널로 제어가 넘어가는 사건인데, 0으로 나누기 오류와 파일 읽기 요청이 모두 운영체제 개입을 유도한다는 점에서 공통점이 있다.
- 동기식 입출력은 결과가 올 때까지 기다리므로 구현이 단순하지만, 대용량 파일 전송 서버에서는 비동기식 입출력이 더 높은 동시성을 제공할 수 있다.
- 장치 드라이버는 운영체제와 하드웨어 사이의 번역 계층으로, 같은 파일 읽기 명령도 저장장치 종류에 따라 서로 다른 제어 방식으로 전달된다.
- 인터럽트는 장치가 작업 완료나 이벤트를 중앙처리장치에 알리는 장치이며, 키보드 입력이나 네트워크 패킷 도착을 즉시 처리하게 만들어 불필요한 대기를 줄인다.
- 실무에서는 시스템 호출 횟수 자체가 성능에 영향을 주므로, 작은 읽기를 여러 번 수행하는 것보다 큰 버퍼로 한 번 읽는 편이 유리한 경우가 많아 응용 프로그램 설계와 운영체제 이해가 함께 필요하다.

## 최신 흐름

- 메모리 안전성을 높이기 위해 운영체제 핵심 구성 요소 일부를 Rust로 작성하려는 흐름이 이어지고 있으며, 리눅스 커널은 이미 관련 문서와 개발 절차를 공식적으로 유지하고 있다.
- 기밀 컴퓨팅이 확산되면서 운영체제는 단순한 격리를 넘어서 실행 중인 데이터 보호와 원격 인증까지 함께 고려하는 방향으로 발전하고 있다.
- 경량 가상화와 마이크로 가상머신 기술이 서버리스와 격리 실행 환경에서 중요해지면서, 전통적인 가상머신보다 빠른 시작 시간과 작은 오버헤드가 강조되고 있다.
- 웹어셈블리를 운영체제 수준의 샌드박싱 도구처럼 활용하려는 흐름이 강해지면서, 작은 기능 단위를 더 안전하게 배포하고 실행하려는 시도가 늘고 있다.
- 이식 가능한 동적 계측 기술인 확장 버클리 패킷 필터는 네트워크 관찰을 넘어 보안 정책과 성능 분석까지 활용 범위가 넓어져, 운영체제가 실행 중에 더 많이 관찰되고 조정되고 있다.
- 입출력 경로를 최적화하기 위해 현대 리눅스 계열에서는 저오버헤드 비동기 입출력 인터페이스 활용이 늘고 있어, 고성능 서버 프로그램의 구조도 함께 바뀌고 있다.
- 컴퓨트 익스프레스 링크 같은 이기종 메모리 기술이 확장되면서 운영체제는 단순한 주기억장치 관리자를 넘어 여러 종류의 메모리를 정책적으로 배치하는 계층이 되고 있다.
- 에너지 효율이 성능만큼 중요한 지표가 되면서 모바일과 데이터센터 모두에서 스케줄링, 전력 상태 제어, 작업 배치 정책이 더욱 정교해지고 있다.
- 인공지능 추론을 단말과 가장자리 환경에서 직접 수행하려는 수요가 커지면서 운영체제는 가속기, 메모리 대역폭, 격리 정책을 함께 조정하는 역할을 더 강하게 맡고 있다.
- 결과적으로 최근 운영체제 연구와 실무는 빠름, 안전함, 관찰 가능함을 동시에 요구받고 있으며, 학부 수준의 스케줄링과 메모리 관리 개념이 최신 기술의 기반으로 그대로 이어지고 있다.
