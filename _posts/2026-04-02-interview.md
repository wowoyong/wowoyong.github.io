## 오늘의 면접 주제
> React, Next.js, App Router 흐름과 함께 렌더링 방식의 차이가 다시 핵심 면접 주제가 되었기 때문입니다.

## 핵심 한 줄 요약
CSR, SSR, SSG는 "언제 HTML을 만들고 어디서 그리느냐"의 차이이며, 성능과 SEO, 개발 복잡도의 균형을 보고 선택해야 합니다.

## 면접 질문 & 모범 답변

### Q1. CSR, SSR, SSG는 각각 무엇인가요?
**기본 답변** (2-3문장): CSR은 브라우저에서 JavaScript가 실행된 뒤 화면을 그리는 방식입니다. SSR은 요청이 올 때마다 서버가 HTML을 만들어 내려줍니다. SSG는 빌드 시점에 미리 HTML을 만들어 두고 정적으로 제공합니다.  
**심화 포인트**: 핵심은 "렌더링 위치"보다 "HTML 생성 시점"입니다. CSR은 주로 클라이언트 실행 후, SSR은 요청 시점, SSG는 배포 전 빌드 시점에 만들어집니다.

### Q2. CSR은 어떻게 동작하나요?
**기본 답변** (2-3문장): 처음에는 비어 있거나 최소한의 HTML만 내려옵니다. 이후 브라우저가 JS 번들을 받아 실행하고, API 호출 결과로 화면을 완성합니다. 그래서 초기 화면 표시가 늦어질 수 있습니다.  
**심화 포인트**: 사용자는 흰 화면을 먼저 볼 수 있습니다. 검색 엔진도 JS 실행에 의존해야 해서 SEO가 불리할 수 있습니다.

### Q3. SSR과 SSG는 어떤 기준으로 선택하나요?
**기본 답변** (2-3문장): 데이터가 사용자마다 다르거나 매 요청마다 바뀌면 SSR이 적합합니다. 자주 바뀌지 않는 소개 페이지, 블로그, 문서 페이지는 SSG가 효율적입니다. 즉, "실시간성"이 높으면 SSR, "캐시 가능성"이 높으면 SSG입니다.  
**심화 포인트**: 면접에서는 성능만 말하면 부족합니다. 운영 비용과 캐시 전략까지 같이 말해야 답변이 살아납니다.

### Q4. SEO 관점에서 왜 SSR이나 SSG가 유리한가요?
**기본 답변** (2-3문장): 검색 엔진은 HTML 본문에 내용이 바로 있는 페이지를 더 쉽게 읽습니다. SSR과 SSG는 초기 응답에 완성된 HTML이 포함되므로 크롤러가 내용을 빠르게 이해할 수 있습니다. 특히 콘텐츠 중심 서비스에서 유리합니다.  
**심화 포인트**: SEO는 렌더링 방식 하나로 끝나지 않습니다. 메타 태그, canonical, 응답 속도, 내부 링크 구조도 함께 중요합니다.

### Q5. SSR의 단점은 무엇인가요?
**기본 답변** (2-3문장): 요청마다 서버가 HTML을 만들어야 하므로 서버 부하가 커집니다. 트래픽이 많아지면 응답 속도가 흔들릴 수 있습니다. 캐시 설계를 잘못하면 CSR보다 더 비싸고 느려질 수도 있습니다.  
**심화 포인트**: SSR은 "무조건 빠르다"가 아닙니다. 첫 HTML은 빠를 수 있지만, 서버 계산과 데이터 fetch가 길면 TTFB가 늘어납니다.

### Q6. SSG의 단점은 무엇인가요?
**기본 답변** (2-3문장): 빌드 시점 기준으로 페이지가 만들어지므로 최신 데이터 반영이 늦을 수 있습니다. 페이지 수가 많으면 빌드 시간도 길어집니다. 실시간성이 필요한 서비스에는 맞지 않을 수 있습니다.  
**심화 포인트**: 상품 수가 많은 쇼핑몰을 전부 SSG로 빌드하면 배포 시간이 과도하게 늘 수 있습니다. 그래서 일부만 정적으로 만들거나 ISR을 섞습니다.

### Q7. ISR은 무엇이고 왜 필요한가요?
**기본 답변** (2-3문장): ISR은 SSG의 빠른 응답과 SSR의 최신성 사이를 절충한 방식입니다. 미리 생성한 페이지를 제공하되, 일정 시간이 지나면 백그라운드에서 새 HTML을 다시 만듭니다. 자주 바뀌지만 매 요청마다 SSR할 필요는 없는 페이지에 적합합니다.  
**심화 포인트**: 면접에서는 "언제 재생성되는가"를 설명해야 합니다. 첫 요청 사용자가 재검증 트리거를 만들 수 있고, 이전 캐시를 먼저 보여준 뒤 새 결과로 교체될 수 있습니다.

### Q8. Next.js App Router에서는 이 개념이 어떻게 보이나요?
**기본 답변** (2-3문장): App Router에서는 페이지 단위보다 데이터 fetch 방식으로 렌더링 성격이 결정되는 경우가 많습니다. `fetch`의 캐시 옵션, `revalidate`, 동적 함수 사용 여부에 따라 SSG처럼 동작하거나 SSR처럼 동작합니다. 즉, "React 컴포넌트"보다 "데이터 전략"이 더 중요해졌습니다.  
**심화 포인트**: 예를 들어 `revalidate = 60`은 ISR에 가깝고, `cache: 'no-store'`는 동적 렌더링에 가깝습니다. App Router는 렌더링 방식을 선언적으로 조합하게 만든 것이 핵심입니다.

## 꼬리 질문 대비

- CSR에서도 SEO를 보완할 방법은 없나요?
- SSG와 CDN 캐시는 어떤 관계가 있나요?
- App Router에서 Server Component는 CSR을 없애는 개념인가요?
- 로그인한 사용자 대시보드는 왜 SSG보다 SSR이나 CSR이 많나요?

## 헷갈리기 쉬운 포인트
CSR은 "서버를 안 쓴다"는 뜻이 아닙니다. 화면 렌더링이 클라이언트에서 일어날 뿐, 데이터 API 서버는 여전히 필요합니다.  
SSR과 Server Component는 같은 말이 아닙니다. SSR은 HTML 생성 방식이고, Server Component는 컴포넌트 실행 위치에 대한 개념입니다.  
SSG가 항상 최고 성능은 아닙니다. 데이터가 자주 바뀌면 재배포와 캐시 무효화 비용이 더 커질 수 있습니다.

## 면접관 시각
이 주제에서는 용어 암기보다 선택 기준을 봅니다. 지원자가 "이 페이지는 왜 SSR이고, 저 페이지는 왜 SSG인지"를 서비스 예시로 설명할 수 있는지가 중요합니다. Next.js 문법을 외웠는지보다, 사용자 경험과 서버 비용, SEO를 함께 연결해서 말하는지를 봅니다.
