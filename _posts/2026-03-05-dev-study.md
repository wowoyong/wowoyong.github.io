## 핵심 요약

컴퓨터구조는 프로그램 명령이 하드웨어에서 어떻게 해석되고 실행되는지를 설명하는 과목으로, 명령어 사이클·자료 경로·기억장치 계층이 핵심 축입니다. 성능은 단순한 동작 속도만이 아니라 지연시간, 처리량, 전력 효율을 함께 봐야 실제 시스템 선택에 도움이 됩니다. 예를 들어 같은 연산 성능이라도 파이프라인 위험 처리 방식이나 캐시 적중률에 따라 체감 성능이 크게 달라집니다. 실무에서는 코드 최적화, 시스템 튜닝, 장치 선택을 할 때 메모리 병목과 입출력 대기 시간을 줄이는 관점이 가장 중요합니다. 최근에는 저전력 고성능, 인공지능 연산 특화, 칩렛과 이기종 병렬 처리처럼 구조적 변화가 빠르게 진행되어 기본 원리와 최신 흐름을 함께 이해해야 합니다.

## 명령어 처리 사이클과 성능 지표 읽기

- 명령어 실행은 인출-해독-실행-기록 단계로 나뉘며, 단계별 지연을 나눠 보면 병목 위치를 찾기 쉽습니다.
- 클록 주기가 짧아도 단계 수가 과도하면 전체 지연이 늘 수 있어, 주파수와 단계 깊이는 함께 비교해야 합니다.
- 명령어당 평균 사이클 수는 실제 프로그램 성능과 직접 연결되며, 분기나 메모리 접근이 많으면 값이 커집니다.
- 처리량은 단위 시간당 완료 작업 수를 뜻하고, 지연시간은 한 작업 완료까지 걸리는 시간이라 용도가 다릅니다.
- 예를 들어 실시간 제어는 지연시간이 더 중요하고, 대량 데이터 처리는 처리량이 더 중요합니다.
- 같은 성능 수치라도 전력 소모가 크면 모바일 환경에서 불리하므로 성능 대비 전력 효율을 함께 봅니다.
- 성능 해석 시 합성 시험 결과만 보지 말고 실제 작업 비중에 맞춘 측정값을 함께 확인해야 오판을 줄일 수 있습니다.

## 자료 경로와 제어 장치의 역할 분해

- 자료 경로는 연산 장치, 레지스터, 버스로 구성되며 실제 데이터가 이동하고 계산되는 통로입니다.
- 제어 장치는 각 시점에 어떤 신호를 줄지 결정해 자료 경로를 동작시키므로, 두 요소는 기능이 명확히 분리됩니다.
- 하드와이어드 제어는 빠르지만 변경이 어렵고, 마이크로프로그램 제어는 유연하지만 제어 오버헤드가 생길 수 있습니다.
- 단일 사이클 설계는 이해가 쉽지만 가장 느린 명령 기준으로 주기를 잡아 비효율이 커질 수 있습니다.
- 다중 사이클 설계는 명령 종류별로 필요한 단계만 사용해 자원 활용이 좋아 교육용 비교에 적합합니다.
- 예를 들어 곱셈 같은 복잡 연산은 여러 내부 단계를 쓰므로 제어 신호 스케줄링이 성능에 큰 영향을 줍니다.
- 실무 해석에서는 자료 이동량을 줄이는 배치가 연산 최적화만큼 중요하며, 버스 경합 관리가 핵심입니다.

## 파이프라인 병렬성과 위험 해결 전략

- 파이프라인은 명령어 단계를 겹쳐 실행해 처리량을 높이는 방식으로, 공장 라인 분업과 유사합니다.
- 구조적 위험은 같은 자원을 동시에 요구할 때 발생하므로 자원 복제나 접근 순서 조정으로 완화합니다.
- 데이터 위험은 앞선 결과가 아직 준비되지 않았을 때 생기며, 전달 경로 추가나 대기가 대표 해결책입니다.
- 제어 위험은 분기 결과 확정 전 다음 명령을 잘못 가져올 때 생겨 분기 예측이 성능 차이를 만듭니다.
- 예를 들어 반복문 분기를 잘 예측하면 대기 사이클이 크게 줄어 실제 체감 속도가 개선됩니다.
- 깊은 파이프라인은 최대 처리량에 유리하지만 예측 실패 페널티가 커져 작업 특성에 따라 손해가 날 수 있습니다.
- 실무에서는 분기 비율 높은 코드와 메모리 의존 코드에서 파이프라인 효율이 크게 달라지므로 성능 분석이 필수입니다.

## 기억장치 계층과 지역성 기반 최적화

- 기억장치는 레지스터-캐시-주기억장치-보조기억장치 순으로 구성되며, 위로 갈수록 빠르고 용량이 작습니다.
- 시간 지역성은 최근 사용 데이터를 다시 쓸 가능성이 높다는 뜻이고, 공간 지역성은 인접 데이터를 함께 쓸 가능성입니다.
- 캐시 적중률이 높으면 평균 접근 시간이 줄어 연산 장치 유휴 시간을 크게 낮출 수 있습니다.
- 직접 사상은 단순하고 빠르지만 충돌에 약하고, 집합 연관은 충돌을 줄이는 대신 제어 복잡도가 증가합니다.
- 쓰기 정책은 즉시 반영과 지연 반영으로 나뉘며, 데이터 안전성과 버스 트래픽 사이에서 선택이 필요합니다.
- 예를 들어 행 우선 순회가 열 우선 순회보다 빠른 경우가 많은데, 이는 연속 주소 접근으로 공간 지역성이 좋아서입니다.
- 실무에서는 자료구조 배치와 반복문 순서만 바꿔도 캐시 효율이 올라 성능이 크게 개선되는 사례가 흔합니다.

## 입출력 구조와 인터럽트 기반 동작

- 입출력은 중앙처리장치가 직접 대기하는 방식보다 인터럽트 기반 처리에서 자원 활용이 더 좋아집니다.
- 인터럽트는 긴급도 우선순위를 두어 처리하며, 우선순위 설계가 잘못되면 응답 지연과 기아가 생길 수 있습니다.
- 직접 메모리 접근은 장치가 주기억장치와 직접 데이터 교환을 수행해 중앙처리장치 부담을 줄입니다.
- 메모리 사상 입출력은 프로그래밍이 단순하지만 주소 공간을 공유하므로 보호 설계를 주의해야 합니다.
- 폴링은 구현이 쉽지만 대기 비용이 커서 고속 장치나 다중 작업 환경에서는 비효율적입니다.
- 예를 들어 대용량 저장장치 전송은 직접 메모리 접근이 유리하고, 짧은 제어 신호는 인터럽트가 더 적합합니다.
- 실무에서는 장치 지연 특성과 데이터 전송 크기를 함께 보고 폴링, 인터럽트, 직접 메모리 접근을 혼합 적용합니다.

## 병렬 처리 구조 비교: 다중 코어와 가속기

- 다중 코어는 범용 작업 분산에 강하고, 가속기는 특정 연산 반복에서 높은 전력 효율을 보입니다.
- 명령어 수준 병렬성은 한 코어 내부 최적화이고, 스레드 수준 병렬성은 여러 실행 흐름 분산이라는 점이 다릅니다.
- 공유 메모리 병렬은 개발이 비교적 쉽지만 동기화 비용이 커질 수 있어 잠금 경합 관리가 중요합니다.
- 분산 메모리 병렬은 확장성이 좋지만 통신 비용이 크므로 데이터 분할 전략이 성능 핵심입니다.
- 예를 들어 영상 처리나 행렬 연산은 가속기 이점이 크고, 웹 서버 제어 로직은 다중 코어 범용성이 유리합니다.
- 병렬화 비율이 낮으면 코어 수를 늘려도 속도 향상이 제한되어, 직렬 구간 최적화가 먼저입니다.
- 실무 해석에서는 작업 성격, 개발 복잡도, 전력 예산을 함께 따져 병렬 구조를 선택해야 총비용을 줄일 수 있습니다.

## 최신 흐름

- 저전력 설계가 성능 경쟁만큼 중요해지며 전력당 성능 지표 중심의 제품 평가가 확대되고 있습니다.
- 칩렛 기반 패키징이 확산되어 기능 블록을 분리 제작한 뒤 결합하는 방식이 비용과 수율 측면에서 주목받고 있습니다.
- 중앙처리장치와 가속기 결합 구조가 늘어나며 범용 연산과 인공지능 연산을 한 시스템에서 유연하게 처리합니다.
- 고대역폭 메모리 채택이 증가해 메모리 병목을 줄이려는 시도가 서버와 고성능 계산 분야에서 강화되고 있습니다.
- 메모리 일관성 모델 최적화와 캐시 일관성 트래픽 절감 기술이 다중 코어 확장성의 핵심 과제로 다뤄집니다.
- 보안 측면에서 미세구조 기반 정보 유출 완화 설계가 기본 요구사항으로 자리잡고 있습니다.
- 근사 연산과 저정밀 연산 활용이 늘어나며 인공지능 추론에서 전력과 지연시간 절감이 가속되고 있습니다.
- 컴파일러와 하드웨어 협력 최적화가 강화되어 명령 스케줄링과 데이터 배치를 통합적으로 조정하는 흐름이 커지고 있습니다.
- 엣지 장치용 이기종 통합 설계가 확대되어 제한된 배터리 환경에서도 실시간 처리를 목표로 합니다.
- 열 설계와 전력 관리가 초기 아키텍처 단계부터 함께 설계되며 성능 유지와 안정성 확보를 동시에 추구합니다.
