## 핵심 요약

인터넷 통신은 TCP/IP 4계층 모델을 기반으로 데이터를 패킷 단위로 분할하여 전달한다. 각 계층은 독립적인 역할을 맡아 캡슐화와 역캡슐화를 통해 데이터를 전송하며, 하위 계층이 상위 계층을 투명하게 지원한다. 전송 계층의 TCP는 신뢰성을 보장하는 반면 UDP는 속도를 우선시하며, 서비스 특성에 따라 선택이 갈린다. 응용 계층의 HTTP는 웹 통신의 핵심이며, 현재는 암호화된 HTTPS가 사실상 표준으로 자리잡았다. 최근에는 UDP 기반의 QUIC 프로토콜이 HTTP/3의 기반이 되면서 웹 통신의 패러다임이 바뀌고 있다. 클라우드와 마이크로서비스 환경에서 네트워크 지식은 개발자에게도 필수 역량이 되었다.

## TCP/IP 4계층 모델: OSI 7계층과 비교

- TCP/IP는 응용(Application), 전송(Transport), 인터넷(Internet), 네트워크 접근(Network Access) 4개 계층으로 구성 — OSI 7계층을 실용적으로 압축한 모델
- OSI의 세션·표현·응용 계층을 TCP/IP에서는 '응용 계층' 하나로 통합 — 실제 구현에서는 경계가 모호한 경우가 많음
- 각 계층은 상위 계층 데이터를 페이로드로 감싸 헤더를 붙이는 '캡슐화'를 수행하고, 수신 측은 역순으로 '역캡슐화' 처리
- 계층 간 의존성이 낮아 HTTP를 바꿔도 TCP는 그대로 유지 — 계층 독립성이 기술 발전의 핵심 원리
- 네트워크 접근 계층은 이더넷, Wi-Fi 등 물리적 매체를 다루며, 프레임 단위로 데이터를 전송
- 인터넷 계층은 IP 주소로 목적지를 지정하고 패킷 단위 라우팅을 담당 — 경로 선택은 라우터가 수행
- 전송 계층은 포트 번호로 프로세스를 구분하며, 세그먼트 또는 데이터그램 단위로 분할

## IP 주소 체계와 서브네팅 계산

- IPv4는 32비트 주소 체계로 약 43억 개 주소 표현 가능 — 이미 고갈되어 NAT, IPv6로 보완 중
- 서브넷 마스크 예시: 192.168.1.0/24는 앞 24비트가 네트워크 주소, 나머지 8비트(256개)가 호스트 주소
- CIDR(Classless Inter-Domain Routing) 방식으로 클래스 개념 없이 유연하게 서브넷을 분할 가능 — /25이면 128개 호스트
- 사설 IP(10.x.x.x, 172.16~31.x.x, 192.168.x.x)는 인터넷에서 라우팅 불가 — NAT 게이트웨이를 통해 공인 IP로 변환
- IPv6는 128비트로 사실상 무한대에 가까운 주소를 제공 — 2001:db8::1 형식, 스마트홈·IoT 확산으로 전환 가속
- AWS VPC, 쿠버네티스 파드 네트워크 등 클라우드 인프라에서 서브네팅은 보안·라우팅 설계의 핵심
- 브로드캐스트 주소(.255)와 네트워크 주소(.0)는 호스트에 할당 불가 — 유효 호스트 수 = 2^n - 2

## TCP 대 UDP: 신뢰성과 속도의 트레이드오프

- TCP는 연결 지향 방식(3방향 핸드셰이크: SYN → SYN-ACK → ACK)으로 세션을 수립한 후 데이터를 교환
- TCP의 흐름 제어(슬라이딩 윈도우)와 혼잡 제어(AIMD 알고리즘)가 패킷 손실 시 재전송을 보장
- UDP는 헤더가 8바이트로 매우 가볍고 연결 수립 없이 즉시 전송 — DNS 조회, 게임, 실시간 영상에 적합
- HTTP/1.1·2는 TCP 위에서 동작하지만, HTTP/3은 UDP 기반 QUIC 위에서 동작하여 헤드오브라인 블로킹 문제를 해소
- TCP 타임아웃은 수백 밀리초 단위 — 실시간 음성 통화(VoIP)는 지연이 치명적이므로 재전송 없는 UDP를 사용
- TCP 연결은 4방향 종료(FIN → FIN-ACK → FIN → ACK) 과정이 필요 — TIME_WAIT 상태로 잠시 포트를 점유
- 최근 QUIC 프로토콜은 UDP 위에서 신뢰성·암호화·다중화를 직접 구현하여 TCP의 장점까지 수용

## HTTP/HTTPS 동작 원리와 상태 코드 해석

- HTTP는 무상태(Stateless) 프로토콜 — 매 요청이 독립적이며, 세션 유지는 쿠키·토큰으로 별도 구현
- HTTP/1.1은 기본적으로 지속 연결(Keep-Alive)을 지원하지만, 요청-응답이 순차적이라 파이프라이닝 효율이 낮음
- HTTP/2는 다중화(멀티플렉싱)로 하나의 TCP 연결에서 여러 스트림을 동시에 처리 — 헤더 압축(HPACK)도 지원
- HTTPS는 TLS(전송 계층 보안) 계층을 추가하여 기밀성·무결성·인증을 보장 — 인증서로 서버 신원 확인
- 상태 코드 분류: 2xx(성공), 3xx(리다이렉션), 4xx(클라이언트 오류: 404 미존재, 401 미인증), 5xx(서버 오류)
- REST API 설계에서 GET은 멱등(동일한 요청을 여러 번 해도 결과 동일), POST는 비멱등 — 캐시 가능 여부와 연결
- CORS(교차 출처 리소스 공유)는 브라우저가 다른 출처의 HTTP 요청을 차단하는 보안 정책 — 서버 응답 헤더로 허용 설정

## DNS 동작 방식: 도메인을 IP로 변환하는 과정

- DNS는 계층적 분산 데이터베이스 — 루트 서버 → 최상위 도메인 서버(.com, .kr) → 권한 있는 서버 순으로 질의
- 재귀 리졸버(통신사 DNS)가 캐시를 확인하고 없으면 루트부터 순차적으로 질의 — 최종 IP를 클라이언트에 반환
- TTL(Time To Live)이 지나면 캐시가 만료 — 값이 짧을수록 변경이 빠르지만 질의 부하는 증가 (트레이드오프)
- A 레코드(IPv4), AAAA 레코드(IPv6), CNAME(별칭), MX(메일 서버), TXT(도메인 인증) 등 레코드 유형 구분
- nslookup google.com 또는 dig google.com 명령으로 DNS 질의 과정과 응답 레코드를 직접 확인 가능
- DNS 스푸핑 공격은 위조된 DNS 응답으로 사용자를 피싱 사이트로 유도 — DNSSEC로 응답 서명을 검증하여 방어
- CDN(콘텐츠 전송 네트워크)은 DNS의 지리적 라우팅을 활용해 사용자에게 가장 가까운 서버의 IP를 반환

## 라우팅과 스위칭: 네트워크 내부 동작

- 스위치(2계층)는 MAC 주소 테이블을 참조하여 같은 네트워크 내 프레임을 전달 — IP 주소는 참조하지 않음
- 라우터(3계층)는 IP 주소와 라우팅 테이블로 서로 다른 네트워크 간 패킷을 전달 — 경로 선택을 담당
- 정적 라우팅은 관리자가 직접 경로를 설정 — 소규모 네트워크에 적합하지만 변화에 취약
- 동적 라우팅은 OSPF, BGP 같은 프로토콜로 라우터 간 경로 정보를 자동 교환 — 인터넷 백본은 BGP를 사용
- ARP(주소 결정 프로토콜)는 IP 주소를 MAC 주소로 변환 — 같은 서브넷 내 통신의 첫 단계
- VLAN은 물리적으로 동일한 스위치에서 논리적으로 네트워크를 분리 — 보안·트래픽 격리에 활용
- 클라우드에서는 소프트웨어 정의 네트워킹(SDN)이 하드웨어 라우터·스위치를 가상화하여 코드로 제어

## 소켓 프로그래밍 기초와 실무 활용

- 소켓은 IP 주소와 포트 번���의 조합으로 프로세스 간 통신의 끝점을 식별
- 서버 소켓 흐름: socket() → bind() → listen() → accept() → read/write → close()
- 클라이언트 소켓 흐름: socket() → connect() → read/write → close() — TCP 연결 수립 후 데이터 교환
- Node.js의 net 모듈, Python의 socket 모듈로 수십 줄만에 에코 서버 구현 가능 — 핵심 개념 실습에 최적
- 웹소켓(WebSocket)은 HTTP 업그레이드 과정을 거쳐 양방향 지속 연결을 유지 — 채팅·실시간 대시보드에 활용
- 포트 번호 80(HTTP), 443(HTTPS), 22(SSH), 3306(MySQL)은 잘 알려진 포트 — 1024 이하는 시스템 권한 필요
- 다중 클라이언트 처리 방식: 멀티스레드, select/epoll(이벤트 루프), 비동기 입출력 중 선택 — Node.js는 이벤트 루프 방식

## 최신 흐름

- HTTP/3과 QUIC 프로토콜이 주요 브라우저와 CDN에서 기본 지원 — 전체 웹 트래픽의 약 30% 이상이 이미 QUIC를 사용
- 제로 트러스트 네트워크 아키텍처(ZTA) 확산 — VPN 대신 모든 접속을 신원 기반으로 지속 검증하는 방식으로 전환 중
- eBPF 기술로 커널 수준에서 고성능 네트워크 패킷 처리 가능 — Cilium, Cloudflare의 핵심 기술로 채택
- 서비스 메시(Istio·Envoy)가 마이크로서비스 간 네트워크 트래픽을 코드 변경 없이 제어·관찰
- IPv6 전환 가속 — 국내 이동통신망에서 IPv6 비율이 꾸준히 증가하며 이중 스택 운영이 표준화
- AI 기반 네트워크 이상 탐지(AIOps) 도입 증가 — 트래픽 패턴 분석으로 DDoS 공격을 실시간 차단
- 엣지 컴퓨팅 확산으로 처리를 사용자와 가까운 곳으로 분산하여 네트워크 지연(레이턴시)을 최소화
- QUIC 기반 WebTransport가 웹소켓을 대체할 차세대 양방향 통신 표준으로 부상 중
- 5G 독립형(SA) 네트워크의 슬라이싱으로 용도별 가상 네트워크 분리 — 클라우드와의 통합 가속
- 패스키(Passkey)와 상호 TLS(mTLS)가 API·서비스 간 인증 표준으로 자리잡으며 비밀번호 기반 인증을 대체
