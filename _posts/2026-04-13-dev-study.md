## 핵심 요약

컴파일러는 사람이 작성한 프로그램을 기계가 이해할 수 있는 형태로 바꾸는 번역기이자, 성능과 안정성을 좌우하는 핵심 기반 기술이다. 학부 수준에서는 어휘 분석, 구문 분석, 의미 분석, 중간 표현, 최적화, 코드 생성의 흐름을 하나로 이어서 이해하는 것이 중요하다. 실제로는 오류 메시지의 품질, 빌드 시간, 이식성, 디버깅 편의성까지 함께 고려해야 하므로, 단순한 번역기라기보다 개발 도구 체인의 일부로 보는 편이 맞다. 같은 프로그램이라도 최적화 단계와 목표 기계 구조에 따라 실행 속도와 메모리 사용량이 크게 달라질 수 있다. 최근에는 다단계 중간 표현, 웹 실행 환경, 이종 가속기 대상 코드 생성, 증분 빌드, 기계학습 기반 최적화가 중요한 흐름으로 떠오르고 있다. 따라서 컴파일러 학습은 언어 이론에만 머무르지 말고, 성능 공학과 도구 체인, 현대 하드웨어에 대한 이해까지 함께 연결해 보는 것이 실용적이다.

## 컴파일러 전체 파이프라인 한눈에 이해하기

- 컴파일러는 보통 어휘 분석, 구문 분석, 의미 분석, 중간 표현 생성, 최적화, 코드 생성, 연결의 순서로 동작하며, 각 단계는 앞선 결과를 더 구조화된 형태로 바꾼다.
- 어휘 분석은 문자열을 토큰으로 나누는 단계로, 예를 들어 `if(a+1)`은 예약어, 식별자, 연산자, 숫자, 괄호 같은 단위로 분해된다.
- 구문 분석은 토큰이 문법에 맞는지 확인하는 단계로, `a + b * c`에서 곱셈이 덧셈보다 먼저 결합된다는 구조를 트리 형태로 만든다.
- 의미 분석은 문법적으로는 맞지만 의미상 잘못된 코드를 찾아내는데, 정수를 기대하는 자리에 문자열을 넣는 경우가 대표적이다.
- 중간 표현은 언어와 기계를 분리해 주는 완충층으로, 앞단은 언어별 차이를 흡수하고 뒷단은 기계별 차이를 처리하기 쉬워진다.
- 최적화는 같은 의미를 유지하면서 더 빠르거나 더 작은 코드로 바꾸는 과정이며, 반복문 안의 불필요한 계산을 바깥으로 빼는 방식이 흔하다.
- 코드 생성은 최종적으로 레지스터, 명령어, 메모리 접근 방식을 결정하는 단계이므로, 같은 알고리즘이라도 목표 기계에 따라 결과 품질이 달라진다.

## 어휘 분석과 구문 분석에서 문법이 구조로 바뀌는 방식

- 어휘 분석기는 문자 흐름을 읽어 의미 있는 조각으로 나누고, 공백과 주석은 걸러 낸 뒤 토큰 종류와 값만 남겨 이후 단계를 단순하게 만든다.
- 정규식 기반 규칙은 토큰을 구분하는 데 강하지만, 중첩 구조처럼 계층 정보가 필요한 문제는 구문 분석 단계에서 처리한다는 점에서 역할이 다르다.
- 구문 분석 트리는 문장 구조를 드러내며, 함수 호출과 배열 접근이 비슷해 보여도 문법 규칙이 다르면 서로 다른 하위 트리로 표현된다.
- 하향식 방식은 사람이 읽기 쉬운 규칙 설계에 유리하고, 상향식 방식은 더 넓은 문법을 안정적으로 처리하는 데 강해 도구 선택이 달라진다.
- 왼쪽 재귀나 모호한 문법은 분석기를 어렵게 만들며, 조건문 결합 문제처럼 실제 언어 설계에서 자주 마주치는 충돌 사례가 있다.
- 오류 복구도 중요해서, 첫 번째 오류에서 바로 멈추는 컴파일러보다 가능한 범위에서 다음 오류까지 함께 보여 주는 컴파일러가 실무 생산성을 높인다.
- 실제로는 문법 적합성만 보는 것이 아니라 연산자 우선순위, 결합 방향, 함수 인자 개수 같은 규칙도 함께 반영해야 사용자가 기대한 구조가 나온다.

## 의미 분석과 형 검사로 프로그램 타당성 검증하기

- 의미 분석은 이름 해석, 범위 관리, 형 검사, 제어 흐름 제약 검사를 포함하며, 문법 검사를 통과한 코드의 실제 의미를 확인한다.
- 이름 해석에서는 같은 이름이라도 지역 변수, 전역 변수, 함수 이름이 문맥에 따라 다르게 연결되며, 심볼 테이블이 이를 관리한다.
- 형 검사는 `정수 + 정수`처럼 허용된 연산만 통과시키며, 자동 형 변환이 많은 언어는 편리하지만 숨은 오류를 만들기 쉬워 비교가 필요하다.
- 함수 호출 검사는 인자의 수와 형을 확인하므로, 선언은 둘인데 호출은 셋인 경우처럼 문법상 자연스러워 보여도 의미 오류로 처리된다.
- 제어 흐름 검사는 반환문이 없는 함수, 초기화 전 변수 사용, 도달할 수 없는 코드 같은 문제를 찾아 디버깅 시간을 줄여 준다.
- 정적 검사는 실행 전에 오류를 잡는 장점이 있고, 동적 검사는 유연하지만 실행 중 실패할 수 있다는 점에서 언어 설계 철학의 차이가 드러난다.
- 실무적으로는 의미 분석의 품질이 좋을수록 오류 메시지가 더 정확해지고, 초보자에게는 문법보다 의미 단계의 친절한 진단이 학습 효과를 크게 높인다.

## 중간 표현과 최적화 단계의 역할 나눠 이해하기

- 중간 표현은 소스 언어와 기계어 사이를 잇는 공통 언어로, 하나의 앞단과 여러 뒷단을 연결할 수 있게 해 재사용성을 높인다.
- 세 주소 코드 같은 단순한 표현은 분석이 쉽고, 정적 단일 대입 형태는 데이터 흐름 최적화에 강해 목적에 따라 표현 방식이 달라진다.
- 상수 접기는 `3*4`를 미리 `12`로 바꾸는 기초 최적화인데, 작아 보여도 뒤 단계의 최적화를 연쇄적으로 이끌어 효과가 커질 수 있다.
- 공통 부분식 제거는 같은 계산을 반복하지 않도록 해 주며, 예를 들어 반복문 안에서 `a*b`가 변하지 않으면 한 번만 계산하도록 바꾼다.
- 죽은 코드 제거는 결과에 영향을 주지 않는 계산을 없애는 방식으로, 디버그용 흔적이나 도달 불가능한 분기를 줄여 코드 크기와 분석 부담을 낮춘다.
- 함수 인라인은 호출 비용을 줄일 수 있지만 코드 크기가 커져 명령어 캐시에 불리할 수 있으므로 언제나 좋은 선택은 아니다.
- 실무에서는 최고 속도만이 목표가 아니라 빌드 시간, 디버깅 가능성, 배터리 사용량도 중요하므로 최적화 수준을 상황에 맞게 조절한다.

## 코드 생성에서 레지스터와 명령어 선택이 성능에 미치는 영향

- 코드 생성기는 중간 표현을 실제 기계 명령으로 바꾸며, 같은 덧셈이라도 어떤 레지스터를 쓰고 어떤 순서로 메모리에 접근하느냐에 따라 성능이 달라진다.
- 명령어 선택은 목표 기계가 제공하는 연산을 최대한 활용하는 과정으로, 곱셈과 덧셈을 한 번에 처리하는 명령이 있으면 따로 나누는 것보다 유리하다.
- 레지스터 할당은 제한된 레지스터 수 안에서 값을 배치하는 문제인데, 부족하면 메모리로 내보내는 일이 늘어나 속도가 떨어진다.
- 명령 재배치는 파이프라인 정체를 줄이기 위해 순서를 바꾸는 기법으로, 의존성이 없는 연산을 적절히 섞어 실행 장치를 바쁘게 유지한다.
- 호출 규약은 함수 ��자를 어디에 두고 결과를 어떻게 돌려주는지 정하는 약속이므로, 다른 언어로 만든 라이브러리와 연결할 때 특히 중요하다.
- 같은 소스라도 중앙처리장치, 그래픽처리장치, 임베디드 칩처럼 목표가 달라지면 코드 생성 전략이 크게 달라져 이식성과 성능 사이의 균형이 필요하다.
- 디버그에 친화적인 코드 생성은 최고 성능보다 변수 추적과 줄 번호 매핑을 우선하기도 하므로, 개발용 빌드와 배포용 빌드를 나누는 이유가 된다.

## 컴파일러를 평가할 때 정확성, 속도, 이식성을 함께 봐야 하는 이유

- 좋은 컴파일러는 빠른 코드만 만드는 것이 아니라 원래 프로그램의 의미를 정확히 보존해야 하며, 성능보다 정확성이 우선이라는 점이 핵심이다.
- 컴파일 시간은 대형 프로젝트의 생산성과 직결되므로, 실행 속도가 조금 좋아도 빌드 시간이 지나치게 길면 실무에서는 불리할 수 있다.
- 오류 메시지의 품질은 학습과 유지보수에 큰 영향을 주는데, 위치와 원인을 구체적으로 알려 주는 컴파일러가 팀 전체 비용을 줄인다.
- 이식성은 같은 소스가 여러 운영 환경에서 동작하게 만드는 능력으로, 중간 표현과 표준화된 도구 체인이 이를 뒷받침한다.
- 최적화 수준을 높일수록 디버깅이 어려워질 수 있어, 개발 단계에서는 낮은 최적화와 풍부한 디버그 정보를 함께 쓰는 경우가 많다.
- 시험 입력 몇 개만 통과하는 것과 언어 명세를 충실히 따르는 것은 다르며, 컴파일러 검증에는 방대한 회귀 시험과 실제 프로그램 모음이 필요하다.
- 학부 수준에서는 알고리즘만 보는 데서 끝나지 말고, 왜 산업용 컴파일러가 빌드 시스템, 디버거, 정적 분석기와 함께 제공되는지도 이해할 필요가 있다.

## 최신 흐름

- 다단계 중간 표현인 MLIR 계열이 주목받으면서, 하나의 고정된 중간 표현보다 단계별 추상화 수준을 유지하는 설계가 늘고 있다.
- WebAssembly를 목표로 하는 컴파일 흐름이 확장되면서, 웹 브라우저뿐 아니라 경량 실행 환경과 임베디드 영역까지 적용 범위가 넓어지고 있다.
- 중앙처리장치뿐 아니라 그래픽처리장치와 인공지능 가속기까지 함께 겨냥하는 이종 가속기 코드 생성의 중요성이 커지고 있다.
- 증분 컴파일과 빠른 재빌드가 개발 경험의 핵심 지표가 되면서, 큰 프로젝트일수록 전체를 다시 컴파일하기보다 바뀐 부분만 다시 만드는 전략이 강화되고 있다.
- 언어 서버와 컴파일러 앞단의 결합이 깊어지면서, 편집기의 자동 완성과 즉시 오류 표시가 사실상 컴파일 기술의 일부처럼 다뤄지고 있다.
- 기계학습 기반 최적화는 일부 실제 배포 환경에서 함수 본문 삽입 결정이나 자원 배치 같은 경험적 규칙을 보완하는 데 활용되는 흐름이 뚜렷하다.
- 안전한 언어의 확산으로 메모리 안전 규칙을 컴파일 단계에서 강하게 검사하는 방향이 넓어지고 있으며, 검증 가능한 컴파일에도 관심이 커지고 있다.
- 도메인 특화 언어와 전용 컴파일러가 늘어나면서, 범용 언어 하나로 모든 문제를 해결하기보다 분야별 중간 표현과 최적화 경로를 따로 두는 경우가 많아졌다.
- 저지연 실행을 중시하는 경량 코드 생성기와 즉시 실행용 뒷단의 중요성이 커지면서, 최고 성능보다 빠른 시작과 짧은 컴파일 시간을 택하는 선택도 늘고 있다.
- 컴파일러는 단순한 번역기를 넘어 성능 분석, 정적 검사, 보안 진단, 플랫폼 적응까지 아우르는 통합 도구 체인으로 발전하는 추세를 보이고 있다.
