## 핵심 요약

운영체제는 프로세스를 생성·실행·종료하는 전 과정을 관리하며, 한정된 CPU 자원을 여러 프로세스에 효율적으로 배분하는 스케줄링이 핵심이다. 프로세스는 실행 중인 프로그램의 인스턴스로, 코드·데이터·스택·힙 영역과 PCB(프로세스 제어 블록)로 구성된다. 스레드는 프로세스 내 실행 흐름의 단위로, 같은 프로세스 내 스레드끼리 메모리를 공유해 문맥 전환 비용이 낮다. 스케줄링 알고리즘은 처리량·응답 시간·공평성을 균형 있게 최적화하며, 비선점형과 선점형으로 나뉜다. 동기화 문제(경쟁 조건, 교착 상태)는 다중 프로세스 환경에서 필연적으로 발생하며 뮤텍스·세마포어로 해결한다. 최근에는 컨테이너 기반 프로세스 격리, 경량 비동기 런타임, 클러스터 수준 스케줄링이 전통적인 운영체제 개념을 재정의하고 있다.

## 프로세스 상태 전이와 PCB 구조

- 프로세스는 생성(new) → 준비(ready) → 실행(running) → 대기(waiting) → 종료(terminated) 5가지 상태로 전이된다
- PCB(프로세스 제어 블록)는 PID, 프로그램 카운터, 레지스터 값, 메모리 맵, 입출력 상태를 저장하는 커널 자료구조다
- 문맥 전환(컨텍스트 스위칭) 시 현재 프로세스의 PCB를 저장하고 다음 프로세스의 PCB를 복원한다 — 이 과정 자체는 순수 오버헤드로 생산적인 작업이 없다
- fork() 시스템 호출은 부모 프로세스를 복제해 자식을 만들며, 리눅스는 쓰기 시 복사 방식으로 실제 메모리 복사를 최대한 미룬다
- 좀비 프로세스는 종료됐지만 부모가 wait()를 호출하지 않아 PCB가 남아 있는 상태 — 장기 방치 시 PID 자원 고갈로 이어진다
- 고아 프로세스는 부모가 먼저 종료된 프로세스로, 유닉스 계열에서는 init(PID 1)이 자동으로 입양해 정리를 보장한다
- 프로세스 계층 구조(트리)는 pstree 명령으로 시각화할 수 있으며, 시스템 디버깅 시 비정상적인 자식 프로세스 탐지에 유용하다

## 스레드와 멀티스레딩 모델 비교

- 스레드는 프로세스 내 실행 흐름으로, 코드·데이터·힙 영역을 공유하고 스택·레지스터만 독립적으로 가진다
- 커널 수준 스레드(1:1 모델)는 운영체제가 직접 관리해 진정한 병렬 실행이 가능하지만, 생성·전환 비용이 높다 — 리눅스 pthreads가 대표 사례
- 사용자 수준 스레드(N:1 모델)는 라이브러리가 관리해 전환이 빠르지만, 하나가 입출력으로 블록되면 전체 프로세스가 멈추는 치명적 단점이 있다
- 혼합 모델(M:N)은 여러 사용자 스레드를 여러 커널 스레드에 매핑해 유연성과 성능을 동시에 추구하며, 고 언어의 고루틴이 이 방식을 사용한다
- 스레드 풀은 스레드 생성 비용을 줄이기 위해 미리 만들어 둔 스레드 집합으로, 웹 서버에서 요청마다 스레드를 새로 만드는 낭비를 방지한다
- 공유 메모리 덕분에 스레드 간 통신은 빠르지만, 경쟁 조건이 발생하기 쉬워 프로세스 간 통신보다 동기화 설계가 더 까다롭다
- 단일 코어에서도 멀티스레딩은 입출력 대기 중 다른 스레드를 실행해 응답성을 높이므로, 병렬성과 동시성을 구분해서 이해해야 한다

## 스케줄링 목표와 성능 평가 기준

- CPU 이용률: CPU를 최대한 바쁘게 유지하는 비율 — 이상적으로는 100%이지만 현실 서버는 40~90% 수준을 목표로 한다
- 처리량: 단위 시간당 완료된 프로세스 수 — 배치 처리 시스템(야간 정산 등)에서 가장 중요한 지표
- 반환 시간: 프로세스 제출부터 완료까지 총 시간으로, 대기 + 실행 + 입출력 시간의 합이다
- 대기 시간: 준비 큐에서 머문 시간의 합 — 사용자가 체감하는 지연과 직결된다
- 응답 시간: 요청 후 첫 응답까지 걸리는 시간 — 터미널·GUI처럼 대화형 시스템에서 처리량보다 중요하다
- 공평성: 모든 프로세스가 적절한 CPU 시간을 받아야 하며, 특정 프로세스가 영원히 실행되지 못하는 기아 현상을 방지해야 한다
- 실시간 시스템은 마감 시한 준수가 최우선이며, 자동차 제동 제어나 의료기기에서 지연은 단순 성능 저하가 아닌 안전 문제다
- 여러 목표는 서로 충돌하므로(처리량 ↑ ↔ 응답시간 ↓), 알고리즘 선택은 사용 환경과 워크로드 특성에 따라 달라진다

## 비선점형 스케줄링: FCFS와 SJF 원리

- FCFS(선입선출): 도착 순서대로 실행 — 구현이 가장 단순하지만, 긴 작업이 앞에 오면 뒤의 짧은 작업이 오래 기다리는 호위 효과가 발생한다
- 호위 효과 예시: P1(24ms), P2(3ms), P3(3ms) 순서면 평균 대기 17ms; P2→P3→P1 순서면 평균 3ms로 5배 이상 차이가 난다
- SJF(단기 작업 우선): 실행 시간이 가장 짧은 프로세스를 먼저 선택 — 평균 대기 시간을 최소화하는 측면에서 이론적으로 최적이다
- SJF의 실무 문제: 미래의 실행 시간을 알 수 없으므로, 과거 실행 시간의 지수 이동 평균으로 다음 실행 시간을 예측(에이징 기법)한다
- 비선점형 SJF는 현재 실행 중인 프로세스를 중단시키지 않아, 한번 시작된 긴 프로세스가 짧은 작업들을 계속 뒤로 밀어내는 기아를 유발할 수 있다
- SRTF(최단 잔여 시간 우선)는 선점형 SJF로, 새 프로세스 도착 시 남은 실행 시간이 더 짧으면 현재 실행을 즉시 중단하고 교체한다
- 비선점형 스케줄링은 일괄 처리 시스템이나 인터럽트가 적은 임베디드 환경에 적합하며, 문맥 전환 오버헤드를 최소화한다

## 선점형 스케줄링: 라운드 로빈과 우선순위

- 라운드 로빈(RR): 각 프로세스에 고정 시간 할당량(타임 퀀텀, 보통 10~100ms)을 주고 순환 실행 — 대화형 시스템의 표준 알고리즘
- 타임 퀀텀이 너무 크면 FCFS와 같아지고, 너무 작으면(1ms 이하) 문맥 전환이 전체 CPU 시간의 대부분을 차지해 실제 실행이 거의 불가능해진다
- 우선순위 스케줄링: 높은 우선순위 프로세스를 먼저 실행 — 실시간 운영체제와 운영체제 내부 데몬 관리에 필수적으로 사용된다
- 우선순위 기아 해결책: 에이징 — 준비 큐에서 오래 대기한 프로세스의 우선순위를 시간이 지날수록 자동으로 높여 준다
- 다단계 큐: 프로세스를 전면(대화형)·후면(배치) 등 여러 큐로 분류해 각 큐에 다른 알고리즘을 적용한다
- 다단계 피드백 큐: CPU를 오래 쓰는 프로세스를 낮은 우선순위 큐로 이동시켜 입출력 집약적 프로세스를 우대하는 동적 방식이다
- 리눅스의 CFS(완전 공정 스케줄러)는 가상 실행 시간 개념으로 레드-블랙 트리��� 프로세스를 배치해 최소 실행 시간 프로세스를 O(log n)으로 선택한다
- 윈도우는 32단계 우선순위 + 선점형 방식을 사용하며, 실시간 클래스(16~31)와 가변 클래스(1~15)를 구분해 관리한다

## 임계 구역과 동기화 기법 실전

- 경쟁 조건(레이스 컨디션): 두 스레드가 공유 변수를 동시에 읽고 쓸 때 실행 순서에 따라 결과가 달라지는 현상 — 카운터 증가(count++)조차 읽기·증가·쓰기 세 단계로 쪼개지므로 안전하지 않다
- 임계 구역은 상호 배제(한 번에 하나만 진입), 진행(진입 희망자 없으면 즉시 허용), 한정 대기(무한 대기 금지) 세 조건을 반드시 만족해야 한다
- 뮤텍스는 소유권 개념이 있어 잠금한 스레드만 해제 가능 — 이진 세마포어와 달리 다른 스레드가 해제할 수 없어 오용 방지에 유리하다
- 세마포어는 정수 카운터로 wait(P)/signal(V) 연산을 사용 — 데이터베이스 연결 풀처럼 동시 접근자 수를 N명으로 제한할 때 유용하다
- 모니터는 객체 지향적 동기화 추상화로, 자바의 synchronized 블록·메서드와 ReentrantLock이 대표적인 구현체다
- 스핀락은 대기 중 계속 CPU를 소비하며 바쁜 대기를 하므로, 임계 구역이 매우 짧고 다중 코어 환경일 때만 문맥 전환보다 효율적이다
- 교착 상태(데드락) 예시: 스레드 A가 락1 보유 후 락2 요청, 스레드 B가 락2 보유 후 락1 요청 — 두 스레드 모두 영원히 대기 상태가 된다

## 교착 상태 조건과 해결 전략

- 교착 상태 발생의 필요 4조건: 상호 배제, 점유 대기, 비선점, 순환 대기 — 이 중 하나만 깨면 교착 상태가 원천적으로 불가능해진다
- 예방: 모든 자원을 한꺼번에 요청하게 해 점유 대기를 제거하거나, 락 획득 순서를 전역적으로 고정해 순환 대기를 제거한다
- 회피: 은행가 알고리즘처럼 자원 할당 전 안전 상태를 계산해 불안전한 할당을 거부 — 최대 요구량을 미리 알아야 하는 비현실적 전제가 단점이다
- 안전 상태: 모든 프로세스를 순서대로 완료 가능한 실행 순서가 존재하면 안전, 없으면 불안전 — 불안전이 즉시 교착 상태는 아니지만 가능성이 열린 것이다
- 탐지와 회복: 주기적으로 자원 할당 그래프에서 순환 탐지 후 프로세스 종료 또는 자원 강제 선점으로 교착 상태를 해소한다
- 무시(타조 알고리즘): 교착 상태가 드물다고 가정하고 무시 — 재부팅이 현실적 해결책인 일반 목적 운영체제에서 실제로 채택하는 방식이다
- 실무에서는 락 획득 순서 규약(항상 동일한 순서로 획득)과 tryLock() 타임아웃 방식으로 교착 상태를 실용적으로 예방하는 경우가 대부분이다

## 최신 흐름

- 리눅스 6.6에서 도입된 EEVDF 스케줄러가 CFS를 대체해 레이턴시 공정성과 대화형 응답성을 개선했다
- 컨테이너(도커, 포드맨)는 cgroup과 네임스페이스로 프로세스를 격리해 가상 머신보다 훨씬 가벼운 운영체제 수준 격리를 구현한다
- 자바 21의 가상 스레드(Project Loom)는 수백만 개의 경량 스레드를 지원해, 입출력 블로킹 없이 기존 블로킹 코드 그대로 동시성을 처리한다
- 러스트의 async/await는 운영체제 스레드 없이 비동기 태스크를 애플리케이션 수준에서 스케줄링해 서버 성능과 메모리 안전성을 동시에 확보한다
- eBPF를 활용한 커널 수준 관찰 가능성 도구(bpftrace, Pixie)가 스케줄링 지연과 문맥 전환 빈도를 운영체제 수정 없이 실시간으로 추적한다
- 에너지 효율 스케줄링이 주목받아, ARM 빅-리틀 구조처럼 고성능 코어와 저전력 코어를 작업 부하 특성에 따라 선택적으로 활용하는 이기종 스케줄링이 발전했다
- 쿠버네티스의 파드 스케줄러는 운영체제 스케줄러 개념을 클러스터 전체로 확장해, 수천 개의 노드에 컨테이너를 최적 배분하는 분산 스케줄링을 구현한다
- 실시간 리눅스(PREEMPT_RT) 패치가 리눅스 메인라인에 통합되면서, 리눅스가 산업용 실시간 운영체제로서 공식적으로 자리를 잡았다
- 서버리스·함수형 컴퓨팅 환경에서 프로세스 수명이 밀리초 단위로 단축되면서, 초기 기동 지연 최소화를 위한 프로세스 재사용·메모리 스냅샷 기법이 핵심 연구 주제가 됐다
- WASM(웹어셈블리) 컴포넌트 모델이 브라우저 밖(서버·엣지)까지 확장되면서, 샌드박스 내 프로세스 유사 격리를 경량화된 형태로 구현하는 새로운 실행 모델이 부상하고 있다
