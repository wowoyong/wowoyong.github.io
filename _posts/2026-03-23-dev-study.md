## 핵심 요약

소프트웨어공학은 단순히 프로그램 코드를 잘 작성하는 기술이 아니라, 요구 분석부터 설계, 구현, 시험, 배포, 유지보수에 이르기까지 전 과정을 체계적으로 다루는 학문입니다. 학부 수준에서는 개발 과정 모델, 설계 원칙, 협업 방식, 품질 보증, 유지보수 전략을 함께 이해해야 실제 프로젝트에서도 흔들리지 않습니다. 같은 기능이라도 요구사항이 불명확하면 일정이 지연되고 재작업이 늘어나며, 설계가 부실하면 수정 비용이 빠르게 커집니다. 실무에서는 형상 관리, 자동화된 시험, 코드 검토, 점진적 배포가 품질과 속도를 함께 지키는 핵심 장치로 활용됩니다. 최근에는 인공지능 보조 개발, 플랫폼 공학, 공급망 보안, 관찰 가능성 강화처럼 개발을 둘러싼 환경도 빠르게 바뀌고 있습니다. 따라서 개념을 외우는 데 그치지 않고, 왜 이런 절차가 필요한지, 어떤 상황에서 어떤 선택이 유리한지까지 연결해 이해하는 것이 중요합니다.

## 요구사항을 흔들림 없이 정의하는 방법

- 요구사항은 시스템이 해야 할 일과 지켜야 할 제약을 문서로 정리한 것으로, 예를 들어 회원가입 기능을 정의할 때도 응답 시간 제한과 개인정보 처리 규칙까지 함께 적어야 실제 개발 범위가 분명해집니다.
- 기능 요구사항은 시스템이 제공해야 할 기능 자체를, 비기능 요구사항은 성능, 보안, 사용성 같은 품질 기준을 다루며, 둘 중 하나만 챙기면 동작은 해도 쓰기 불편한 서비스가 되기 쉽습니다.
- 이해관계자 분석은 사용자, 개발자, 운영자, 관리자 등의 요구를 구분하는 과정으로, 학생 수강신청 시스템처럼 사용자와 관리자 관점이 충돌하는 경우에는 우선순위 조정이 반드시 필요합니다.
- 유스케이스는 사용자가 시스템과 상호작용하는 흐름을 보여 주므로, 긴 문장 설명보다 예외 상황까지 드러내기 쉬워 초반 의사소통에 유리합니다.
- 요구사항 명세서는 나중에 시험 기준이 되므로 모호한 표현보다 측정 가능한 표현이 적절하며, '빠르게'보다 '2초 이내'가 훨씬 검증하기 쉽습니다.
- 요구사항 추적성은 요구사항이 설계, 코드, 시험 항목과 어떻게 연결되는지 관리하는 것으로, 문제가 생겼을 때 어떤 기능이 영향을 받는지 빠르게 찾는 데 유용합니다.
- 요구사항 변경 관리는 실무에서 특히 중요한데, 중간에 결제 수단이 추가되면 화면만 바꾸는 것이 아니라 데이터 구조, 시험 항목, 운영 절차까지 함께 수정해야 합니다.

## 설계를 견고하게 만드는 분해와 추상화

- 소프트웨어 설계는 큰 문제를 작은 모듈로 나누고 그 관계를 정리하는 일이며, 온라인 서점이라면 주문, 결제, 재고를 나누어야 수정 범위를 통제하기 쉽습니다.
- 추상화는 핵심만 남기고 세부를 숨기는 개념으로, 데이터 저장 방식이 바뀌더라도 상위 기능이 그대로 동작하게 만들어 변경 비용을 줄여 줍니다.
- 모듈화는 기능을 독립된 단위로 나누는 방식인데, 한 모듈이 너무 많은 책임을 가지면 작은 수정도 연쇄적인 오류로 이어질 수 있어 적절한 분리가 중요합니다.
- 응집도는 한 모듈 내부 요소들이 얼마나 밀접하게 관련되어 있는지, 결합도는 모듈끼리 얼마나 얽혀 있는지를 뜻하며, 일반적으로 응집도는 높고 결합도는 낮을수록 유지보수에 유리합니다.
- 계층 구조 설계는 화면, 업무 로직, 데이터 접근을 나누는 식으로 많이 쓰이는데, 사용자 화면이 바뀌어도 핵심 계산 로직은 그대로 유지하기 쉽다는 장점이 있습니다.
- 설계 원칙은 재사용성과 변경 대응력을 높여 주며, 예를 들어 새로운 결제 방식을 추가할 때 기존 결제 코드를 크게 고치지 않아야 좋은 설계라고 볼 수 있습니다.
- 설계 문서는 단순한 보고서가 아니라 팀이 함께 보는 지도의 역할을 하므로, 구현 전에 인터페이스와 데이터 흐름을 맞춰 두면 코드 충돌과 재작업을 크게 줄일 수 있습니다.

## 개발 과정 모델을 상황에 맞게 고르는 기준

- 폭포수형은 분석, 설계, 구현, 시험을 순서대로 진행하는 방식이라 문서화가 분명하지만, 요구 변경이 잦은 프로젝트에는 대응이 느리다는 한계가 있습니다.
- 반복형은 작은 단위를 여러 차례 개선해 나가는 방식으로, 초기에 완벽하지 않아도 점진적으로 품질을 높일 수 있어 실제 서비스 개발과 잘 맞습니다.
- 애자일은 짧은 주기마다 동작 가능한 결과물을 내고 피드백을 반영하는 접근으로, 학생 팀 프로젝트처럼 기능 우선순위를 자주 조정해야 할 때 효과적입니다.
- 스크럼은 역할과 회의를 통해 협업을 구조화하는 방법으로, 짧은 개발 주기와 회고를 통해 문제를 빠르게 드러낸다는 점에서 전통적인 방식과 다릅니다.
- 프로토타입 모델은 화면이나 핵심 흐름을 먼저 보여 주어 요구를 검증하는 데 유리하며, 사용자 경험이 중요한 응용 프로그램 개발에서 특히 자주 쓰입니다.
- 나선형 모델은 반복 개발에 위험 분석을 결합한 형태로, 실패 비용이 큰 시스템에서 위험을 초기에 줄이려는 목적이 강합니다.
- 실무에서는 한 가지 모델만 순수하게 쓰기보다, 큰 일정은 계획적으로 관리하고 세부 구현은 반복적으로 진행하는 혼합 방식이 더 흔합니다.

## 협업 품질을 좌우하는 형상 관리와 코드 검토

- 형상 관리는 코드, 문서, 설정 파일의 변경 이력을 관리하는 활동으로, 누가 언제 무엇을 바꿨는지 남기기 때문에 팀 개발의 기본 인프라가 됩니다.
- 버전 관리를 사용하면 실수로 기능이 깨져도 이전 상태를 비교하거나 복구할 수 있어, 개인 개발보다 팀 개발에서 그 가치가 훨씬 커집니다.
- 브랜치 전략은 동시에 여러 작업을 안전하게 진행할 수 있게 해 주는데, 새 기능 개발과 긴급 오류 수정을 분리하면 배포 안정성이 높아집니다.
- 병합 충돌은 단순한 문법 문제가 아니라 설계 충돌의 신호인 경우가 많아서, 충돌이 잦다면 역할 분리나 인터페이스 정의를 다시 살펴볼 필요가 있습니다.
- 코드 검토는 문법 검사보다 구현 의도를 확인하는 과정에 가깝고, 예를 들어 동작은 맞더라도 확장성이 떨어지는 구현을 미리 찾아낼 수 있다는 점이 중요합니다.
- 지속적 통합은 변경 사항을 자주 합치고 자동 검사를 수행하는 방식으로, 마지막에 한꺼번에 합치는 방식보다 오류 원인을 찾기 쉽습니다.
- 이슈 관리와 작업 기록을 함께 남기면 단순히 코드만 보는 것보다 의사결정의 맥락이 드러나므로, 유지보수와 인수인계의 품질이 크게 좋아집니다.

## 시험 전략으로 결함을 초기에 줄이는 방법

- 시험은 프로그램이 단순히 실행되는지만 보는 것이 아니라, 요구사항을 만족하는지 확인하는 활동이므�� 요구 분석과 직접 연결됩니다.
- 단위 시험은 함수나 클래스 같은 작은 단위를 검증하는 방식으로, 계산 로직 오류처럼 국소적인 문제를 빠르게 찾아내는 데 적합합니다.
- 통합 시험은 모듈 간 연결을 확인하므로, 로그인 자체는 되지만 데이터베이스 저장이 실패하는 식의 연동 오류를 찾는 데 필요합니다.
- 시스템 시험은 전체 서비스 관점에서 동작을 검증하며, 실제 사용자 흐름을 따라가기 때문에 부분 시험만으로 놓치기 쉬운 문제를 발견할 수 있습니다.
- 회귀 시험은 수정 후 기존 기능이 망가지지 않았는지 확인하는 절차로, 작은 기능을 추가한 뒤 예상 밖의 오류가 생기는 실무 상황에서 매우 중요합니다.
- 검은상자 시험은 입력과 출력 중심으로, 흰상자 시험은 내부 경로 중심으로 본다는 차이가 있어, 둘을 함께 사용해야 결함 탐지 범위를 넓힐 수 있습니다.
- 자동화 시험은 반복 작업 비용을 줄여 주지만 모든 시험을 자동화하는 것이 정답은 아니며, 화면 사용성처럼 사람의 판단이 필요한 부분은 수동 확인이 여전히 필요합니다.

## 유지보수 비용을 키우는 기술 부채 이해하기

- 유지보수는 배포 후에도 오류 수정, 기능 개선, 환경 적응을 계속 수행하는 활동으로, 전체 비용에서 큰 비중을 차지하는 경우가 많습니다.
- 기술 부채는 당장 빠르게 만들기 위해 미래의 비용을 미루는 선택으로, 과제 마감 직전에 넣은 임시 코드가 나중에 전체 구조를 흔드는 상황과 비슷합니다.
- 수정 유지보수는 결함을 고치는 것이고, 적응 유지보수는 운영체제나 외부 규칙 변화에 맞추는 것이어서 원인과 대응 방식이 다릅니다.
- 완전 유지보수는 성능이나 사용성을 높이는 개선 활동으로, 같은 기능이라도 응답 속도와 화면 흐름을 다듬으면 체감 품질이 크게 달라집니다.
- 리팩터링은 외부 동작을 바꾸지 않고 내부 구조를 개선하는 작업이며, 중복 코드를 제거하는 작은 정리만으로도 장기적으로 오류 가능성을 줄일 수 있습니다.
- 문서가 낡으면 코드보다 더 위험할 수 있는데, 실제 동작과 설명이 다르면 새로 합류한 팀원이 잘못된 가정으로 수정해 장애를 만들 수 있습니다.
- 유지보수성을 높이려면 작은 함수, 명확한 이름, 낮은 결합도, 자동 시험, 기록된 의사결정이 함께 필요하며, 어느 하나만으로는 충분하지 않습니다.

## 최신 흐름

- 2025년 7월 가트너는 인공지능 내재형 소프트웨어공학을 핵심 흐름으로 제시하면서, 개발자의 역할이 단순 구현보다 문제 정의와 결과 검증 쪽으로 이동한다고 보았습니다.
- 같은 발표에서 가트너는 2028년까지 기업 소프트웨어 엔지니어의 90퍼센트가 인공지능 코드 보조 도구를 사용할 것이라고 전망하며, 도구 활용 능력이 기본 역량이 되고 있음을 보여 주었습니다.
- 2025년 이후의 흐름은 단순 자동완성을 넘어 설계, 문서화, 시험 생성까지 인공지능이 관여하는 방향으로 넓어졌고, 그만큼 결과 검증과 책임 경계 설정이 더욱 중요해졌습니다.
- 가트너는 2027년까지 절반이 넘는 소프트웨어공학 팀이 대규모 언어 모델 기반 기능을 직접 만들 것이라고 보았는데, 이는 일반 서비스도 대화형 기능과 자동화 기능을 함께 포함하게 됨을 뜻합니다.
- Google Cloud의 2025년 도라 관련 자료는 인공지능 보조 개발을 소프트웨어 전달 체계의 중요한 변화로 다루며, 단순한 속도 향상보다 팀 차원의 생산성 측정이 필요하다는 점을 강조했습니다.
- 플랫폼 공학은 여러 팀이 공통 개발 환경을 제품처럼 제공하는 흐름으로 성장했고, 도구가 제각각인 조직보다 표준화된 내부 개발 플랫폼을 갖춘 조직이 배포 일관성을 확보하기 쉽습니다.
- 관찰 가능성 강화도 중요한 흐름인데, 단순히 로그만 수집하는 수준을 넘어 지표, 추적, 오류 예산을 함께 보며 장애 원인을 빠르게 찾는 방식이 보편화되고 있습니다.
- 2026년 2월 Datadog 보고서는 배포된 서비스의 87퍼센트가 알려진 악용 가능 취약점을 하나 이상 포함하고 있다고 밝혀, 보안이 배포 이후의 점검이 아니라 개발 전 과정의 문제임을 보여 주었습니다.
- 같은 2026년 보고서에서는 서비스의 42퍼센트가 더 이상 활발히 관리되지 않는 라이브러리에 의존한다고 분석해, 오픈소스 선택과 버전 관리가 유지보수 전략의 일부가 되었음을 보여 주었습니다.
- 또한 2026년 보고서에서 공개 동작 자동화 도구를 정확한 버전으로 고정한 조직이 4퍼센트에 그쳤다는 점은 공급망 보안이 최신 흐름의 핵심임을 보여 주며, 빠른 개발과 안전한 검증을 함께 설계해야 함을 시사합니다.
