## 핵심 요약

컴퓨터구조는 프로그램이 실제 하드웨어에서 어떻게 실행되는지를 이해하게 해 주는 분야로, 중앙처리장치와 메모리, 입출력 장치가 서로 어떻게 맞물려 동작하는지를 다룬다. 학습의 핵심은 명령어가 실행되는 흐름, 성능을 높이기 위한 구조적 기법, 메모리 계층이 속도 차이를 어떻게 줄이는지를 연결해서 보는 데 있다. 실제로는 이론을 단순히 외우기보다 캐시 적중률, 파이프라인 정체, 병렬 처리 효율처럼 성능 병목을 해석하는 관점이 중요하다. 같은 알고리즘이라도 메모리에 접근하는 방식이 다르면 실행 시간이 크게 달라질 수 있으므로, 구조에 대한 이해는 최적화와 직접 이어진다. 2026-03-22 기준 최근 흐름은 인공지능 연산 가속, 칩렛 기반 설계, 고대역폭 메모리, 개방형 명령어 집합 확대처럼 하드웨어를 더 유연하고 전문화된 방향으로 발전시키고 있다.

## 명령어 실행 흐름과 성능 지표 이해하기

- 명령어 실행은 보통 가져오기, 해독, 실행, 결과 저장 단계로 나뉘며, 계산기에서 덧셈 한 줄을 수행할 때도 내부에서는 여러 제어 신호가 순서대로 움직인다.
- 성능은 기본적으로 실행 시간으로 판단해야 하며, 단순한 클록 수치만 보면 오해하기 쉽다. 같은 3기가헤르츠라도 구조가 다르면 실제 속도는 달라질 수 있다.
- 실행 시간은 명령어 수, 사이클 수, 클록 주기의 곱으로 해석할 수 있어 병목이 어디에서 생기는지 나누어 살펴볼 수 있다.
- 명령어당 평균 사이클 수는 낮을수록 유리하지만, 복잡한 명령어가 줄어드는 대신 클록이 느려질 수 있어 항상 단순 비교만으로 판단하기는 어렵다.
- 벤치마크는 실제 사용 환경과 비슷해야 의미가 있으며, 게임용 구조와 과학 계산용 구조는 유리한 성능 지표가 서로 다를 수 있다.
- 처리량은 여러 작업을 얼마나 많이 끝내는지를 뜻하고, 지연 시간은 한 작업이 얼마나 빨리 끝나는지를 뜻하므로 서버와 실시간 제어에서는 중요도가 다르게 나타난다.
- 실무에서는 코드 최적화만 볼 것이 아니라 캐시 미스와 분기 실패까지 함께 살펴봐야 성능 저하의 원인을 정확히 찾을 수 있다.

## 중앙처리장치 내부 구성과 데이터 경로

- 중앙처리장치는 산술논리장치, 레지스터, 제어장치, 데이터 경로로 이루어지며, 레지스터는 메모리보다 훨씬 빠른 임시 작업 공간 역할을 한다.
- 프로그램 계수기는 다음에 실행할 명령어의 위치를 가리키며, 반복문과 함수 호출도 이 값의 변화로 이해하면 흐름이 더 분명해진다.
- 명령어 레지스터는 현재 처리 중인 명령어를 담아 해독 단계로 넘기며, 조리대 위에 올려둔 요리 순서표처럼 생각할 수 있다.
- 산술논리장치는 덧셈, 비교, 비트 연산을 수행하고 주소 계산에도 자주 쓰이므로, 단순한 계산기보다 훨씬 넓은 역할을 맡는다.
- 제어장치는 어떤 회로가 언제 동작할지를 결정하는데, 하드와이어드 방식은 빠르고 마이크로프로그램 방식은 유연하다는 차이가 있다.
- 데이터 경로는 부품들을 잇는 실제 이동 통로이므로, 같은 기능을 가진 두 구조라도 경로 설계에 따라 성능과 복잡도가 달라진다.
- 예를 들어 곱셈 명령어 하나를 추가할 때도 명령어 형식만 바꾸는 것이 아니라 제어 신호, 데이터 경로, 예외 처리까지 함께 고려해야 한다.

## 파이프라인의 원리와 한계

- 파이프라인은 여러 명령어를 단계별로 겹쳐 처리하는 방식으로, 세탁기와 건조기를 동시에 돌리는 것처럼 전체 처리량을 높인다.
- 이상적으로는 단계 수만큼 처리량이 늘어나지만, 실제로는 단계 간 균형이 맞지 않거나 충돌이 생겨 기대만큼 오르지 않는 경우가 많다.
- 구조적 위험은 같은 자원을 동시에 요구할 때 발생하며, 예를 들어 명령어 메모리와 데이터 메모리를 분리하면 이를 줄일 수 있다.
- 데이터 위험은 앞선 명령어의 결과가 아직 나오지 않았는데 다음 명령어가 그 값을 필요로 할 때 생기며, 전달 기법으로 대기 시간을 줄일 수 있다.
- 제어 위험은 분기문 때문에 다음 명령어를 확정하지 못할 때 발생하며, 반복문이 많은 코드에서는 성능 손실이 크게 나타난다.
- 분기 예측은 맞으면 큰 이득을 주지만 틀리면 파이프라인을 비워야 하므로, 단순한 조건문이 많은 프로그램에서도 영향이 뚜렷하게 드러난다.
- 실무에서는 파이프라인이 깊을수록 클록을 높이기 쉬운 대신 분기 실패 비용이 커지기 때문에, 무조건 깊게 만드는 것이 항상 정답은 아니다.

## 메모리 계층과 지역성

- 메모리 계층은 레지스터, 캐시, 주기억장치, 보조기억장치 순으로 이루어지며, 위로 갈수록 빠르고 작고 비싸다.
- 지역성은 최근에 쓴 데이터나 가까운 위치의 데이터를 다시 사용할 가능성이 높다는 성질로, 배열 순회가 연결 리스트보다 유리한 이유를 설명해 준다.
- 시간 지역성은 같은 변수를 반복해서 사용하는 경우를 말하며, 반복문 안에서 누적 합을 계산하는 상황이 대표적인 예다.
- 공간 지역성은 인접한 데이터를 연속해서 사용하는 경우를 뜻하며, 행렬을 행 우선으로 접근할 때 캐시 효율이 높아지는 이유와 연결된다.
- 가상 메모리는 넓은 주소 공간을 제공하지만 실제 메모리보다 느린 보조기억장치와 함께 동작하므로, 페이지 부재가 많아지면 성능이 급격히 떨어진다.
- 페이지와 캐시는 모두 지역성을 활용하지만, 캐시는 더 작은 단위의 빠른 저장소이고 페이지는 운영체제 수준에서 관리하는 더 큰 단위라는 차이가 있다.
- 성능 분석에서는 계산량이 많아 보여도 실제 병목이 메모리 대기일 수 있으므로, 연산 중심 코드와 메모리 중심 코드를 구분해서 보는 습관이 중요하다.

## 캐시 설계와 체감 성능

- 캐시는 자주 쓰는 데이터를 가까운 곳에 두는 장치이며, 적중하면 주기억장치까지 가지 않아도 되어 응답 속도가 크게 좋아진다.
- 직접 사상은 단순하고 빠르지만 충돌이 많을 수 있고, 집합 연관 사상은 충돌을 줄이는 대신 회로가 더 복잡해진다.
- 완전 연관 방식은 유연하지만 비교 비용이 커서 보통 작은 버퍼나 특수한 구조에 더 잘 어울린다.
- 교체 정책은 무엇을 버릴지 결정하며, 최근에 덜 사용한 블록을 버리는 방식은 일반적인 작업에 강하지만 항상 최선은 아니다.
- 쓰기 정책은 즉시 반영 방식과 나중 반영 방식으로 나뉘며, 데이터 안정성과 성능 사이에서 선택이 갈린다.
- 다중 코어 환경에서는 같은 데이터 사본이 여러 캐시에 있을 수 있어 일관성 문제가 생기며, 공유 변수가 많은 병렬 프로그램에서 특히 중요해진다.
- 예를 들어 잘못 작성한 반복문은 계산 자체는 단순해도 캐시 줄 크기와 충돌 패턴 때문에 체감 속도가 몇 배씩 차이 날 수 있다.

## 병렬성 확대: 명령어 수준부터 다중 코어까지

- 명령어 수준 병렬성은 서로 독립적인 연산을 동시에 처리하는 개념으로, 덧셈과 곱셈이 서로의 결과를 기다리지 않으면 겹쳐서 실행할 수 있다.
- 초스칼라 구조는 한 사이클에 여러 명령어를 발행하려 하지만, 실제 효율은 데이터 의존성과 분기 예측 정확도에 크게 좌우된다.
- 비순차 실행은 입력이 준비된 명령어를 먼저 실행해 유휴 시간을 줄이는 방식으로, 순서대로만 처리하는 방식보다 복잡하지만 성능 이점이 크다.
- 벡터 처리와 단일 명령어 다중 데이터 방식은 같은 연산을 많은 데이터에 반복할 때 강하며, 영상 처리나 행렬 계산에서 특히 효과적이다.
- 다중 코어는 여러 작업이나 스레드를 동시에 처리하지만, 코어 수가 두 배가 되었다고 해서 프로그램 속도가 반드시 두 배가 되지는 않는다.
- 병렬 프로그램의 성능은 잠금 경쟁, 메모리 대역폭, 캐시 일관성 비용 때문에 제한되며, 이 점에서 단일 코어 최적화와 다른 접근이 필요하다.
- 실무적으로는 어떤 작업이 연산 병목인지 메모리 병목인지 먼저 구분해야 중앙처리장치 벡터화, 다중 스레드, 가속기 활용 가운데 올바른 선택을 할 수 있다.

## 입출력 구조와 저장장치 연결 방식

- 입출력 장치는 중앙처리장치보다 훨씬 느린 경우가 많아서, 명령어 성능만 높여서는 전체 체감 속도가 크게 좋아지지 않을 수 있다.
- 프로그램 제어 입출력은 구현이 단순하지만 중앙처리장치가 계속 상태를 확인해야 하므로 낭비가 크고, 작은 실습에는 적합해도 대용량 전송에는 비효율적이다.
- 인터럽트 방식은 장치가 준비 완료를 알려 주므로 중앙처리장치가 다른 일을 할 수 있어, 키보드 입력처럼 간헐적으로 발생하는 사건 처리에 유리하다.
- 직접 메모리 접근은 장치가 메모리와 직접 데이터를 주고받아 대용량 전송에 효율적이며, 저장장치 읽기 작업에서 자주 쓰인다.
- 버스는 여러 장치가 함께 쓰는 통신 통로이므로, 폭과 클록뿐 아니라 중재 방식도 성능에 영향을 미친다.
- 저장장치는 자기디스크보다 반도체 저장장치가 훨씬 빠르지만, 가격과 쓰기 수명 특성이 달라 모든 상황에서 완전히 대체할 수 있는 것은 아니다.
- 예를 들어 데이터베이스 서버는 계산 능력보다 저장장치 지연과 입출력 대기 시간이 더 큰 병목이 되는 경우가 많아, 구조에 대한 이해가 중요하다.

## 최신 흐름

- 인공지능 연산 수요가 폭증하면서 범용 중앙처리장치만으로는 한계가 뚜렷해졌고, 그래픽처리장치와 신경망처리장치 같은 전용 가속기 중심 설계의 중요성이 더욱 커지고 있다.
- 칩 하나를 크게 만드는 대신 여러 조각을 묶는 칩렛 구조가 확산되면서, 수율 개선과 기능 조합의 유연성 면에서 대형 단일 칩보다 유리한 경우가 늘고 있다.
- 고대역폭 메모리인 HBM 계열은 인공지능 가속기의 핵심으로 자리 잡았고, 2025년 이후 HBM4 논의와 공급 경쟁이 메모리 병목 완화의 중심 이슈가 되었다.
- 메모리 용량 부족 문제를 줄이기 위해 CXL 기반 메모리 확장과 메모리 풀링이 주목받고 있으며, 여러 계산 장치가 메모리 자원을 더 유연하게 나누는 방향으로 발전하고 있다.
- 개방형 명령어 집합인 RISC-V는 교육용을 넘어 임베디드와 일부 가속기 제어 영역까지 확장되고 있으며, 맞춤형 구조 연구에 특히 적합한 기반으로 평가받는다.
- 전력 효율이 성능만큼 중요해지면서 같은 계산을 더 적은 에너지로 끝내는 구조가 경쟁력이 되었고, 모바일과 데이터센터 모두 와트당 성능을 엄격하게 따지고 있다.
- 메모리 가까이에서 계산을 수행하거나 데이터 이동을 줄이려는 접근이 늘어나고 있으며, 이는 연산 자체보다 이동 비용이 더 큰 현대 작업의 특성을 반영한다.
- 보안 요구가 높아지면서 단순한 기능 구현을 넘어 추측 실행 취약점, 격리, 메모리 보호를 구조 단계에서부터 고려하는 설계가 중요해지고 있다.
- 서버와 개인 기기 모두에서 이종 계산이 일반화되면서, 중앙처리장치가 전체를 조정하고 가속기가 특정 작업을 맡는 분업 구조가 사실상 표준에 가까워지고 있다.
- 2026-03-22 시점의 큰 흐름을 한 문장으로 요약하면, 컴퓨터구조는 더 빠른 단일 코어 경쟁보다 전문화, 병렬화, 메모리 최적화, 모듈화 쪽으로 무게중심이 옮겨가고 있다.
