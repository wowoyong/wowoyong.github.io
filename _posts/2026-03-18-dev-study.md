## 핵심 요약

컴파일러는 사람이 작성한 소스 코드를 기계가 효율적으로 실행할 수 있는 형태로 바꾸는 번역기다. 학부 수준에서는 어휘 분석, 구문 분석, 의미 분석, 중간 표현, 최적화, 코드 생성이 하나의 흐름으로 이어진다는 점을 이해하는 것이 핵심이다. 실제 적용에서는 오류를 빨리 잡는 진단 품질, 실행 속도, 메모리 사용량, 이식성이 모두 컴파일러 설계의 중요한 기준이 된다. 예를 들어 같은 프로그램도 어떤 최적화를 적용하느냐에 따라 실행 시간과 전력 소모가 크게 달라질 수 있다. 2026년 3월 기준 최근 흐름은 웹어셈블리 표준 확장, 다단계 중간 표현의 확산, 인공지능 가속기용 컴파일러 발전, 형식적 명세 강화처럼 정확성과 이식성을 함께 높이는 방향으로 정리할 수 있다.

## 소스 코드가 실행 파일로 바뀌는 전체 경로

- 컴파일러의 출발점은 소스 코드를 다른 표현으로 옮기는 번역이며, 씨 언어 프로그램을 실행 파일로 만드는 과정이 대표적 예시다.
- 전처리 단계는 매크로 치환과 파일 포함을 처리하므로, 문법 오류가 아니라 포함 순서 문제를 먼저 만드는 경우가 많다는 점이 실무적으로 중요하다.
- 어휘 분석은 글자를 토큰으로 묶고, 구문 분석은 토큰 구조를 트리로 만든다; 즉 숫자와 기호를 알아보는 일과 문장 구조를 이해하는 일은 다르다.
- 의미 분석은 타입과 선언 범위를 검사하므로, 문법은 맞지만 정수에 문자열을 더하는 코드 같은 오류를 잡아낸다.
- 중간 표현은 최적화를 쉽게 하려고 쓰는 내부 언어인데, 원본 언어와 기계어 사이의 완충지대라고 보면 이해가 쉽다.
- 코드 생성 단계는 목표 기계의 명령어로 바꾸는 과정이라서, 같은 알고리즘도 중앙처리장치 종류에 따라 결과 코드가 달라진다.
- 마지막 연결 단계는 여러 목적 파일과 라이브러리를 하나로 묶는데, 함수는 맞게 작성했어도 심볼을 못 찾으면 여기서 실패한다.

## 어휘 분석과 구문 분석을 분리해서 보는 이유

- 어휘 분석은 문자열을 식별자, 숫자, 연산자 같은 토큰으로 자르며, 예를 들어 `합계=10+20`을 의미 있는 조각으로 나눈다.
- 정규 표현식은 어휘 분석기에 잘 맞지만, 중첩 괄호처럼 계층 구조가 필요한 문법은 구문 분석기가 맡는다는 비교가 중요하다.
- 구문 분석은 문법 규칙에 따라 트리를 만들기 때문에, `3+4*5`에서 곱셈이 먼저라는 구조를 반영할 수 있다.
- 왼쪽부터 읽으며 바로 처리하는 방식은 빠르지만 문법 표현력이 제한될 수 있어, 단순 언어와 복잡 언어에서 선택이 달라진다.
- 오류 메시지 품질은 이 단계에서 크게 갈리는데, 좋은 파서는 단순히 실패했다고 하지 않고 어느 토큰 근처가 문제인지 알려준다.
- 실무에서는 자동 생성 파서와 수작업 파서를 섞어 쓰기도 하며, 문법은 생성기로 처리하고 예외 메시지는 수작업으로 다듬는 경우가 많다.
- 어휘 분석과 구문 분석을 분리하면 구현이 모듈화되어, 새로운 예약어를 추가할 때 전체 컴파일러를 다시 설계하지 않아도 된다.

## 의미 분석에서 타입과 범위를 검사하는 방법

- 의미 분석은 프로그램이 문법적으로 맞는지를 넘어서 뜻이 맞는지를 보는 단계이며, 선언되지 않은 변수 사용이 대표적인 오류다.
- 심볼 테이블은 변수, 함수, 타입 정보를 저장하는 표로, 중첩 블록 안에서 같은 이름이 가려지는 현상을 설명할 때 유용하다.
- 정적 타입 검사는 실행 전에 오류를 찾으므로 안정성이 높고, 동적 검사는 유연하지만 실행 중 실패 가능성이 크다는 비교가 자주 나온다.
- 함수 호출 검사는 인자 개수와 타입을 확인하므로, `최대값(3)`처럼 매개변수가 둘이어야 하는 함수를 잘못 부르면 여기서 잡힌다.
- 형변환 규칙은 언어 성격을 드러내는데, 자동 형변환이 많으면 편하지만 예상 못 한 정밀도 손실이 생길 수 있다.
- 배열 인덱스나 반환문 검사처럼 흐름과 문맥을 함께 보는 규칙도 많아서, 단순 문법 트리만으로는 충분하지 않다.
- 실무적으로 의미 분석 품질이 좋으면 개발자는 실행 전에 더 많은 버그를 줄일 수 있어, 생산성과 신뢰성이 함께 올라간다.

## 중간 표현이 최적화의 중심이 되는 이유

- 중간 표현은 여러 언어의 앞부분과 여러 기계의 뒷부분을 연결하는 공통 층으로, 하나의 최적화를 여러 언어에 재사용하게 해 준다.
- 세 주소 코드 같은 형태는 `a+b*c`를 작은 단계로 나누어 표현하므로, 공통 부분식 제거 같은 최적화가 쉬워진다.
- 정적 단일 대입 형태는 변수 값을 한 번만 정의하게 만들어 데이터 흐름을 분석하기 좋고, 최적화 알고리즘 설명에 자주 등장한다.
- 고수준 중간 표현은 배열과 반복 구조를 잘 보존해 인공지능 연산 최적화에 유리하고, 저수준 중간 표현은 기계 명령어와 더 가깝다.
- 같은 소스라도 중간 표현 설계가 다르면 최적화 가능 범위가 달라져, 컴파일러 성능 차이가 단순 구현 속도만의 문제가 아님을 보여 준다.
- 실제 도구에서는 프런트엔드가 언어별 중간 표현을 만들고 미들엔드가 이를 정리한 뒤 백엔드가 기계별 코드로 내보낸다.
- 시험에서는 중간 표현을 단순한 내부 자료구조로 외우기보다, 언어 독립성과 최적화 효율을 동시에 얻는 장치로 이해하는 편이 낫다.

## 최적화가 속도와 메모리를 어떻게 바꾸는가

- 최적화는 무조건 빠르게 만드는 마법이 아니라 목표에 따라 속도, 코드 크기, 전력 소모 사이에서 균형을 잡는 작업이다.
- 상수 접기는 `3*4`를 미리 `12`로 바꾸는 식이라 가장 직관적이며, 계산을 실행 시간에서 컴파일 시간으로 옮기는 예시다.
- 죽은 코드 제거는 절대 실행되지 않는 분기를 없애 코드 크기와 분석 복잡도를 줄이는데, 디버깅용 코드가 사라질 수 있다는 점도 알아야 한다.
- 루프 최적화는 반복문 성능에 큰 영향을 주며, 배열 합산처럼 같은 연산이 많을수록 효과가 커진다.
- 인라인 확장은 함수 호출 비용을 줄이지만 코드 크기를 늘릴 수 있어, 작은 함수에서는 이익이고 큰 함수에서는 손해일 수 있다.
- 공통 부분식 제거는 같은 계산을 한 번만 하게 만들어 효율을 높이는데, 부작용이 있는 함수 호출에는 함부로 적용하면 안 된다.
- 실무에서는 최고 최적화 옵션이 항상 최선이 아니며, 빌드 시간 증가와 디버깅 어려움까지 함께 판단해야 한다.

## 코드 생성과 레지스터 배치에서 성능이 갈린다

- 코드 생성은 중간 표현을 실제 기계 명령어로 바꾸는 단계이며, 같은 덧셈도 목표 구조에 따라 다른 명령 조합이 나온다.
- ��령 선택은 어떤 기계 명령을 쓸지 고르는 일이라, 곱셈을 쉬프트로 바꿀 수 있는 경우처럼 하드웨어 특성을 반영한다.
- 레지스터 배치는 자주 쓰는 값을 빠른 저장 공간에 놓는 과정인데, 레지스터가 부족하면 메모리에 내보내 성능이 떨어진다.
- 그래프 색칠 같은 배치 기법은 이론 수업에서 자주 나오며, 실제로는 컴파일 시간과 결과 품질 사이 절충이 필요하다.
- 명령 스케줄링은 독립적인 명령 순서를 바꿔 파이프라인 대기를 줄이는데, 현대 프로세서일수록 효과가 커진다.
- 호출 규약은 함수 인자를 어디에 둘지 정하는 약속이라서, 서로 다른 언어로 만든 모듈을 연결할 때 특히 중요하다.
- 실무적으로 백엔드 품질이 좋으면 같은 알고리즘도 더 적은 전력으로 더 빨리 실행되므로, 임베디드와 고성능 계산에서 차이가 크게 난다.

## 사전 컴파일, 즉시 컴파일, 바이트코드 실행의 차이

- 사전 컴파일은 실행 전에 기계어를 만들어 두는 방식이라 시작 속도가 빠르고 배포 결과가 안정적이라는 장점이 있다.
- 즉시 컴파일은 실행 중 자주 쓰이는 코드를 최적화하므로, 프로그램 실제 사용 패턴에 맞춘 최적화가 가능하다.
- 바이트코드 방식은 이식성이 좋다; 한 번 만든 중간 코드를 여러 환경에서 가상 머신이 실행할 수 있기 때문이다.
- 사전 컴파일은 초기 빌드 시간이 늘 수 있지만 서버 배포처럼 예측 가능한 성능이 필요한 환경에 잘 맞는다.
- 즉시 컴파일은 긴 실행 시간의 응용에서 강한데, 자주 반복되는 경로를 관찰한 뒤 더 공격적으로 최적화할 수 있다.
- 바이트코드 해석은 구현이 단순하지만 순수 기계어 실행보다 느릴 수 있어, 실제 시스템은 해석과 즉시 컴파일을 섞는 경우가 많다.
- 학습 포인트는 세 방식의 우열이 아니라, 시작 지연·최고 성능·이식성 중 무엇을 더 중시하느냐에 따라 선택이 달라진다는 점이다.

## 최신 흐름

- 웹어셈블리 표준은 2025년 9월 17일에 3.0 공개 완료가 발표되어, 브라우저 밖 실행과 이식 가능한 배포 형식으로서 의미가 더 커졌다.
- 웹어셈블리 명세는 살아 있는 표준 방식으로 갱신되고 있어, 컴파일러는 한 번 끝나는 도구보다 지속적으로 표준 변화를 따라가는 인프라가 되고 있다.
- 형식적 명세 도구 채택이 강화되면서 언어 표준을 기계적으로 검증하려는 흐름이 커졌고, 이는 컴파일러 정확성 연구와 직접 연결된다.
- 다단계 중간 표현이 널리 쓰이면서 고수준 최적화와 저수준 코드 생성을 한 체계 안에서 다루는 방식이 주류로 자리 잡고 있다.
- 인공지능 가속기용 컴파일러는 행렬 연산, 텐서 분할, 메모리 이동 최적화가 핵심이어서 전통적 중앙처리장치 중심 컴파일러와 관심사가 달라지고 있다.
- 오픈엑스엘에이 계열 생태계처럼 프레임워크와 하드웨어 사이를 잇는 공통 컴파일 계층이 중요해져, 특정 장치 종속성을 줄이는 방향이 강해졌다.
- 프로파일링과 성능 분석을 중간 표현 단계에 더 깊게 통합하려는 움직임이 늘어, 최적화가 사후 측정이 아니라 설계 단계부터 반영되고 있다.
- 안전성과 이식성을 중시하는 언어들의 성장으로, 경계 검사 제거와 별칭 분석처럼 성능과 안전을 함께 다루는 최적화가 더 중요해졌다.
- 브라우저, 서버, 임베디드, 인공지능 추론 환경을 하나의 코드 기반으로 연결하려는 수요가 커져, 크로스 컴파일과 공통 백엔드 가치가 다시 커지고 있다.
- 결과적으로 최근 컴파일러 흐름은 더 빠른 코드 생성만이 아니라, 정확한 명세, 다양한 하드웨어 지원, 관찰 기반 최적화, 장기 유지보수성까지 함께 추구하는 방향이다.
