## 핵심 요약

데이터베이스는 데이터를 구조화해 저장하고, 빠르고 안전하게 조회·수정하도록 돕는 핵심 기술이다. 학부 수준에서는 관계형 모델, 정규화, 질의 처리, 인덱스, 트랜잭션을 함께 이해해야 실제 시스템 동작이 보인다. 예를 들어 쇼핑몰 주문 처리에서는 테이블 설계가 잘못되면 중복이 늘고, 인덱스가 부적절하면 검색 속도가 급격히 느려진다. 실무에서는 단순히 명령문을 작성하는 것보다 데이터 모델링과 동시성 제어, 성능 튜닝을 함께 보는 관점이 중요하다. 2026-04-11 기준 최근 흐름은 벡터 검색, 클라우드 관리형 서비스, 실시간 분석 처리처럼 전통적 관계형 기술 위에 새로운 활용 방식이 더해지는 방향으로 이어지고 있다.

## 관계형 모델과 테이블 설계의 출발점

- 관계형 모델은 데이터를 행과 열의 표 형태로 표현하는 방식이며, 학생 테이블과 수강 테이블을 분리하면 의미가 더 명확해진다.
- 속성은 열, 튜플은 행을 뜻하며, 같은 학생 정보라도 학번을 기준으로 구별하면 중복 이름 문제를 줄일 수 있다.
- 기본 키는 각 행을 유일하게 식별하는 값으로, 주민번호 대신 학번을 쓰는 것은 변경 가능성이 낮아서 실무적으로 유리하다.
- 외래 키는 테이블 간 연결을 표현하며, 주문 테이블의 회원 번호가 회원 테이블을 참조하면 잘못된 주문 데이터 입력을 막기 쉽다.
- 도메인은 속성이 가질 수 있는 값의 범위로, 점수는 0부터 100까지만 허용하는 식의 제약이 품질을 높인다.
- 널 값은 값이 없음을 의미하지만 0이나 빈 문자열과 다르므로, 전화번호 미입력과 번호 없음은 구분해서 해석해야 한다.
- 스키마는 구조의 설계도이고 인스턴스는 현재 저장된 실제 데이터이므로, 같은 스키마라도 학기마다 데이터 상태는 달라진다.

## 정규화로 중복과 이상 현상 줄이기

- 정규화는 데이터를 잘게 나누어 중복을 줄이는 과정이며, 한 테이블에 학생과 학과 사무실 정보를 함께 넣으면 수정 이상이 생기기 쉽다.
- 제1정규형은 한 칸에 하나의 값만 두는 규칙으로, 취미를 축구, 독서처럼 쉼표로 묶기보다 별도 행으로 분리하는 편이 낫다.
- 제2정규형은 부분 함수 종속을 제거하는 단계로, 복합 키 일부에만 의존하는 정보를 떼어내면 테이블 의미가 또렷해진다.
- 제3정규형은 이행 함수 종속을 제거하는 단계이며, 학과 번호로 학과 이름과 건물 위치가 함께 정해진다면 학과 테이블로 분리하는 것이 적절하다.
- 정규화를 너무 강하게 하면 조인이 늘어 조회가 복잡해질 수 있어, 읽기 중심 서비스에서는 일부 반정규화를 쓰기도 한다.
- 삽입 이상은 아직 학생이 없으면 학과 정보를 넣지 못하는 문제처럼 나타나며, 테이블 분리로 해결하는 경우가 많다.
- 삭제 이상은 마지막 수강생이 삭제되며 과목 정보까지 사라지는 상황으로, 개체별 테이블 분리가 실무에서 특히 중요하다.

## 질의 언어로 원하는 데이터 정확히 뽑기

- 질의 언어는 원하는 데이터를 선언적으로 요청하는 수단이며, 절차를 일일이 적지 않아도 조건만 말하면 된다는 점이 일반 프로그램 코드와 다르다.
- 선택은 행을 거르는 연산으로, 평점이 4.0 이상인 학생만 찾는 작업에 대응한다.
- 투영은 필요한 열만 남기는 연산이며, 이름과 이메일만 출력하면 개인정보 노출을 줄이는 데도 도움이 된다.
- 조인은 여러 테이블을 연결해 의미 있는 결과를 만드는 핵심 연산으로, 주문과 회원을 묶어 주문자 이름을 함께 보여줄 수 있다.
- 집계 함수는 개수, 합계, 평균 등을 계산하며, 하루 매출 총액을 구하는 예시에서 자주 쓰인다.
- 그룹화는 같은 기준끼리 묶어 분석하는 방식으로, 학과별 평균 학점을 구하면 전체 평균보다 더 해석하기 쉽다.
- 서브질의는 질의 안에 또 다른 질의를 넣는 방식이며, 전체 평균보다 높은 학생 찾기처럼 비교 조건에 유용하지만 과도하면 가독성이 떨어질 수 있다.

## 인덱스와 저장 구조가 성능을 좌우하는 이유

- 인덱스는 책의 찾아보기처럼 원하는 데이터를 빨리 찾게 해 주며, 학번 검색에서는 전체 스캔보다 훨씬 적은 접근으로 끝날 수 있다.
- 기본 인덱스가 없으면 테이블의 모든 행을 훑는 전체 테이블 스캔이 발생해 데이터가 커질수록 지연이 크게 늘어난다.
- 비트리 계열 인덱스는 범위 검색에 강해서 2026년 1월부터 3월까지 주문 조회처럼 연속 구간 조건에 적합하다.
- 해시 기반 접근은 정확히 일치하는 값 검색에 유리하지만 범위 검색은 약하므로, 이름 동일 검색과 날짜 구간 검색의 성격을 구분해야 한다.
- 복합 인덱스는 여러 열을 함께 정렬해 두는 방식으로, 회원 번호와 주문일을 함께 조회하는 화면에서 효과적일 수 있다.
- 인덱스가 많아지면 조회는 빨라질 수 있지만 삽입과 수정 때 갱신 비용이 커지므로 무조건 많이 만드는 것은 비효율적이다.
- 실행 계획을 확인하면 왜 느린지 파악할 수 있으며, 같은 질의라도 인덱스 유무에 따라 접근 방식이 완전히 달라진다.

## 트랜잭션과 동시성 제어로 데이터 지키기

- 트랜잭션은 하나의 논리적 작업 단위로, 계좌 이체처럼 출금과 입금이 함께 성공하거나 함께 실패해야 하는 작업에 꼭 필요하다.
- 원자성은 전부 수행되거나 전부 취소되는 성질이며, 주문은 저장됐는데 재고 차감은 안 된 상태를 막는 데 중요하다.
- 일관성은 제약 조건이 보존되는 성질로, 재고가 음수가 되지 않아야 한다는 규칙이 대표적인 예다.
- 격리성은 동시에 실행되는 작업이 서로 부적절하게 간섭하지 않도록 하는 성질이며, 같은 좌석을 두 명이 예약하는 문제와 연결된다.
- 지속성은 완료된 결과가 장애 후에도 유지되는 성질로, 결제 완료 기록이 서버 재시작 뒤 사라지면 안 된다는 뜻이다.
- 동시성 이상에는 더티 리드, 반복 불가능 읽기, 팬텀 읽기 등이 있으며, 격리 수준을 높일수록 안전하지만 처리량은 줄 수 있다.
- 잠금 기반 제어는 이해하기 쉽지만 경합이 심할 수 있고, 다중 버전 기반 제어는 읽기 성능에 유리해 현대 시스템에서 널리 쓰인다.

## 복구와 분산 환경에서 신뢰성 확보하기

- 로그는 변경 이력을 순서대로 남기는 기록이며, 장애가 나도 마지막 정상 상태로 복구할 수 있게 해 주는 핵심 장치다.
- 즉시 갱신 방식에서는 로그를 먼저 기록한 뒤 실제 데이터를 반영하는 규칙이 중요하며, 이를 지키지 않으면 복구 근거가 사라질 수 있다.
- 체크포인트는 중간 저장 지점 역할을 하여, 긴 로그 전체를 다시 보지 않고도 복구 시간을 줄여 준다.
- 백업은 복구의 최후 수단으로, 실수로 테이블을 삭제했을 때 로그만으로 부족하면 백업본이 필요하다.
- 복제는 같은 데이터를 여러 서버에 두는 방식이며, 읽기 확장에 유리하지만 지연 복제로 인해 잠깐 다른 결과가 보일 수 있다.
- 샤딩은 데이터를 여러 서버에 나누어 저장하는 기법으로, 사용자 수가 매우 많을 때 확장성이 좋지만 조인이 복잡해질 수 있다.
- 분산 데이터베이스에서는 일관성과 가용성의 균형이 중요하며, 금융 서비스와 사회관계망 서비스는 요구 수준이 서로 다르다.

## 최신 흐름

- 벡터 검색 기능이 데이터베이스 안으로 들어오면서 의미 기반 검색과 추천 기능을 별도 저장소 없이 함께 처리하는 사례가 늘고 있다.
- 생성형 인공지능 응용이 확산되며 문서 조각, 임베딩, 메타데이터를 함께 관리하는 설계가 실무에서 중요한 주제가 되었다.
- 클라우드 관리형 데이터베이스 사용이 일반화되어 설치와 운영보다 비용 관리, 확장 정책, 장애 대응 수준 비교가 더 중요한 선택 기준이 되고 있다.
- 실시간 분석 처리 수요가 커지면서 거래 처리와 분석 처리를 완전히 분리하기보다, 빠른 집계가 가능한 구조를 함께 고려하는 흐름이 강해졌다.
- 변경 데이터 캡처 기반 파이프라인이 널리 쓰이며, 주문 변경 내용을 바로 검색 시스템이나 분석 창고로 보내는 구조가 흔해졌다.
- 다중 모델 지원이 늘어나 관계형 데이터와 문서형 데이터, 그래프형 조회를 한 플랫폼에서 다루려는 시도가 계속되고 있다.
- 개인정보 보호와 규제 대응이 강화되면서 열 단위 암호화, 마스킹, 감사 로그 같은 보안 기능이 기본 요구사항에 가까워졌다.
- 서버리스 방식은 작은 서비스에는 운영 부담을 줄여 주지만, 장시간 연결이나 예측 어려운 비용 측면에서는 전통 방식과 비교 검토가 필요하다.
- 자동 튜닝 기능이 발전해 인덱스 추천과 질의 성능 분석을 도와주지만, 잘못 수용하면 쓰기 비용이 증가하므로 사람의 검토가 여전히 필요하다.
- 2026-04-11 기준 교육 현장에서도 단순 명령문 암기보다 인공지능 응용, 데이터 파이프라인, 분산 저장까지 연결해서 가르치는 방향이 뚜렷하다.
