## 오늘의 면접 주제
> GitHub Actions와 Jenkins가 다시 자주 언급되면서, 배포 자동화와 운영 안정성을 함께 설명하는 CI/CD가 면접에서 더 중요해졌기 때문입니다.

## 핵심 한 줄 요약
CI/CD는 코드 변경을 빠르고 안전하게 검증하고 배포하기 위한 자동화 체계이며, 핵심은 속도보다도 일관성과 실패 대응 능력입니다.

## 면접 질문 & 모범 답변

### Q1. CI/CD란 무엇인가요?
**기본 답변**: CI는 Continuous Integration입니다. 여러 개발자가 올린 코드를 자주 합치고, build와 test를 자동으로 실행해 문제를 빨리 찾는 과정입니다. CD는 Continuous Delivery 또는 Continuous Deployment를 뜻하며, 검증된 코드를 운영 환경까지 안전하게 배포하는 흐름입니다.  
**심화 포인트**: 많은 지원자가 “자동 배포”만 말합니다. 하지만 면접에서는 자동화보다 피드백 속도, 배포 품질, 사람 실수 감소를 함께 설명해야 좋습니다.

### Q2. CI는 왜 필요한가요?
**기본 답변**: 코드 충돌과 숨은 버그는 늦게 발견할수록 수정 비용이 커집니다. CI는 PR이나 push 시점에 test를 돌려서 문제를 초기에 잡습니다. 즉, 큰 장애를 나중에 찾지 말고 작은 오류를 빨리 잡는 장치입니다.  
**심화 포인트**: CI의 핵심은 “테스트 자동 실행”만이 아닙니다. build 가능 여부, lint, dependency 검사까지 포함해 배포 가능한 상태를 계속 유지하는 것이 중요합니다.

### Q3. GitHub Actions와 Jenkins는 어떻게 비교할 수 있나요?
**기본 답변**: GitHub Actions는 GitHub와 강하게 통합되어 있어서 설정이 쉽고, PR 중심 개발에 잘 맞습니다. Jenkins는 플러그인이 많고 커스터마이징이 강력해서 복잡한 사내 환경에 유리합니다. 작은 팀은 Actions가 빠르고, 복잡한 인프라를 직접 통제해야 하면 Jenkins가 더 적합할 수 있습니다.  
**심화 포인트**: 도구 비교에서 중요한 것은 기능 개수보다 운영 비용입니다. Jenkins는 유연하지만 서버 관리, plugin 충돌, 보안 업데이트 책임도 같이 따라옵니다.

### Q4. 실제로 CI/CD 파이프라인은 어떤 순서로 구성하겠습니까?
**기본 답변**: 보통 `checkout -> build -> test -> package -> deploy` 순서로 설계합니다. 예를 들어 PR 단계에서는 unit test와 lint만 빠르게 돌리고, main merge 후에는 image build와 staging 배포까지 진행할 수 있습니다. 운영 배포 전에는 승인 단계나 health check를 둘 수 있습니다.  
**심화 포인트**: 모든 단계를 한 번에 무겁게 돌리면 느려집니다. 면접에서는 브랜치 전략과 단계별 검증 수준을 나눠 설명하면 실무 감각이 좋아 보입니다.

### Q5. Blue/Green 배포는 무엇이고 왜 쓰나요?
**기본 답변**: Blue/Green은 현재 서비스 중인 환경과 새 버전 환경을 분리해서 운영하는 방식입니다. 새 버전을 Green에 올리고 확인한 뒤, 트래픽만 전환합니다. 문제가 생기면 다시 Blue로 돌리기 쉬워서 rollback이 빠릅니다.  
**심화 포인트**: 장점만 말하면 부족합니다. 환경을 두 벌 운영해야 해서 비용이 들고, DB schema 변경이 호환되지 않으면 전환이 어렵다는 점도 알아야 합니다.

### Q6. Canary 배포는 무엇이고 Blue/Green과 어떻게 다르나요?
**기본 답변**: Canary는 새 버전을 일부 사용자에게만 먼저 노출하는 방식입니다. 예를 들어 5% 트래픽만 새 버전으로 보내고 에러율을 본 뒤 점진적으로 늘립니다. Blue/Green이 한 번에 전환하는 느낌이라면, Canary는 조금씩 확인하며 확장하는 방식입니다.  
**심화 포인트**: Canary의 핵심은 배포 자체보다 관측입니다. 로그, 메트릭, 알람이 없으면 조금씩 배포해도 문제를 늦게 발견할 수 있습니다.

### Q7. CI/CD를 도입했는데 배포 실패가 자주 난다면 어떻게 개선하겠습니까?
**기본 답변**: 먼저 실패 지점을 나눠 봐야 합니다. test 실패인지, 환경 변수 문제인지, 배포 후 health check 실패인지 구분해야 합니다. 그 다음 재현 가능한 배포 스크립트, secret 관리, rollback 자동화부터 정리합니다.  
**심화 포인트**: 주니어 면접에서는 “다시 해본다” 수준에서 멈추기 쉽습니다. 면접관은 로그 기반 원인 분석과 실패를 시스템적으로 줄이는 접근을 봅니다.

### Q8. CI/CD에서 가장 중요한 품질 기준은 무엇이라고 생각하나요?
**기본 답변**: 저는 재현성과 가시성이 가장 중요하다고 봅니다. 같은 커밋이면 누구나 같은 결과가 나와야 하고, 실패했을 때 어디서 왜 실패했는지 바로 보여야 합니다. 그래야 자동화가 팀 속도를 올리지, 오히려 불신을 만들지 않습니다.  
**심화 포인트**: “배포 속도”만 강조하면 위험합니다. 빠른 배포보다 중요한 것은 예측 가능한 배포와 빠른 복구입니다.

## 헷갈리기 쉬운 포인트
CI/CD는 무조건 운영까지 자동 배포하는 개념이 아닙니다. Continuous Delivery는 배포 가능한 상태까지 자동화하고, 최종 배포는 사람 승인일 수도 있습니다.  
GitHub Actions가 Jenkins보다 “무조건 더 좋다”는 식의 답변은 위험합니다. 팀 규모, 보안 요구사항, 자체 인프라 통제 수준에 따라 선택이 달라집니다.  
Blue/Green과 Canary는 둘 다 무중단 배포에 쓰이지만 방식이 다릅니다. 하나는 환경 전환 중심이고, 다른 하나는 트래픽 점진 분산 중심입니다.

## 면접관 시각
이 주제에서는 정의 암기보다 운영 감각을 봅니다. 지원자가 자동화의 목적을 장애 예방, 빠른 복구, 사람 실수 감소로 연결해서 설명하는지 확인합니다. 도구 이름을 많이 아는지보다, 왜 그 도구를 선택하고 실패 시 어떻게 대응할지 말할 수 있는지가 더 중요합니다.
