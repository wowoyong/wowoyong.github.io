## 핵심 요약

데이터베이스 인덱스는 테이블 전체를 탐색하지 않고도 원하는 데이터를 빠르게 찾을 수 있게 해주는 자료 구조로, 책의 색인과 유사한 역할을 합니다. 가장 널리 쓰이는 B+트리 인덱스는 디스크 입출력을 최소화하도록 설계되었으며, 해시 인덱스는 등가 검색에 특화되어 있습니다. 인덱스를 잘못 설계하면 오히려 쓰기 성능이 저하되고 저장 공간이 낭비되므로, 선택도와 카디널리티를 고려한 설계가 중요합니다. 실무에서는 복합 인덱스의 컬럼 순서, 커버링 인덱스 활용, 실행 계획 분석이 성능 튜닝의 핵심입니다. 최근에는 학습 기반 인덱스, 부분 인덱스, 칼럼 지향 저장소의 인덱스 전략 등 새로운 접근법이 활발히 연구되고 있습니다.

## 인덱스가 필요한 이유와 전체 탐색의 비용

- 인덱스 없이 조회하면 테이블 전체를 순차 탐색해야 하며, 100만 행 테이블에서 1건을 찾아도 최악의 경우 100만 번 비교가 필요합니다.
- 디스크 기반 데이터베이스에서 전체 탐색의 병목은 연산 처리가 아니라 디스크 입출력이며, 하드디스크 한 번의 무작위 읽기는 약 10밀리초가 소요됩니다.
- 인덱스는 정렬된 키와 포인터를 별도로 저장해 검색 경로를 줄이는 구조로, 책 뒤의 색인 페이지처럼 전체를 읽지 않고 원하는 항목을 찾게 해줍니다.
- 예를 들어 학생 테이블에서 학번으로 검색할 때, 인덱스가 있으면 로그 수준으로 탐색 범위가 줄어듭니다.
- 다만 인덱스 자체도 저장 공간을 차지하고, 삽입·수정·삭제 시 인덱스도 함께 갱신해야 하므로 쓰기 비용이 증가합니다.
- 읽기 비율이 높은 시스템(웹 서비스 등)에서는 인덱스의 이점이 크지만, 대량 삽입이 빈번한 로그 시스템에서는 신중하게 설계해야 합니다.
- 실무에서 '느린 쿼리'의 80% 이상은 적절한 인덱스 추가만으로 해결되며, 이것이 성능 튜닝의 첫 단계입니다.

## B+트리 인덱스의 구조와 디스크 최적화 원리

- B+트리는 균형 탐색 트리의 일종으로, 모든 리프 노드가 같은 깊이에 위치해 최악의 경우에도 일정한 탐색 시간을 보장합니다.
- 내부 노드에는 키와 자식 포인터만 저장하고, 실제 데이터 포인터는 리프 노드에만 있어서 내부 노드 하나에 더 많은 키를 담을 수 있습니다.
- 이진 탐색 트리와 비교하면, B+트리는 한 노드에 수백 개의 키를 저장해 트리 높이를 3~4단계로 유지하므로 디스크 접근 횟수가 크게 줄어듭니다.
- 리프 노드끼리 연결 리스트로 이어져 있어 범위 검색(예: 가격이 1만~5만 원인 상품)을 효율적으로 처리합니다.
- 노드 크기는 보통 디스크의 페이지 크기(4KB~16KB)에 맞춰 설계되며, 한 번의 입출력으로 노드 하나를 통째로 읽습니다.
- 1000만 건의 데이터에서도 B+트리 높이가 3~4 정도이므로, 최대 4번의 디스크 읽기로 원하는 행을 찾을 수 있습니다.
- 삽입·삭제 시 노드 분할과 병합이 발생하지만, 이 연산도 로그 복잡도로 제한됩니다.

## 해시 인덱스와 B+트리의 용도별 차이

- 해시 인덱스는 해시 함수로 키를 버킷에 매핑하며, 등가 검색(WHERE id = 42)에서 상수 시간 수준의 성능을 제공합니다.
- 반면 범위 검색(WHERE age > 20)이나 정렬(ORDER BY)에는 사용할 수 없는데, 해시 함수가 키의 순서를 보존하지 않기 때문입니다.
- MySQL의 메모리 엔진은 해시 인덱스를 기본으로 사용하고, InnoDB는 적응형 해시 인덱스를 내부적으로 자동 생성합니다.
- PostgreSQL은 해시 인덱스를 지원하지만, 버전 10 이전에는 WAL 로깅이 안 되어 장애 복구 시 문제가 있었습니다.
- 해시 충돌이 많아지면 같은 버킷에 여러 키가 몰려 성능이 저하되므로, 카디널리티가 높은(고유 값이 많은) 컬럼에 적합합니다.
- 실무에서는 대부분 B+트리를 기본으로 사용하고, 캐시 테이블이나 세션 저장소처럼 정확한 키 매칭만 필요한 경우에 해시를 고려합니다.
- B+트리는 범용성이 높아 '기본 인덱스 = B+트리'라고 이해해도 무방하며, 특수한 요구가 있을 때만 다른 유형을 검토합니다.

## 클러스터드 인덱스와 보조 인덱스의 저장 방식 비교

- 클러스터드 인덱스는 테이블 데이터 자체를 인덱스 키 순서로 물리적으로 정렬해 저장하는 방식으로, 테이블당 하나만 존재할 수 있습니다.
- MySQL InnoDB에서는 기본 키가 자동으로 클러스터드 인덱스가 되며, 기본 키 순서대로 데이터가 디스크에 배치됩니다.
- 보조 인덱스는 인덱스 키와 함께 클러스터드 인덱스의 기본 키 값을 저장하며, 실제 데이터를 찾으려면 기본 키 인덱스를 한 번 더 탐색합니다.
- 이 이중 탐색을 '북마크 탐색'이라 하며, 보조 인덱스 검색이 클러스터드 인덱스보다 느린 핵심 이유입니다.
- PostgreSQL은 힙 테이블 구조를 사용해 클러스터드 인덱스 개념이 다르며, CLUSTER 명령으로 일회성 정렬만 가능합니다.
- UUID를 기본 키로 사용하면 삽입 시 무작위 위치에 데이터가 들어가 페이지 분할이 빈번해지므로, 순차 증가 정수형이 클러스터드 인덱스에 유리합니다.
- 실무에서 자동 증가 정수 기본 키를 선호하는 이유 중 하나가 바로 클러스터드 인덱스의 삽입 성능 때문입니다.

## 복합 인덱스 설계와 컬럼 순서의 중요성

- 복합 인덱스는 두 개 이상의 컬럼을 조합해 하나의 인덱스를 만들며, (성별, 나이)처럼 자주 함께 조회되는 컬럼에 사용합니다.
- 최좌선 접두사 규칙에 따라 (A, B, C) 인덱스는 A 단독, (A, B), (A, B, C) 검색에는 사용되지만 B 단독이나 (B, C) 검색에는 사용되지 않습니다.
- 선택도가 높은(고유 값이 많은) 컬럼을 앞에 배치하면 초기 탐색에서 후보군이 크게 줄어 효율적입니다.
- 예를 들어 (도시, 성별)보다 (성별, 도시)가 비효율적인데, 성별은 2가지뿐이라 첫 단계에서 절반밖에 걸러내지 못합니다.
- 등가 조건(=)에 사용되는 컬럼을 앞에, 범위 조건(>, <, BETWEEN)에 사용되는 컬럼을 뒤에 배치하는 것이 일반적인 최적화 전략입니다.
- 커버링 인덱스는 쿼리에 필요한 모든 컬럼이 인덱스에 포함되어 테이블 접근 없이 인덱스만으로 결과를 반환하는 기법입니다.
- MySQL에서 EXPLAIN 결과의 부가 정보 컬럼에 '인덱스만 사용'이 표시되면 커버링 인덱스가 적용된 것이며, 이는 가장 빠른 조회 방식입니다.
- 인덱스를 무분별하게 추가하면 쓰기 성능 저하와 저장 공간 낭비가 발생하므로, 실제 쿼리 패턴을 분석한 후 설계해야 합니다.

## 실행 계획 분석으로 인덱스 활용 여부 확인하기

- EXPLAIN(MySQL) 또는 EXPLAIN ANALYZE(PostgreSQL) 명령으로 쿼리 최적화기가 선택한 실행 계획을 확인할 수 있습니다.
- 실행 계획에서 '전체 탐색'이 보이면 테이블 전체를 읽고 있다는 뜻이며, 인덱스 추가를 검토해야 합니다.
- 인덱스 참조 탐색은 인덱스를 사용한 조회를 의미하며, 범위 탐색은 인덱스를 이용한 구간 검색입니다.
- 쿼리 최적화기는 통계 정보를 기반으로 인덱스 사용 여부를 판단하며, 테이블의 30% 이상을 읽어야 한다면 오히려 전체 탐색을 선택합니다.
- WHERE 절에서 컬럼에 함수를 적용하면(예: WHERE YEAR(created_at) = 2026) 인덱스를 사용하지 못하므로, WHERE created_at >= '2026-01-01'로 변환해야 합니다.
- 묵시적 형변환도 인덱스를 무력화하는 원인으로, 문자열 컬럼을 숫자와 비교하면(WHERE phone = 01012345678) 인덱스가 무시됩니다.
- 느린 쿼리 로그를 활성화하면 일정 시간 이상 걸리는 쿼리를 자동 수집할 수 있어, 인덱스 최적화 대상을 쉽게 파악할 수 있습니다.

## 특수 인덱스: 부분, 함수 기반, 전문 검색

- 부분 인덱스(PostgreSQL)는 WHERE 조건을 만족하는 행만 인덱싱하는 것으로, 예를 들어 활성 사용자만 인덱싱하면 인덱스 크기를 크게 줄일 수 있습니다.
- 함수 기반 인덱스는 LOWER(email)처럼 연산 결과를 인덱싱해, 대소문자 무시 검색에서 인덱스를 활용할 수 있게 합니다.
- 전문 검색 인덱스는 긴 텍스트에서 단어 단위 검색을 지원하며, LIKE '%검색어%'보다 훨씬 빠릅니다.
- 공간 인덱스(R-트리 기반)는 좌표, 다각형 등 지리 데이터를 효율적으로 검색하며, '반경 1km 내 음식점 찾기' 같은 쿼리에 사용됩니다.
- 비트맵 인덱스는 카디널리티가 낮은(값의 종류가 적은) 컬럼에 적합하며, 데이터 웨어하우스에서 주로 사용됩니다.
- GIN 인덱스(PostgreSQL)는 JSON, 배열, 전문 검색 등 다중 값 데이터 타입에 특화되어 있습니다.
- 각 인덱스 유형은 특정 쿼리 패턴에 최적화되어 있으므로, 데이터 특성과 접근 패턴에 맞는 인덱스를 선택하는 것이 핵심입니다.

## 최신 흐름

- 학습 기반 인덱스는 기계 학습 모델로 키의 분포를 학습해 B+트리보다 빠른 탐색과 적은 메모리 사용을 목표로 연구되고 있습니다.
- LSM 트리(로그 구조 병합 트리)는 쓰기 집약적 워크로드에 최적화된 구조로, RocksDB, Cassandra 등 현대 분산 데이터베이스의 기반입니다.
- 칼럼 지향 데이터베이스(ClickHouse, DuckDB 등)에서는 전통적 B+트리 대신 존 맵, 희소 인덱스 등 대용량 분석에 적합한 인덱스를 사용합니다.
- PostgreSQL 15 이후 BRIN(블록 범위 인덱스) 성능이 개선되어, 시계열 데이터처럼 물리적으로 정렬된 대용량 테이블에서 효율적입니다.
- MySQL 8.0의 보이지 않는 인덱스 기능으로 인덱스를 삭제하지 않고 비활성화해 성능 영향을 안전하게 테스트할 수 있습니다.
- 다중 모델 데이터베이스(예: SurrealDB)에서는 문서, 그래프, 관계형 데이터를 통합 인덱싱하는 새로운 접근법이 등장하고 있습니다.
- 클라우드 네이티브 데이터베이스(Aurora, AlloyDB 등)는 저장소와 컴퓨팅을 분리해 인덱스 재구축 없이 확장이 가능합니다.
- 벡터 인덱스(HNSW, IVFFlat)는 인공지능 임베딩 검색에 필수적이며, pgvector를 통해 PostgreSQL에서도 유사도 검색이 가능합니다.
- 자동 인덱스 튜닝 도구(예: MySQL의 인덱스 힌트 자동화, Azure의 자동 인덱스 권장)가 발전하면서 수동 인덱스 설계의 부담이 줄고 있습니다.
- 인메모리 데이터베이스(Redis, VoltDB)에서는 디스크 입출력 최적화가 불필요하므로 해시 인덱스와 스킵 리스트 같은 메모리 친화적 구조를 선호합니다.
