## 핵심 요약

컴퓨터구조는 프로그램이 실제 하드웨어에서 어떻게 실행되는지 이해하는 학문으로, 중앙처리장치, 메모리, 입출력장치, 버스가 서로 어떻게 상호작용하는지를 함께 살핀다. 학습의 핵심은 명령어 실행 흐름, 메모리 계층, 파이프라인, 명령어 집합 구조를 연결해 성능과 전력 사이의 균형을 읽는 데 있다. 실무에서는 같은 알고리즘이라도 캐시에 잘 맞는 접근인지, 분기 예측에 불리한지, 메모리 병목이 큰지에 따라 체감 속도가 크게 달라진다. 그래서 컴퓨터구조는 단순히 하드웨어를 외우는 과목이 아니라, 프로그램 최적화와 시스템 설계를 이해하는 공통 언어로 쓰인다. 2026년 기준 최근 흐름은 인공지능 수요에 맞춘 고대역폭 메모리, 칩렛, 개방형 명령어 집합 구조, 메모리 확장 연결 기술처럼 시스템 전체를 다시 설계하는 방향으로 움직이고 있다. 기초 개념을 정확히 익혀 두면 모바일 기기, 서버, 자율주행, 엣지 장치처럼 환경이 달라도 구조적으로 판단하기가 쉬워진다.

## 저장 프로그램 방식과 명령어 실행 사이클 이해

- 저장 프로그램 방식은 프로그램과 데이터를 같은 주기억장치에 두는 구조로, 코드를 고치고 다시 실행하기 쉬워 범용 컴퓨터의 출발점이 되었다.
- 명령어 실행 사이클은 보통 인출, 해독, 실행, 기록 단계로 보며, 계산기 앱에서 덧셈 버튼을 눌러도 내부적으로는 이 흐름이 반복된다.
- 프로그램 계수기는 다음에 읽을 명령어의 주소를 가리키며, 반복문에서는 순차적으로 증가하고 분기문에서는 갑자기 다른 주소로 바뀐다.
- 명령어 레지스터는 현재 처리 중인 명령어를 잠시 담아 해독을 돕는데, 요리할 때 현재 단계가 적힌 쪽지를 손에 들고 있는 모습과 비슷하다.
- 메모리 주소와 데이터가 분리되어 움직인다는 점을 이해하면, 값 자체보다 값을 어디서 읽어오느냐가 성능에 더 중요할 수 있다는 점을 파악할 수 있다.
- 같은 덧셈 연산도 레지스터에 이미 값이 있으면 빠르고, 주기억장치에서 다시 읽어와야 하면 느려져서 자료 배치가 실행 시간에 직접 영향을 준다.
- 저장 프로그램 방식의 장점은 유연성이지만, 명령어와 데이터가 같은 통로를 함께 써 병목이 생기기 쉬우므로 이후 캐시와 파이프라인이 보완책으로 등장했다.

## 중앙처리장치 내부: 레지스터, 연산장치, 제어장치의 역할 분담

- 레지스터는 중앙처리장치 내부에서 가장 빠른 저장공간으로, 변수 값을 주기억장치에 두는 것보다 훨씬 빠르기 때문에 컴파일러 최적화의 핵심 대상이 된다.
- 산술논리연산장치는 덧셈, 뺄셈, 비교, 비트 연산을 담당하며, 예를 들어 배열 인덱스를 계산하는 일도 결국 여기서 처리된다.
- 제어장치는 명령어 해독 결과에 따라 어떤 회로를 언제 활성화할지 결정하는데, 교통경찰이 각 차선을 번갈아 통제하는 모습과 비슷하다.
- 범용 레지스터는 다양한 계산에 쓰이고, 특수 레지스터는 프로그램 계수기나 상태 레지스터처럼 특정 목적에만 쓰여 역할이 분명히 나뉜다.
- 상태 레지스터의 0 여부, 올림 여부, 음수 여부 같은 정보는 조건 분기에 쓰이므로, 단순한 계산 결과보다 다음 제어 흐름에 더 큰 영향을 주기도 한다.
- 고정 제어 방식은 빠르지만 바꾸기 어렵고, 마이크로프로그램 제어 방식은 유연하지만 상대적으로 복잡해져서 설계 목표에 따라 선택이 달라진다.
- 실무적으로는 중앙처리장치 성능을 볼 때 클록 수치만 볼 것이 아니라, 레지스터 수, 실행 유닛 수, 분기 처리 능력까지 함께 봐야 실제 체감과 맞는다.

## 메모리 계층과 지역성: 왜 캐시 친화성이 중요한가

- 메모리 계층은 레지스터, 캐시, 주기억장치, 보조기억장치 순으로 내려가며 용량은 커지고 속도는 느려져, 크고 저렴한 저장공간과 빠른 저장공간을 함께 쓰는 구조를 이룬다.
- 시간 지역성은 최근 사용한 데이터를 다시 쓸 가능성이 크다는 뜻으로, 반복문 안의 누산 변수는 이 성질 덕분에 캐시에 잘 남는다.
- 공간 지역성은 가까운 주소를 함께 쓸 가능성이 크다는 뜻으로, 배열을 순차적으로 탐색하면 빠르지만 연결 리스트를 여기저기 따라가면 느려지기 쉽다.
- 캐시 적중률이 높으면 중앙처리장치가 주기억장치를 기다리는 시간이 줄어드는데, 같은 코드라도 자료구조 선택에 따라 속도가 크게 달라질 수 있다.
- 직접 사상, 집합 연관, 완전 연관은 캐시 배치 방식의 차이로, 단순함과 충돌 감소 사이에서 설계 균형이 다르다.
- 가상 메모리는 큰 주소 공간을 제공하지만 페이지 부재가 자주 발생하면 보조기억장치 접근 때문에 급격히 느려져, 용량만 크다고 좋은 것은 아니다.
- 실제로 행 우선 저장 배열을 열 기준으로 순회하면 캐시 실패가 늘어 성능이 떨어지므로, 자료 접근 순서를 바꾸는 것만으로도 최적화가 가능하다.

## 파이프라인과 명령어 수준 병렬성의 성능 해석

- 파이프라인은 명령어 처리를 여러 단계로 나눠 겹쳐 수행하는 방식으로, 세탁기와 건조기를 동시에 돌리는 생산 라인과 비슷하다.
- 핵심 장점은 한 명령어의 지연 시간을 크게 줄이는 데 있기보다, 전체 처리량을 높여 단위 시간당 더 많은 명령어를 끝내는 데 있다.
- 구조적 위험은 같은 자원을 동시에 쓰려 할 때 생기며, 예를 들어 메모리가 하나뿐이면 인출과 데이터 접근이 충돌할 수 있다.
- 데이터 위험은 앞선 명령의 결과가 아직 나오지 않았는데 다음 명령이 그 값을 필요로 할 때 생기며, 전달 기법으로 일부를 줄일 수 있다.
- 제어 위험은 분기문 때문에 다음 명령어 주소를 확정하기 어려운 문제로, 반복이 많은 코드에서 분기 예측 실패는 체감 성능을 크게 떨어뜨린다.
- 슈퍼스칼라 구조는 여러 명령어를 동시에 발행해 더 높은 병렬성을 노리지만, 명령어 사이의 독립성이 부족하면 이론적인 성능이 잘 나오지 않는다.
- 개발자 입장에서는 단순히 연산 수를 줄이는 것보다 분기 패턴을 단순하게 만들고 연속 접근을 늘리는 편이 현대 처리기에서 더 효과적일 때가 많다.

## 명령어 집합 구조 비교: 복잡 명령어와 축소 명령어의 관점

- 명령어 집합 구조는 소프트웨어와 하드웨어 사이의 약속으로, 어떤 명령어가 있고 레지스터가 몇 개인지 같은 규칙을 정한다.
- 복잡 명령어 방식은 한 명령에 많은 일을 담아 코드 길이를 줄이려는 경향이 있고, 축소 명령어 방식은 단순한 명령을 빠르게 반복하려는 성격이 강하다.
- 예를 들어 메모리에서 읽고 바로 계산하는 복잡 명령어는 편리하지만 구현이 복잡해질 수 있고, 축소 명령어는 여러 단계로 나누는 대신 파이프라인에 유리하다.
- 오늘날에는 둘이 완전히 갈라져 있다기보다, 외부 명령어 집합과 내부 실행 구조가 분리되어 장점을 섞어 쓰는 경우가 많다.
- 개방형 명령어 집합 구조인 RISC-V는 교육용 예제로 이해하기 쉽고, 연구나 맞춤형 칩 설계에서도 확장성이 높아 비교 학습에 유리하다.
- 같은 고급 언어 코드라도 목표 명령어 집합 구조에 따라 생성되는 기계어와 최적화 전략이 달라져, 컴파일러 수업과도 자연스럽게 연결된다.
- 실무에서는 특정 처리기가 빠른 이유를 볼 때 운영체제나 언어만 보지 말고, 명령어 집합 구조와 구현 세부가 어떤 제약을 주는지도 함께 살펴야 한다.

## 버스, 입출력, 저장장치 연결이 시스템 병목을 만드는 방식

- 버스는 중앙처리장치, 메모리, 주변장치가 데이터를 주고받는 공용 통로로, 도로가 좁으면 좋은 차가 많아도 정체가 생기는 것과 같다.
- 주소 버스, 데이터 버스, 제어 버스는 역할이 나뉘며, 어느 하나만 부족해도 전체 전송 효율이 제한된다.
- 입출력장치는 속도가 중앙처리장치보다 훨씬 느린 경우가 많아, 인터럽트와 직접 메모리 접근 같은 기법으로 대기 시간을 줄인다.
- 인터럽트는 장치가 일이 끝났다고 처리기에 알리는 방식이라, 키보드 입력처럼 드문 사건에 효율적이고 계속 감시하는 방식보다 낭비가 적다.
- 직접 메모리 접근은 대용량 전송에서 중앙처리장치 개입을 줄여 저장장치 읽기나 영상 전송에서 병목을 완화한다.
- 보조기억장치는 용량은 크지만 지연이 크므로, 운영체제와 파일시스템이 접근 순서를 잘 조정해야 실제 성능이 안정적으로 나온다.
- 서버 설계에서는 계산 성능만 높여도 저장장치나 연결 통로가 따라오지 못하면 체감 처리량이 늘지 않아, 균형 잡힌 구조가 더 중요하다.

## 최신 흐름

- 2026년 기준 인공지능 연산 수요 때문에 계산 성능보다 메모리 대역폭이 병목이 되는 경우가 많아, 고대역폭 메모리의 중요성이 급격히 커졌다.
- HBM4가 본격적인 양산 단계에 들어가면서 단순한 용량 경쟁보다 대역폭, 적층 수, 전력 효율을 함께 보는 구조 설계가 중요해졌다.
- 칩렛은 하나의 큰 칩을 여러 조각으로 나누어 묶는 방식으로, 수율과 설계 유연성을 높여 대형 인공지능 가속기와 서버용 칩에서 빠르게 확산되고 있다.
- 칩렛 생태계에서는 UCIe 같은 연결 표준과 Arm CSA 같은 시스템 수준 규약이 중요해져, 이제는 칩 내부보다 칩 사이의 호환성이 경쟁력으로 떠오르고 있다.
- 메모리 확장 연결 기술인 CXL은 중앙처리장치와 가속기, 메모리를 더 유연하게 묶어 대규모 작업에서 메모리 풀링과 자원 공유를 가능하게 하는 방향으로 발전하고 있다.
- 개방형 명령어 집합 구조인 RISC-V는 교육용을 넘어 자동차, 임베디드, 인공지능 가속 영역까지 확장되며 산업 생태계의 성숙도가 빠르게 높아지고 있다.
- 서버 구조는 중앙처리장치 단독 중심에서 그래픽 처리장치, 신경망 처리장치, 전용 가속기를 함께 묶는 이기종 구조로 이동해, 컴퓨터구조의 학습 범위도 더 넓어지고 있다.
- 성능 지표도 클록 속도 중심에서 전력당 성능, 메모리 이동 비용, 패키지 수준 지연처럼 시스템 전체 효율을 보는 방향으로 바뀌고 있다.
- 엣지 장치에서도 생성형 인공지능 탑재가 늘면서 작은 전력 예산 안에서 메모리 계층과 가속기 구성을 최적화하는 설계가 중요한 주제가 되었다.
- 결국 최근 흐름은 더 빠른 한 개의 코어를 만드는 경쟁보다, 메모리, 패키징, 연결, 명령어 표준을 함께 조합해 목적에 맞는 시스템을 만드는 경쟁으로 요약할 수 있다.
