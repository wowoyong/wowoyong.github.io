## 핵심 요약

보안은 기밀성, 무결성, 가용성을 지키기 위해 위협을 분석하고 통제 수단을 설계하는 분야입니다. 학부 수준에서는 암호화, 인증, 접근통제, 네트워크 방어, 안전한 소프트웨어 개발, 사고 대응을 함께 이해해야 전체 그림이 잡힙니다. 실제 적용에서는 기술 하나만 강하게 두기보다 사용자, 서버, 네트워크, 개발 과정, 운영 절차를 겹쳐 방어하는 방식이 중요합니다. 예를 들어 비밀번호만 강화하는 것보다 다중 인증, 최소 권한, 기록 분석, 백업을 함께 두는 편이 훨씬 현실적입니다. 2026-04-01 기준 최근 흐름은 제로 트러스트, 패스키 확산, 생성형 인공지능 보안, 소프트웨어 공급망 점검, 양자내성암호 전환 준비처럼 설계 단계부터 보안을 넣는 방향으로 모이고 있습니다.

## 기밀성·무결성·가용성으로 보는 위협 모델링

- 기밀성은 허가되지 않은 사람이 정보를 보지 못하게 하는 성질이며, 학생 성적 파일에 접근 권한을 제한하는 경우가 대표적입니다.
- 무결성은 데이터가 허락 없이 바뀌지 않는 성질이며, 계좌 금액이 전송 중 변조되면 기밀성보다 무결성 문제가 더 큽니다.
- 가용성은 필요한 순간에 서비스를 쓸 수 있는 성질이며, 대량 요청 공격으로 수강신청 서버가 멈추면 가용성 침해입니다.
- 위협 모델링은 무엇을 지킬지, 누가 공격자인지, 어디가 약한지를 먼저 그리는 과정이며, 기능 구현 전에 하면 비용이 크게 줄어듭니다.
- 자산은 비밀번호만이 아니라 개인정보, 결제 기록, 소스코드, 운영 권한까지 포함되며, 실무에서는 자산 목록이 보안 우선순위를 정합니다.
- 공격 표면은 외부에 드러난 입력 창, 공개 서버, 관리자 화면 같은 접점 전체를 뜻하며, 기능이 많을수록 공격 가능 지점도 늘어납니다.
- 위험도 평가는 피해 크기와 발생 가능성을 함께 보며, 실습용 게시판 취약점과 금융 서비스 취약점은 같은 결함이어도 대응 우선순위가 다릅니다.

## 암호화·해시·전자서명의 역할 구분

- 암호화는 내용을 숨기는 기술이고, 해시는 원문을 짧은 값으로 요약하는 기술이라서 비밀번호 저장에는 보통 해시가 더 적합합니다.
- 대칭키 암호는 같은 열쇠로 잠그고 푸는 방식이라 빠르지만, 열쇠 전달이 어렵다는 점에서 공개키 방식과 대비됩니다.
- 공개키 암호는 공개키와 개인키를 나눠 쓰므로 키 배포가 편리하며, 실제 서비스에서는 키 교환과 인증에 자주 쓰입니다.
- 해시는 파일이 바뀌었는지 확인할 때 유용하며, 내려받은 설치 파일의 해시값을 비교하면 변조 여부를 빠르게 볼 수 있습니다.
- 전자서명은 보낸 사람 확인과 내용 위조 방지에 쓰이며, 단순 암호화와 달리 '누가 보냈는가'를 확인하는 데 강점이 있습니다.
- 비밀번호는 원문 저장보다 느린 해시와 소금을 함께 써서 저장해야 하며, 같은 비밀번호라도 저장값이 달라져 대입 공격을 줄입니다.
- 실무에서는 전송 구간 보호, 저장 데이터 보호, 비밀번호 보관, 코드 서명을 각각 다른 도구로 해결하므로 암호 기술의 용도 구분이 중요합니다.

## 인증과 인가를 나누어 이해하는 접근통제

- 인증은 '누구인가'를 확인하는 과정이고 인가는 '무엇을 할 수 있는가'를 정하는 과정이라, 로그인 성공이 곧 관리자 권한을 뜻하지는 않습니다.
- 단일 비밀번호 인증은 편하지만 피싱에 약하므로, 실무에서는 추가 인증 수단을 붙여 계정 탈취 위험을 낮춥니다.
- 다중 인증은 아는 것, 가진 것, 생체 특징처럼 서로 다른 요소를 조합하는 방식이며, 문자 인증보다 기기 기반 인증이 더 강한 경우가 많습니다.
- 최소 권한 원칙은 필요한 권한만 주는 방식으로, 학생 조교 계정에 성적 열람은 주되 전체 삭제 권한은 주지 않는 식입니다.
- 역할 기반 접근통제는 직무별 권한 묶음을 만들어 관리하므로, 사용자마다 직접 권한을 붙이는 방식보다 운영이 단순합니다.
- 세션 관리가 약하면 로그인 자체가 안전해도 탈취된 세션으로 우회될 수 있어, 만료 시간과 재인증 정책이 중요합니다.
- 실무에서는 인증 서버, 권한 정책, 감사 기록을 함께 설계해야 하며, 권한 오남용은 외부 침입만큼 큰 내부 보안 사고로 이어질 수 있습니다.

## 네트워크와 시스템을 지키는 다층 방어

- 방화벽은 통신 규칙을 기준으로 허용과 차단을 결정하며, 건물 출입문을 지키는 경비와 비슷하지만 내부 이상 행위까지 모두 막지는 못합니다.
- 침입 탐지와 침입 방지는 수상한 패턴을 찾는 장치로, 단순 차단보다 기록과 경보를 함께 제공해 사고 분석에 도움이 됩니다.
- 망 분리는 중요한 자산을 일반 업무 구역과 떨어뜨려 피해 확산을 줄이는 방식이며, 연구망과 행정망 분리가 전형적 예시입니다.
- 서비스 거부 공격 대응은 대역폭만 늘리는 문제가 아니라 분산 차단, 캐시, 우회 경로 설계까지 포함하는 운영 문제입니다.
- 운영체제 보안은 계정 분리, 패치, 불필요한 서비스 제거가 핵심이며, 화려한 보안 도구보다 기본 설정이 더 큰 효과를 내는 경우가 많습니다.
- 기록 수집과 분석은 공격 흔적을 찾는 기본 재료이므로, 실패한 로그인과 권한 상승 시도를 남기지 않으면 사고 원인 파악이 어려워집니다.
- 다층 방어는 한 장벽이 뚫려도 다음 장벽이 버티게 만드는 전략이며, 네트워크 차단과 단말 보호와 권한 통제를 겹쳐 두는 것이 현실적입니다.

## 입력 검증부터 배포까지 이어지는 안전한 개발

- 안전한 개발은 완성 뒤에 검사하는 일이 아니라 요구사항, 설계, 구현, 시험, 배포 전 과정을 묶는 활동입니다.
- 입력 검증은 외부 입력을 믿지 않는 원칙으로, 검색창 하나도 악성 문자열이 들어올 수 있다고 보고 처리해야 합니다.
- 출력 시 문자 이스케이프는 화면 표시 단계의 방어로, 저장 단계 검증과 역할이 다르므로 둘 중 하나만으로는 부족합니다.
- 의존 라이브러리 관리는 직접 작성한 코드만 보는 것보다 더 중요할 때가 많으며, 작은 외부 패키지 하나가 전체 서비스를 흔들 수 있습니다.
- 비밀정보 분리는 소스코드에 열쇠를 박아 넣지 않는 습관을 뜻하며, 환경 변수나 전용 비밀 저장소를 쓰는 편이 안전합니다.
- 정적 분석과 동적 분석은 서로 보완적이라, 전자는 코드 구조를 보고 결함을 찾고 후자는 실행 중 이상 동작을 확인합니다.
- 실무에서는 배포 자동화에 보안 검사 단계를 넣어야 효과가 크며, 사람 기억에만 의존하면 바쁜 일정 속에서 보안 점검이 빠지기 쉽습니다.

## 사고 대응·포렌식·복구 체계의 실제 의미

- 사고 대응은 탐지, 분석, 격리, 제거, 복구, 재발 방지의 흐름으로 보며, 단순히 서버를 끄는 것이 끝이 아닙니다.
- 초기 대응에서 가장 중요한 것은 증거 보존이므로, 로그를 지우거나 성급히 재설치하면 원인 분석이 더 어려워질 수 있습니다.
- 격리는 피해 확산을 막기 위한 조치로, 감염된 단말을 네트워크에서 떼어내는 것이 서비스 전체 중단보다 나은 선택일 때가 많습니다.
- 포렌식은 시스템 흔적을 바탕으로 사건을 재구성하는 활동이며, 공격자가 언제 어떤 경로로 들어왔는지 시간순으로 정리합니다.
- 백업은 복구의 핵심이지만 백업 파일도 함께 감염될 수 있으므로, 오프라인 백업이나 분리된 저장소가 더 안전합니다.
- 재해 복구 계획은 목표 복구 시간과 목표 데이터 손실 범위를 미리 정하는 일이며, 병원과 게임 서비스는 허용 가능한 중단 시간이 다릅니다.
- 사후 분석은 비난보다 개선에 초점을 두어야 하며, 같은 실수를 막기 위한 절차 수정과 자동화 추가가 실제 성과로 이어집니다.

## 최신 흐름

- 제로 트러스트는 경계 안쪽을 자동 신뢰하지 않는 방식으로 자리잡고 있으며, 원격 근무와 클라우드 환경이 늘수록 중요성이 커지고 있습니다.
- 패스키 같은 피싱 저항형 인증이 빠르게 확산되고 있어, 비밀번호 중심 설계에서 기기 기반 공개키 인증으로 이동하는 흐름을 이해할 필요가 있습니다.
- 생성형 인공지능 서비스 보안이 별도 주제로 분리되고 있으며, 프롬프트 주입과 데이터 유출 방지가 웹 취약점만큼 중요한 항목이 되었습니다.
- 소프트웨어 공급망 보안은 외부 패키지와 빌드 과정까지 점검하는 방향으로 강화되고 있으며, 소프트웨어 자재명세서 관리가 실무 기본으로 자리잡고 있습니다.
- 설계 단계부터 안전해야 한다는 '안전한 설계' 관점이 강해져, 사용자가 복잡한 설정으로 위험을 막게 하기보다 제품 자체가 기본적으로 안전해야 한다는 요구가 커졌습니다.
- 양자내성암호 전환 준비가 본격화되어, 당장 모든 체계를 바꾸기보다 먼저 어디에 기존 공개키 암호가 쓰이는지 자산 조사부터 시작하는 조직이 늘고 있습니다.
- 클라우드와 컨테이너 보안은 서버 한 대 보호보다 권한 구성과 비밀정보 관리가 더 중요해지고 있어, 설정 오류 탐지가 핵심 역량으로 평가됩니다.
- 공격과 방어 모두에서 인공지능 활용이 늘어, 피싱 문구 자동 생성과 이상 징후 탐지 자동화가 동시에 발전하는 양면성이 나타나고 있습니다.
- 규제와 표준 준수는 선택이 아니라 기본 운영 조건이 되어, 기술팀도 로그 보존, 접근 기록, 취약점 공개 절차 같은 문서화를 함께 이해해야 합니다.
- 보안 인력 양성 흐름도 변해 단순 이론보다 모의훈련, 침해사고 대응 연습, 실습형 문제 해결 능력이 학부 교육과 채용에서 더 중시되고 있습니다.
