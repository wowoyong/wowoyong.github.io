## 핵심 요약

운영체제는 하드웨어 자원을 효율적으로 배분하고 프로그램 실행을 통제하는 핵심 소프트웨어다. 학부 수준에서는 프로세스와 스레드, 중앙처리장치 스케줄링, 메모리 관리, 가상 메모리, 파일 시스템, 동기화, 교착 상태를 함께 이해해야 전체 구조를 파악할 수 있다. 실제 환경에서는 응답 속도, 처리량, 안정성, 보안성처럼 서로 충돌할 수 있는 목표 사이에서 어떤 정책을 선택하느냐가 중요하다. 같은 기능이라도 일괄 처리 서버, 모바일 기기, 실시간 제어 장치에서는 운영체제가 우선하는 기준이 달라진다. 최근에는 메모리 안전성 강화, 커널 내부 관측 기술, 인공지능 인프라 최적화, 기밀 컴퓨팅, 경량 가상화 같은 흐름이 운영체제의 설계와 활용 방식을 바꾸고 있다.

## 프로세스와 스레드의 실행 단위 차이 이해하기

- 프로세스는 실행 중인 프로그램의 독립된 자원 묶음이고, 스레드는 그 안에서 실제 작업을 수행하는 실행 흐름이다. 예를 들어 웹 브라우저는 하나의 프로세스 안에 여러 스레드를 두어 화면 표시와 입력 처리를 나눌 수 있다.
- 프로세스는 주소 공간이 서로 분리되어 있어 안정성은 높지만 전환 비용이 크고, 스레드는 메모리를 공유하므로 빠르지만 충돌 위험이 커진다. 그래서 서버 환경에서는 성능이 중요하면 스레드를, 격리가 중요하면 프로세스를 더 선호한다.
- 프로세스 상태는 보통 생성, 준비, 실행, 대기, 종료로 설명한다. 디스크 입력을 기다리는 동안에는 실행이 멈추고, 그 사이 중앙처리장치는 다른 작업을 수행한다.
- 문맥 전환은 운영체제가 현재 작업의 레지스터 값과 상태를 저장한 뒤 다른 작업으로 바꾸는 과정이다. 이 과정이 너무 자주 일어나면 실제 계산보다 전환 비용이 커져 성능이 떨어질 수 있다.
- 부모 프로세스와 자식 프로세스 구조는 새로운 작업이 만들어지는 방식을 설명할 때 자주 쓰인다. 명령 해석기가 자식 프로세스를 만들어 프로그램을 실행하는 경우가 대표적이다.
- 스레드가 공유하는 자원에는 코드, 힙, 열린 파일 등이 포함되므로 협업이 쉽다. 하지만 같은 변수를 동시에 바꾸면 결과가 달라질 수 있어 동기화가 필요하다.
- 학습할 때는 프로세스를 독립된 집, 스레드를 그 집 안에서 일하는 여러 작업자로 비유하면 이해하기 쉽다. 집이 나뉘면 안전하지만 오가는 비용이 크고, 작업자가 많으면 빠르지만 조율이 중요하다.

## 중앙처리장치 스케줄링에서 응답 시간과 공정성 비교하기

- 스케줄링은 준비 상태에 있는 작업 가운데 무엇을 먼저 실행할지 정하는 정책이다. 운영체제는 짧은 응답 시간, 높은 처리량, 공정성, 예측 가능성 사이에서 균형을 잡아야 한다.
- 선입선출 방식은 구현이 단순하지만 긴 작업 하나가 뒤에 있는 짧은 작업들을 오래 막을 수 있다. 은행 창구에서 처리 시간이 긴 고객 한 명 때문에 전체 줄이 지연되는 상황과 비슷하다.
- 가장 짧은 작업 우선 방식은 평균 대기 시간을 줄이는 데 유리하지만, 실행 시간을 미리 알기 어렵고 긴 작업이 계속 밀릴 수 있다. 이론적으로는 좋지만 실제 환경에서는 추정 오차가 문제다.
- 라운드 로빈 방식은 일정한 시간 조각마다 작업을 번갈아 실행하므로 대화형 시스템에 적합하다. 모바일 앱이나 터미널처럼 반응성이 중요한 환경에서 자주 떠올리는 모델이다.
- 우선순위 스케줄링은 중요한 작업을 먼저 처리하지만, 낮은 우선순위 작업이 계속 실행되지 못하는 문제가 생길 수 있다. 이를 막기 위해 오래 기다린 작업의 우선순위를 높이는 기법을 함께 쓴다.
- 실시간 스케줄링은 평균 성능보다 마감 시간을 지키는 일이 더 중요하다. 예를 들어 자동차 제어 장치에서는 늦어도 되는 계산보다 제시간에 끝나는 계산이 훨씬 중요하다.
- 실무에서는 한 가지 방식만 쓰기보다 여러 정책을 섞어 사용한다. 사용자가 체감하는 성능은 단순한 총처리량보다 응답 시간의 분포에 더 크게 좌우된다는 점이 핵심이다.

## 동기화 문제를 임계 구역 관점에서 이해하기

- 동기화는 여러 실행 흐름이 공유 자원에 안전하게 접근하도록 순서를 맞추는 일이다. 은행 계좌 잔액을 동시에 수정하는 상황을 떠올리면 왜 필요한지 쉽게 이해할 수 있다.
- 임계 구역은 한 번에 하나의 실행 흐름만 들어가야 하는 코드 구간이다. 예를 들어 전역 카운터를 증가시키는 몇 줄의 코드도 실제로는 읽기와 쓰기가 나뉘어 있어 충돌 지점이 된다.
- 경쟁 상태는 실행 순서에 따라 결과가 달라지는 오류다. 같은 프로그램이 어떤 때는 맞고 어떤 때는 틀리다면 알고리즘보다 동기화 설계를 먼저 의심해야 한다.
- 뮤텍스는 상호 배제를 위한 대표적인 도구로, 잠금과 해제를 통해 한 시점에 하나의 실행 흐름만 자원에 접근하게 한다. 구현은 단순하지만 잠금을 오래 유지하면 병목이 생긴다.
- 세마포어는 사용 가능한 자원 개수를 세어 여러 실행 흐름의 접근을 조절한다. 주차장의 빈자리 수를 세는 방식으로 비유하면 이해하기 쉽다.
- 조건 변수는 특정 조건이 만족될 때까지 기다렸다가 다시 깨우는 데 적합하다. 생산자와 소비자 문제에서 버퍼가 비었는지 찼는지를 기준으로 협력할 때 자주 등장한다.
- 현실의 시스템에서는 잠금 자체보다 잠금 범위를 어떻게 설계하느냐가 더 중요하다. 너무 넓게 잠그면 느려지고, 너무 좁게 잠그면 오류가 나므로 안전성과 병렬성 사이의 타협이 핵심이다.

## 교착 상태의 발생 조건과 예방 전략 연결하기

- 교착 상태는 여러 작업이 서로가 가진 자원을 기다리며 영원히 진행하지 못하는 상태다. 두 사람이 서로 상대의 열쇠를 받아야 문을 열 수 있는 상황으로 비유할 수 있다.
- 대표적인 발생 조건은 상호 배제, 점유하며 대기, 비선점, 환형 대기 네 가지다. 시험에서는 외워야 할 항목처럼 보이지만, 실제로는 어떤 조건을 깨서 문제를 줄일지 판단하는 틀이 된다.
- 예방은 처음부터 교착 상태가 생기지 않도록 규칙을 두는 방법이다. 예를 들어 모든 잠금을 항상 같은 순서로 획득하게 하면 환형 대기를 줄일 수 있다.
- 회피는 현재 자원 할당이 앞으로도 안전한지 계산해 위험한 요청을 미루는 방식이다. 은행가 알고리즘이 대표적이지만 실제 대규모 시스템에서는 비용과 정보 부족 때문에 제한적으로만 쓰인다.
- 탐지와 복구 방식은 일단 허용한 뒤 문제가 생기면 찾아내고 일부 작업을 종료하거나 되돌리는 접근이다. 데이터베이스나 분산 시스템에서는 이 방식이 더 현실적인 경우도 있다.
- 교착 상태와 기아 상태는 다르다. 교착 상태는 서로 막혀 모두 진행하지 못하는 경���이고, 기아 상태는 특정 작업만 계속 선택받지 못하는 경우이므로 대책도 달라진다.
- 실무에서는 잠금 개수보다 자원 의존 관계를 단순하게 만드는 설계가 더 중요하다. 자원 획득 순서를 문서화하는 것만으로도 디버깅 비용을 크게 줄일 수 있다.

## 메모리 관리에서 연속 할당과 분할 방식 비교하기

- 메모리 관리는 제한된 주기억장치를 여러 프로그램이 나누어 쓰도록 배치하는 기능이다. 운영체제는 공간 낭비를 줄이면서도 빠르게 할당하고 회수해야 한다.
- 연속 할당은 하나의 프로그램에 이어진 공간을 주는 방식이라 주소 계산이 단순하다. 하지만 빈 공간이 잘게 쪼개지는 외부 단편화가 생기기 쉽다.
- 고정 분할은 구현이 쉽지만 큰 작업이 들어오지 못하거나 작은 작업이 큰 칸을 차지해 낭비가 생긴다. 기숙사 방 크기가 모두 같을 때 생기는 비효율과 비슷하다.
- 가변 분할은 작업 크기에 맞춰 공간을 주므로 유연하지만, 시간이 지나면 빈 공간이 여기저기 흩어져 관리가 어려워진다. 그래서 압축 같은 추가 작업이 필요할 수 있다.
- 최초 적합, 최적 적합, 최악 적합 같은 배치 전략은 같은 빈 공간 목록을 보고도 다른 결과를 만든다. 어느 방식이 항상 가장 좋은 것은 아니며 작업 패턴에 따라 장단점이 갈린다.
- 내부 단편화는 할당받은 공간 안에서 남는 낭비이고, 외부 단편화는 빈 공간이 흩어져 큰 요청을 받지 못하는 낭비다. 이 둘을 구분해야 왜 특정 기법이 필요한지 설명할 수 있다.
- 학부 수준에서는 할당 방식이 단순한 자료 배치 문제가 아니라 성능, 예측 가능성, 구현 복잡도를 함께 바꾸는 운영 정책이라는 점을 이해하는 것이 중요하다.

## 가상 메모리와 페이지 교체를 성능 관점에서 살펴보기

- 가상 메모리는 프로그램마다 큰 연속 주소 공간이 있는 것처럼 보이게 하는 기술이다. 실제 물리 메모리보다 큰 프로그램도 실행할 수 있다는 점이 핵심 장점이다.
- 페이지는 고정 크기의 블록으로 나눈 단위이고, 페이지 테이블은 가상 주소를 실제 주소로 바꾸는 지도 역할을 한다. 덕분에 프로그램은 메모리 배치를 직접 알 필요가 없다.
- 페이지 부재는 필요한 페이지가 주기억장치에 없어 보조기억장치에서 가져와야 하는 상황이다. 이때 디스크 접근이 끼어들면 속도 차이가 매우 커져 체감 성능이 급격히 떨어진다.
- 최선 교체 방식은 미래를 알아야 하므로 이론적인 비교 기준으로만 쓴다. 실제 구현에서는 최근 사용 이력에 기대는 최근 최소 사용 방식이나 그 근사 기법을 더 많이 사용한다.
- 최근 최소 사용 방식은 방금 사용한 페이지를 다시 사용할 가능성이 높다는 지역성 가정에 기반한다. 동영상 편집보다 반복문 계산처럼 비슷한 데이터를 자주 쓰는 작업에서 특히 잘 맞는다.
- 페이지 교체가 지나치게 많아지면 스래싱이 발생해 중앙처리장치보다 입출력 대기 시간이 더 길어진다. 프로그램 수가 많다고 해서 항상 처리량이 늘지는 않는다는 대표적인 반례다.
- 실무적으로는 페이지 크기, 교체 정책, 작업 수 조절이 함께 맞물린다. 운영체제 성능 저하는 계산 능력 부족보다 메모리 압박 때문에 생기는 경우가 많다는 점을 기억할 필요가 있다.

## 파일 시스템 구조를 저장 장치 추상화로 이해하기

- 파일 시스템은 저장 장치를 파일과 디렉터리라는 익숙한 형태로 추상화한다. 사용자는 블록 번호를 몰라도 문서와 사진을 이름으로 다룰 수 있다.
- 파일 메타데이터에는 크기, 권한, 소유자, 수정 시각 같은 정보가 들어간다. 내용 자체가 아니라 관리 정보라는 점에서 파일 데이터와 구분해야 한다.
- 연속 할당은 순차 접근이 빠르지만 파일 크기가 늘어날 때 불편하고, 연결 할당은 확장에 유리하지만 임의 접근이 느리다. 색인 기반 방식은 그 중간 지점을 노리는 대표적인 설계다.
- 디렉터리는 단순한 폴더가 아니라 이름과 실제 파일 정보를 연결하는 구조다. 같은 이름 규칙과 경로 체계는 사용자 경험뿐 아니라 관리 효율에도 영향을 준다.
- 캐시는 자주 쓰는 블록을 메모리에 유지해 디스크 접근을 줄인다. 같은 파일을 반복해서 열 때 빠르게 느껴지는 이유가 바로 이 계층 구조 덕분이다.
- 저널링은 변경 내용을 먼저 기록해 갑작스러운 전원 차단 뒤에도 손상을 줄이는 기법이다. 완전한 무손실을 보장하는 것은 아니지만 복구 시간을 크게 줄여 준다.
- 실무에서는 파일 시스템 선택이 성능뿐 아니라 복구 가능성, 대용량 처리 능력, 작은 파일 처리 효율까지 바꾼다. 데이터 특성에 맞는 저장 정책이 중요하다는 점에서 메모리 관리와 닮아 있다.

## 최신 흐름

- 메모리 안전성을 강화하기 위해 커널과 장치 구동기의 일부를 Rust 계열 언어로 작성하려는 흐름이 커지고 있다. 버퍼 오버플로와 해제 후 사용 같은 고전적인 오류를 줄이려는 목적이 크다.
- Linux와 Android 진영에서는 실제 제품 수준에서 Rust 지원이 점차 확대되고 있다. 학습 관점에서는 언어 선택이 단순한 취향 문제가 아니라 운영체제의 안정성과 유지보수 비용에 연결된다는 점이 중요하다.
- eBPF 기반 관측과 보안 정책 적용이 커널 내부까지 확장되고 있다. 예전에는 별도 도구로 여겼던 성능 측정과 실행 중 보안이 이제는 같은 데이터 경로에서 함께 다뤄지는 추세다.
- 클라우드 환경에서는 경량 가상머신과 컨테이너의 경계가 더욱 가까워지고 있다. 빠른 시작 속도는 컨테이너의 장점이고 강한 격리는 가상머신의 장점인데, 최근에는 두 장점을 함께 얻으려는 설계가 많다.
- 기밀 컴퓨팅이 운영체제 설계의 중요한 요구 조건으로 떠오르고 있다. 사용 중인 데이터 보호와 원격 검증의 중요성이 커지면서 가상머신 내부 운영체제의 신뢰 경계도 다시 검토되고 있다.
- 마이크로커널과 형식 검증 계열 운영체제는 안전이 중요한 분야에서 다시 주목받고 있다. 범용 데스크톱에서는 덜 쓰이더라도 항공, 국방, 산업 제어처럼 실패 비용이 큰 곳에서는 의미가 크다.
- 인공지능 학습과 추론 작업이 늘면서 운영체제는 가속기, 고속 네트워크, 대용량 메모리를 더 정교하게 다루는 방향으로 발전하고 있다. 이제는 단순한 중앙처리장치 스케줄링만으로 전체 성능을 설명하기 어려워졌다.
- 엣지 장치와 사물인터넷 환경에서는 실시간 운영체제와 경량 운영체제의 산업 적용이 확대되고 있다. 긴 수명, 낮은 전력 소모, 예측 가능한 응답이 서버용 운영체제보다 더 중요한 경우가 많다.
- 중단 없는 갱신과 빠른 복구를 위한 커널 갱신 기술도 계속 개선되고 있다. 서비스 중단 비용이 큰 대규모 인프라에서는 재부팅 한 번조차 운영 정책의 중요한 변수다.
- 운영체제 학습 역시 단일 컴퓨터 중심에서 벗어나고 있다. 이제는 커널, 가상화, 컨테이너, 분산 자원 관리, 보안 검증을 함께 보는 시각이 실무 적응력과 더 직접적으로 연결된다.
