## 핵심 요약

데이터베이스는 데이터를 구조화해 저장하고, 빠르게 조회하며, 여러 사용자가 동시에 안전하게 다룰 수 있게 만드는 기술이다. 학부 과정에서는 관계형 모델, 정규화, 인덱스, 질의 처리, 트랜잭션, 복구 같은 원리를 함께 이해해야 실제 시스템을 읽을 수 있다. 예를 들어 쇼핑몰에서는 상품 조회 속도를 위해 인덱스를 쓰고, 주문 처리에서는 트랜잭션으로 결제와 재고 변경을 하나의 작업처럼 묶는다. 실무에서는 데이터 모델링을 잘못하면 기능은 돌아가도 성능 저하와 중복 갱신 문제가 빠르게 커진다. 최근에는 벡터 검색 통합, 서버리스 분산 데이터베이스, 레이크하우스 연계처럼 전통적 관계형 기능과 대규모 인공지능 활용이 결합되는 흐름이 강해지고 있다.

## 관계형 모델과 키 설계의 의미

- 관계형 모델은 데이터를 표 형태로 표현하는 방식이며, 학생 표와 수강 표를 분리하면 한 학생이 여러 과목을 듣는 상황을 자연스럽게 다룰 수 있다.
- 기본키는 각 행을 유일하게 구분하는 값으로, 학번이나 주문번호처럼 중복되면 안 되는 식별자에 해당한다.
- 외래키는 표와 표를 연결하는 장치로, 수강 표의 학번이 학생 표의 학번을 가리키게 하면 없는 학생의 수강 기록이 들어가는 오류를 막을 수 있다.
- 후보키는 기본키가 될 수 있는 모든 속성 집합이고, 그중 실제 선택된 하나가 기본키라는 점에서 개념을 구분해야 설계가 명확해진다.
- 자연키는 주민번호처럼 현실 의미가 있는 값이고, 대리키는 시스템이 만든 번호인데, 변경 가능성이 큰 데이터라면 대리키가 유지보수에 유리하다.
- 개체와 관계를 먼저 파악하고 표로 옮기는 습관이 중요하며, 도서관 시스템에서는 책, 회원, 대출을 따로 나누는 식으로 시작하면 설계가 흔들리지 않는다.
- 키 설계를 대충 하면 조회는 되더라도 중복 데이터와 연결 오류가 늘어나므로, 초기 설계 단계에서 업무 규칙을 키에 반영하는 해석이 실무적으로 중요하다.

## 정규화가 중복과 이상 현상을 줄이는 방식

- 정규화는 중복을 줄이고 갱신 오류를 막기 위해 표를 더 작은 구조로 나누는 과정이며, 한 표에 학생 정보와 학과 전화번호를 함께 넣으면 같은 값이 반복되기 쉽다.
- 삽입 이상은 필요한 데이터를 넣고 싶어도 다른 값이 없어서 못 넣는 경우인데, 아직 학생이 없는 새 학과를 등록하지 못하는 예가 대표적이다.
- 삭제 이상은 한 행을 지우면서 다른 중요한 정보도 함께 사라지는 문제로, 마지막 수강생 기록을 지웠더니 과목 정보까지 없어지는 상황이 이에 해당한다.
- 갱신 이상은 같은 정보를 여러 곳에서 고쳐야 하는 문제이며, 교수 연구실 번호가 여러 행에 중복되면 하나만 수정되고 나머지는 남을 수 있다.
- 제일정규형은 칸마다 원자값만 두는 것이고, 전화번호 여러 개를 한 칸에 적는 대신 별도 표로 분리하는 예로 이해하면 쉽다.
- 제이정규형과 제삼정규형은 부분 함수 종속과 이행 함수 종속을 줄이는 단계인데, 핵심은 기본키와 직접 관련 없는 속성을 떼어내는 데 있다.
- 정규화가 항상 정답은 아니며, 조회 성능이 매우 중요한 화면에서는 일부 중복을 허용하는 반정규화를 쓰기도 하므로 성능과 일관성의 비교가 필요하다.

## 인덱스가 조회 속도를 바꾸는 이유

- 인덱스는 책의 맨 뒤 색인처럼 원하는 데이터를 빨리 찾게 해 주는 구조이며, 이름 없는 전체 탐색보다 훨씬 적은 비교로 결과에 도달할 수 있다.
- 학생 수가 몇십 명이면 전체 탐색도 괜찮지만, 수백만 행이 되면 학번 인덱스 유무가 응답시간을 체감적으로 갈라놓는다.
- 기본 인덱스는 정렬된 저장 구조와 함께 쓰이고, 보조 인덱스는 별도 구조로 관리되는데, 읽기 성능은 좋아져도 쓰기 비용이 늘어난다는 비교가 중요하다.
- 범위 조회에는 정렬 기반 인덱스가 유리하고, 정확히 같은 값 찾기에는 해시 기반 방식이 빠를 수 있어 질의 형태에 따라 선택이 달라진다.
- 복합 인덱스는 여러 열을 함께 다루며, 학과와 학년으로 자주 찾는다면 그 순서를 고려해 설계해야 실제 효과가 난다.
- 인덱스를 많이 만들수록 무조건 좋은 것이 아니며, 주문 입력이 많은 시스템에서는 삽입 때마다 인덱스도 갱신되어 쓰기 성능이 떨어질 수 있다.
- 실무에서는 느린 질의를 보기 전에 무작정 인덱스를 추가하기보다 어떤 조건절과 정렬이 자주 쓰이는지 먼저 관찰하는 해석이 더 중요하다.

## 질의 처리와 최적화의 핵심 판단

- 질의 처리는 사용자가 쓴 구조적 질의 언어를 실행 계획으로 바꾸는 과정이며, 같은 결과라도 어떤 순서로 표를 읽느냐에 따라 성능 차이가 크다.
- 선택, 투영, 결합 같은 관계 대수 연산은 질의의 내부 동작을 이해하는 틀로 쓰이며, 불필요한 열을 먼저 줄이면 메모리 사용이 감소한다.
- 결합 순서는 매우 중요해서 작은 결과를 먼저 만드는 계획이 보통 유리하며, 학생 표와 수강 표를 바로 크게 합치는 것보다 조건을 먼저 거는 편이 낫다.
- 질의 최적화기는 통계 정보를 보고 비용이 낮은 계획을 고르는데, 통계가 오래되면 좋은 인덱스가 있어도 엉뚱한 계획을 택할 수 있다.
- 중첩 반복 결합, 정렬 병합 결합, 해시 결합은 각각 장단점이 다르며, 데이터 크기와 정렬 여부에 따라 적절한 방식이 달라진다.
- 뷰는 복잡한 질의를 단순하게 보이게 하지만, 매번 계산되는 뷰와 결과를 저장하는 물질화 뷰는 성능과 최신성에서 차이가 난다.
- 실제 운영에서는 질의문 한 줄보다 실행 계획을 읽는 능력이 더 중요하며, 병목이 어디서 생기는지 해석해야 근본적인 개선이 가능하다.

## 트랜잭션과 동시성 제어의 실제 의미

- 트랜잭션은 여러 작업을 하나의 논리 단위로 묶는 것으로, 계좌이체에서 출금과 입금이 함께 성공하거나 함께 취소되어야 하는 예가 대표적이다.
- 원자성은 전부 수행되거나 전부 취소되는 성질이고, 중간에 전원이 꺼져도 반쪽짜리 상태가 남지 않게 해 준다.
- 일관성은 규칙이 깨지지 않는 상태를 유지하는 성질이며, 재고가 음수가 되지 않아야 한다는 제약이 이에 해당한다.
- 격리성은 동시에 실행되는 작업이 서로 부당하게 간섭하지 않게 하는 성질로, 두 사용자가 같은 좌석을 동시에 예약하는 문제를 막는 데 필요하다.
- 지속성은 완료된 결과가 장애 후에도 남는 성질이며, 결제 완료 후 서버가 재시작되어도 주문 기록이 유지되어야 한다.
- 동시성 문제로는 더러운 읽기, 반복 불가능 읽기, 팬텀 현상 등이 있으며, 격리 수준을 높일수록 안전하지만 성능 비용이 커질 수 있다.
- 잠금 기반 방식과 다중 버전 방식은 대표적인 제어 기법인데, 읽기 중심 서비스에서는 다중 버전 방식이 사용자 체감 지연을 줄이는 경우가 많다.

## 로그와 복구가 장애 이후를 책임지는 구조

- 복구는 장애가 난 뒤 데이터베이스를 일관된 상태로 되돌리는 과정이며, 단순 백업만으로는 최근 변경분을 모두 살리기 어렵다.
- 로그는 변경 내역을 순서대로 남기는 기록으로, 주문 금액을 바꾸기 전 값과 후 값을 저장해 두면 되돌리기와 다시 실행이 가능하다.
- 선기록 후갱신 원칙은 실제 데이터 페이지보다 로그를 먼저 안정 저장하는 규칙이며, 이것이 있어야 장애 후에도 복구 근거가 남는다.
- 즉시 갱신 방식은 실행 중에도 디스크 반영이 가능하지만 취소 정보가 꼭 필요하고, 지연 갱신 방식은 단순하지만 적용 시점이 늦다.
- 검사점은 복구 시작 위치를 줄여 주는 기준점으로, 긴 로그를 처음부터 다 읽지 않아도 되게 만들어 복구 시간을 단축한다.
- 되돌리기는 완료되지 않은 작업을 취소하고, 재실행은 완료되었지만 디스크에 완전히 반영되지 않은 작업을 다시 적용하는 차이가 있다.
- 실무에서는 복구 이론보다 복구 시간 목표와 데이터 손실 허용치가 더 직접적인 판단 기준이므로, 백업 주기와 로그 보관 전략을 함께 설계해야 한다.

## 비관계형 모델을 선택해야 하는 상황

- 비관계형 데이터베이스는 고정된 표 구조보다 유연한 저장을 중시하며, 사용자마다 속성이 다른 상품 정보를 다루는 전자상거래에 잘 맞을 수 있다.
- 문서형 모델은 한 객체를 묶어서 저장하므로 게시글과 댓글 일부를 함께 읽는 서비스에서 개발 속도가 빠를 수 있다.
- 키값 모델은 매우 단순하고 빠르며, 로그인 세션이나 임시 인증 토큰처럼 짧게 저장하고 빨리 찾는 작업에 자주 쓰인다.
- 열 지향 모델은 대규모 분석에 강하고, 시계열 로그나 클릭 기록처럼 열 단위 집계가 많은 경우 효율적일 수 있다.
- 그래프 모델은 사람 관계나 추천 경로처럼 연결 자체가 중요한 문제에서 강점을 보이며, 친구의 친구 추천 같은 질의가 자연스럽다.
- 관계형 모델은 강한 일관성과 복잡한 결합에 유리하고, 비관계형 모델은 유연한 확장과 대규모 분산 처리에 유리한 편이라 요구사항 비교가 핵심이다.
- 실무에서는 하나만 고집하기보다 주문은 관계형, 캐시는 키값, 추천은 그래프처럼 목적별로 혼합하는 다중 저장소 전략이 흔하다.

## 최신 흐름

- 벡터 검색이 데이터베이스 내부 기능으로 통합되는 흐름이 강해져, 별도 저장소를 두지 않고 추천이나 의미 기반 검색을 함께 처리하는 사례가 늘고 있다.
- 서버리스 데이터베이스가 확산되면서 개발자는 서버 증설보다 사용량 기반 과금과 자동 확장에 더 익숙해지고 있으며, 학습할 때도 운영 자동화 개념이 중요해졌다.
- 분산 구조적 질의 언어 데이터베이스가 관계형 일관성과 수평 확장을 함께 제공하려고 발전하고 있어, 전통적 단일 서버 모델만으로는 설명이 부족해지고 있다.
- 트랜잭션 처리와 분석 처리를 한 플랫폼에서 묶으려는 흐름이 커져, 운영 데이터와 분석 데이터를 따로 복제하던 구조가 점차 단순화되고 있다.
- 레이크하우스 연계가 보편화되면서 데이터베이스만 공부해도 파일 저장소, 배치 처리, 분석 엔진과의 연결 관점을 함께 이해해야 실무 맥락이 잡힌다.
- 하이브리드 검색이 중요해져, 키워드 검색과 벡터 검색을 결합해 정확도와 의미 이해를 동시에 높이는 방식이 실제 서비스 품질 개선에 쓰이고 있다.
- 자동 임베딩 생성과 같은 인공지능 연동 기능이 늘어, 개발자는 모델과 데이터 저장을 분리하기보다 한 흐름으로 설계하는 역량이 요구되고 있다.
- 다지역 고가용성 요구가 커지면서 장애가 나도 쓰기 작업을 계속 받는 분산 설계가 주목받고 있고, 이는 복제와 일관성 학습의 중요성을 다시 높인다.
- 관측 가능성과 비용 최적화가 핵심 운영 지표가 되어, 단순히 빠른 데이터베이스보다 어떤 질의가 얼마의 자원을 쓰는지 설명 가능한 구조가 선호된다.
- 결론적으로 최근 데이터베이스 학습은 저장 구조만 외우는 수준을 넘어, 인공지능 검색, 분산 확장, 운영 자동화까지 함께 보는 방향으로 넓어지고 있다.
