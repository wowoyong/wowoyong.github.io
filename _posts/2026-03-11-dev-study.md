## 핵심 요약

컴파일러는 고수준 프로그래밍 언어로 작성된 소스 코드를 기계가 실행할 수 있는 저수준 코드로 변환하는 핵심 시스템 소프트웨어이다. 어휘 분석, 구문 분석, 의미 분석, 중간 코드 생성, 최적화, 코드 생성의 여섯 단계를 거치며 각 단계가 명확히 분리되어 있다. 실제 현업에서는 LLVM과 같은 공통 백엔드 구조가 여러 언어와 플랫폼을 동시에 지원하는 핵심 인프라로 자리잡았다. 최근에는 즉시 실행(JIT) 컴파일 기법과 점진적 컴파일 방식이 성능과 개발 편의성을 동시에 높이는 방향으로 발전하고 있다. 또한 인공지능 기반 코드 최적화와 언어 서버 프로토콜(LSP)의 보편화가 컴파일러 기술의 활용 범위를 개발 도구 전반으로 확장시키고 있으며, 웹어셈블리의 등장으로 웹 환경에서도 네이티브 수준의 성능이 가능해졌다.

## 컴파일러 vs 인터프리터: 번역 방식의 차이와 절충점

- 컴파일러는 소스 코드 전체를 한 번에 분석해 기계어로 변환한다 — 인터프리터는 한 줄씩 읽고 즉시 실행하므로 실행 속도는 느리지만 개발 중 즉각적인 결과 확인이 가능하다
- 컴파일 언어의 대표는 C, C++, Rust이며 인터프리터 언어는 Python, Ruby가 대표적이다 — Java는 바이트코드로 컴파일 후 JVM이 인터프리트하는 혼합 방식을 채택한다
- 크로스 컴파일러는 컴파일 환경과 실행 환경이 다를 때 사용한다 — 예: x86 PC에서 ARM 기반 스마트폰용 코드를 컴파일하는 임베디드 개발 시나리오
- 트랜스파일러(소스 간 컴파일러)는 같은 추상화 수준의 언어 간 변환을 수행한다 — TypeScript → JavaScript, Babel을 통한 최신 JS → 구형 JS 변환이 대표 사례
- 컴파일러의 전단부(프런트엔드)는 언어 종속적 분석을 담당하고 후단부(백엔드)는 플랫폼 종속적 코드 생성을 담당해 두 축을 독립적으로 교체하거나 재사용할 수 있다
- 사전 컴파일(AOT)은 배포 전 전체를 컴파일해 실행 속도를 극대화하고, 즉시 실행 컴파일(JIT)은 실행 중 자주 호출되는 경로만 동적으로 컴파일해 유연성과 성능을 동시에 확보한다
- 컴파일 시점에 많은 오류를 잡을수록 더 안전한 소프트웨어가 된다 — 정적 타입 언어(Rust, Java)는 타입 오류를 실행 전에 잡아내어 런타임 충돌을 줄인다

## 어휘 분석: 소스 코드를 의미 있는 토큰 스트림으로 변환하기

- 어휘 분석기(렉서/스캐너)는 소스 코드를 읽어 예약어, 식별자, 연산자, 상수, 구두점 같은 최소 의미 단위인 토큰들의 흐름으로 변환한다
- 정규 표현식과 유한 오토마타로 토큰 패턴을 정의하고 인식한다 — 예: `[a-zA-Z][a-zA-Z0-9]*`는 식별자 패턴, `[0-9]+`는 정수 상수 패턴
- 공백, 주석, 줄바꿈은 어휘 분석 단계에서 제거되어 구문 분석기로 넘어가지 않는다 — 소스 코드에서 의미 있는 정보만 추출하는 필터 역할을 한다
- 어휘 오류의 예: `@`처럼 해당 언어가 정의하지 않은 문자가 등장하면 렉서가 즉시 오류를 보고하고 이후 단계로 진행하지 않는다
- Flex는 정규 표현식 기반 렉서를 자동 생성하는 도구로, Bison(구문 분석기 생성기)과 함께 C 컴파일러 개발에 전통적으로 많이 사용된다
- 토큰에는 종류(태그)와 값이 함께 저장된다 — 예: 숫자 `42`는 태그=정수_상수, 값=42 / 키워드 `if`는 태그=예약어, 값=if
- 심벌 테이블은 어휘 분석 단계부터 구축되기 시작해 이후 모든 단계에서 변수·함수 이름과 속성 정보를 공유하는 핵심 자료구조로 활용된다

## 구문 분석: 문법 규칙으로 추상 구문 트리(AST) 만들기

- 구문 분석기(파서)는 토큰 스트림을 받아 문맥 자유 문법에 따라 파스 트리 또는 추상 구문 트리(AST)를 생성한다
- 상향식 파싱(LR 파싱): 입력을 차례로 읽으며 문법 규칙에 맞게 축약해 나가는 방식 — YACC/Bison이 대표 도구이며 복잡한 문법도 자동으로 처리할 수 있다
- 하향식 파싱(LL 파싱, 재귀 하강): 시작 기호에서 출발해 입력을 예측하며 유도 — 문법 규칙을 함수로 직접 구현할 수 있어 파서를 직접 작성할 때 선호된다
- 추상 구문 트리(AST)는 불필요한 괄호·구두점을 제거하고 의미 구조만 남긴 트리로, 이후 의미 분석과 코드 생성의 공통 입력이 된다
- 구문 오류 예: `if (x > 0` — 닫는 괄호가 없으면 파서가 이를 감지하고, 가능하면 오류 복구를 시도한 뒤 분석을 이어가 추가 오류도 보고한다
- Pratt 파서는 연산자 우선순위를 우아하게 처리하는 하향식 파싱 기법으로, Rust·TypeScript 컴파일러 등 현대 언어의 손수 작성 파서에서 많이 채택된다
- 파싱 테이블은 현재 상태와 입력 토큰에 따라 파서가 취할 행동(이동/축약)을 정의한 표로, LR 파서의 핵심 자료구조이며 자동 생성 도구가 이를 만들어 준다

## 의미 분석: 타입 검사와 범위 규칙으로 논리 오류 잡기

- 의미 분석기는 구문적으로는 올바르지만 의미적으로 틀린 코드를 검출한다 — 예: 선언하지 않은 변수 사용, 정수형 변수에 문자열 대입
- 타입 검사는 정적(컴파일 시점)과 동적(실행 시점) 두 종류로 나뉜다 — C·Rust·Java는 정적 타입 언어, Python·JavaScript는 동적 타입 언어로 각각 장단점이 다르다
- 범위(스코프) 분석은 심벌 테이블을 통해 각 변수·함수가 어떤 블록에서 유효한지 결정하고, 같은 이름이 다른 스코프에서 충돌하지 않도록 관리한다
- 타입 추론: Rust나 Kotlin은 선언 시 타입을 명시하지 않아도 컴파일러가 문맥에서 추론한다 — 코드 간결성과 정적 안전성을 동시에 확보하는 현대적 기법
- 속성 문법은 AST 노드에 타입·값 등의 속성을 부여하고 규칙으로 전파하는 의미 분석의 형식화 기법으로 이론적 기반이 된다
- 함수 중복 정의 해결: 같은 이름의 함수 여러 개 중 인자 타입에 맞는 것을 선택하는 것도 의미 분석 단계에서 처리되며, C++에서는 이 과정이 상당히 복잡하다
- 의미 분석이 끝나면 심벌 테이블이 완성되고 AST 각 노드에 타입 정보가 추가되어 중간 코드 생성 단계로 넘어간다

## 중간 표현(IR)과 최적화: 플랫폼 독립적 변환으로 성능 끌어올리기

- 중간 표현(IR)은 특정 CPU 구조에 종속되지 않은 추상적 코드로, 전단부와 후단부를 분리해 여러 언어·플랫폼 조합을 지원하기 쉽게 만든다
- 3-주소 코드: 피연산자 2개와 결과 1개로 구성된 명령어 형태 — 예: `t1 = a + b; t2 = t1 * c` — 사람이 읽기 쉽고 최적화 분석도 편리하다
- 정적 단일 대입(SSA) 형식: 각 변수가 ��확히 한 번만 대입되는 IR 형태로, 대부분의 현대 최적화 알고리즘이 이를 전제로 동작한다
- LLVM IR은 현대 컴파일러 생태계의 사실상 표준 중간 표현 — Clang(C/C++), Rust, Swift가 LLVM IR을 공통 백엔드로 사용해 최적화와 코드 생성을 공유한다
- 루프 최적화(불변 코드 이동, 루프 펼침)와 함수 인라인 치환은 실행 속도에 큰 영향을 주는 대표적 최적화 기법이다
- 상수 접기: `3 + 4`를 컴파일 시점에 `7`로 미리 계산한다 — 가장 단순하지만 효과적인 최적화로 거의 모든 컴파일러가 기본 적용한다
- 불용 코드 제거: 어떤 실행 경로로도 도달할 수 없는 코드를 삭제해 바이너리 크기와 실행 시간을 줄인다
- 최적화 수준(O0~O3, Os)은 컴파일 속도와 실행 속도의 절충점을 조절하며, 디버깅 시에는 O0로 최적화를 끄고 심벌 정보를 보존한다

## 코드 생성: 레지스터 할당과 명령어 선택으로 기계어 완성하기

- 코드 생성기는 IR을 특정 CPU(x86-64, ARM 등)의 기계어 또는 어셈블리로 변환하는 컴파일러 후단부의 핵심 단계이다
- 명령어 선택: IR의 각 연산을 하나 이상의 목표 기계 명령어로 대응시키는 과정 — 타일링 기법으로 최적 명령어 조합을 찾아 코드 크기와 실행 속도를 균형 있게 맞춘다
- 레지스터 할당: CPU 레지스터 수가 제한적이므로 어떤 변수를 레지스터에, 어떤 것을 메모리 스택에 저장할지 결정한다 — 그래프 채색 알고리즘이 대표적 기법이다
- 스필: 레지스터가 부족할 때 변수를 스택에 임시 저장하는 것 — 스필이 많아지면 메모리 접근이 증가해 성능이 크게 저하된다
- 명령어 스케줄링: 데이터 의존성을 지키면서 명령어 실행 순서를 재배치해 CPU 파이프라인 정체를 줄이고 처리량을 높인다
- 응용 프로그램 이진 인터페이스(ABI)는 함수 호출 규약(인자 전달 방식, 반환값 저장 레지스터 등)을 정의해 서로 다른 언어나 컴파일 단위 간 연동을 가능하게 한다
- 목적 파일과 링킹: 컴파일러는 목적 파일(.o)을 생성하고, 링커가 여러 목적 파일과 라이브러리를 합쳐 최종 실행 파일을 만드는 두 단계로 빌드가 완성된다

## 최신 흐름

- LLVM 기반 컴파일러 생태계 확장: Clang, Rust, Swift, Kotlin/Native 등이 LLVM을 공통 백엔드로 채택하며 플랫폼 간 이식성과 최적화 재사용이 표준화되었다
- 웹어셈블리(Wasm) 컴파일 대상화: C, Rust, Go 등 다양한 언어가 Wasm을 컴파일 목표로 지원해 웹 브라우저에서도 네이티브 수준의 성능으로 코드를 실행할 수 있게 되었다
- 점진적·증분 컴파일의 부상: 변경된 파일만 재컴파일하는 기법이 대규모 프로젝트 빌드 시간을 수십 배 단축시켰다 — Rust의 증분 컴파일, TypeScript의 tsbuildinfo가 대표 사례
- 언어 서버 프로토콜(LSP)의 보편화: 컴파일러 전단부 기술이 편집기의 자동 완성·오류 표시·리팩터링에 실시간으로 활용되며 개발 경험을 혁신했다
- 인공지능 기반 코드 최적화 연구: 딥러닝 모델이 컴파일러 최적화 순서나 휴리스틱을 결정하는 연구가 활발히 진행 중이다 — Meta의 CompilerGym이 대표적인 연구 환경이다
- Cranelift의 등장: Rust로 작성된 경량 코드 생성 백엔드로, 웹어셈블리 런타임 Wasmtime과 Firefox에서 즉시 실행 컴파일러로 채택되어 빠른 코드 생성 속도를 입증했다
- 트리-시터(Tree-sitter): 파서를 증분 구문 분석 방식으로 실시간 파싱하는 라이브러리로, Neovim 플러그인·GitHub 코드 강조 표시·인공지능 코드 분석 도구에 광범위하게 사용된다
- 다수준 중간 표현(MLIR): 서로 다른 추상화 수준의 중간 표현을 하나의 프레임워크에서 다루는 구조로, 머신러닝 컴파일러(IREE, TensorFlow XLA)와 도메인 특화 언어 개발에 핵심 기반이 되고 있다
- 다면체 모델 기반 자동 벡터화: 루프 변환을 수학적으로 정형화해 SIMD 명령어를 자동 생성하는 기술이 과학 계산·머신러닝 컴파일러에서 핵심 성능 최적화 기법으로 자리잡았다
- 조합 가능한 컴파일러: 파서·타입 시스템·코드 생성기를 플러그인처럼 조합해 새 언어를 빠르게 시제품으로 만드는 Menhir, Cranelift, MLIR 기반 프레임워크가 부상하고 있다
