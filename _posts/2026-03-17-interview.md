## 오늘의 면접 주제
> Java 백엔드 채용에서는 Spring 기반 서비스 이해를 확인하기 위해 JVM과 GC를 매우 자주 묻습니다.

## 핵심 한 줄 요약
JVM은 메모리 영역을 역할별로 나눠 관리하고, GC는 Heap의 불필요한 객체를 자동으로 회수해 성능과 안정성을 맞추는 메커니즘입니다.

## 면접 질문 & 모범 답변

### Q1. JVM은 무엇이고 왜 필요한가요?
**기본 답변** (2-3문장): JVM은 Java 바이트코드를 실행하는 가상 머신입니다. 운영체제마다 실행 방식이 달라도, Java 코드는 JVM 위에서 같은 형태로 동작할 수 있습니다. 그래서 "Write Once, Run Anywhere"가 가능합니다.  
**심화 포인트**: JVM은 단순 실행기만이 아닙니다. 메모리 관리, GC, JIT 컴파일까지 담당합니다. 즉, 성능과 안정성에 직접 영향을 줍니다.

### Q2. JVM의 메모리 구조를 설명해보세요.
**기본 답변** (2-3문장): 대표적으로 Stack, Heap, Metaspace가 있습니다. Stack에는 메서드 호출 정보와 지역 변수가 쌓입니다. Heap에는 `new`로 만든 객체가 저장되고, Metaspace에는 클래스 메타데이터가 올라갑니다.  
**심화 포인트**: Stack은 스레드마다 따로 생기고, Heap은 여러 스레드가 함께 사용합니다. 그래서 Stack 문제는 보통 개별 호출 깊이와 관련되고, Heap 문제는 객체 생성량과 누수와 더 자주 연결됩니다.

### Q3. Stack과 Heap의 차이를 어떻게 설명하시겠어요?
**기본 답변** (2-3문장): Stack은 메서드 실행이 끝나면 자동으로 정리되는 임시 작업 공간입니다. Heap은 객체가 살아 있는 동안 유지되는 공용 저장소입니다. 비유하면 Stack은 책상 위 메모지이고, Heap은 창고입니다.  
**심화 포인트**: 지역 변수 자체는 Stack에 있어도, 그 변수가 가리키는 객체는 Heap에 있습니다. 이 차이를 헷갈리면 메모리 동작을 잘못 설명하게 됩니다.

### Q4. GC는 왜 필요한가요?
**기본 답변** (2-3문장): GC는 더 이상 참조되지 않는 객체를 찾아 메모리를 회수합니다. 개발자가 직접 `free`를 호출하지 않아도 되기 때문에 생산성과 안전성이 높아집니다. 특히 대규모 서비스에서는 메모리 해제를 사람이 일일이 관리하기 어렵습니다.  
**심화 포인트**: GC가 있다고 해서 메모리 문제가 사라지지는 않습니다. 참조가 남아 있으면 실제로 안 쓰는 객체도 회수되지 않기 때문에, 캐시나 static 컬렉션 때문에 누수가 자주 발생합니다.

### Q5. GC는 어떤 방식으로 동작하나요?
**기본 답변** (2-3문장): GC는 "도달 가능한 객체"만 살려두고 나머지를 정리합니다. 보통 GC Root에서 시작해 참조를 따라가며 살아 있는 객체를 표시합니다. 표시되지 않은 객체를 제거하는 방식이 기본입니다.  
**심화 포인트**: 핵심은 객체가 필요하냐가 아니라 "참조 가능하냐"입니다. 예를 들어 아래 코드에서 `user`가 `null`이 되면 객체는 회수 대상이 될 수 있습니다.

```java
User user = new User();
user = null;
```

### Q6. Serial GC, G1 GC, ZGC의 차이를 설명해보세요.
**기본 답변** (2-3문장): Serial GC는 단순하지만 GC 중 애플리케이션이 멈추는 시간이 깁니다. G1 GC는 Heap을 여러 영역으로 나눠 우선순위 기반으로 회수해 균형 잡힌 성능을 제공합니다. ZGC는 매우 짧은 pause time을 목표로 해서 대용량 메모리 환경에 유리합니다.  
**심화 포인트**: 면접에서는 "무조건 최신 GC가 최고"라고 답하면 위험합니다. 서비스 크기, 응답 시간 요구사항, 운영 복잡도에 따라 선택이 달라집니다.

### Q7. 실무에서 GC 문제를 어떻게 확인하고 대응하나요?
**기본 답변** (2-3문장): 응답 지연이 생기면 GC log, Heap usage, pause time부터 확인합니다. 객체가 너무 빨리 많이 생성되는지, 오래 살아남는 객체가 많은지 봅니다. 필요하면 Heap dump를 떠서 누수 의심 객체를 분석합니다.  
**심화 포인트**: GC 튜닝 전에 코드 레벨 문제를 먼저 봐야 합니다. 예를 들어 불필요한 객체 생성, 과한 JSON 변환, 무제한 캐시가 더 근본 원인인 경우가 많습니다.

### Q8. GC의 장점만 있는 것은 아닌데, 어떤 트레이드오프가 있나요?
**기본 답변** (2-3문장): GC는 메모리 관리 부담을 줄여주지만, 수집 시점에 pause가 발생할 수 있습니다. 또 CPU를 추가로 사용하므로 처리량에 영향을 줄 수 있습니다. 즉, 개발 편의성과 런타임 비용을 바꾸는 선택입니다.  
**심화 포인트**: 저지연 시스템에서는 짧은 pause도 문제입니다. 그래서 단순히 평균 성능보다 tail latency를 같이 봐야 합니다.

## 헷갈리기 쉬운 포인트
Metaspace는 Heap이 아닙니다. 클래스 메타데이터를 저장하는 별도 영역입니다.  
지역 변수가 Stack에 있다고 해서, 그 변수가 가리키는 객체까지 Stack에 있는 것은 아닙니다. 객체는 대부분 Heap에 있습니다.  
GC가 실행된다고 바로 메모리가 항상 크게 줄어드는 것은 아닙니다. 참조가 남아 있으면 회수되지 않습니다.

## 면접관 시각
이 주제에서는 용어 암기보다 구조를 연결해서 설명하는지를 봅니다. Heap, Stack, GC를 따로 말하는 지원자보다, 객체 생성부터 회수까지 흐름으로 설명하는 지원자가 더 강합니다. 또 "GC 튜닝"만 말하지 않고, 코드 레벨 원인과 운영 관점의 지표까지 연결하면 실무 이해도가 있다고 판단합니다.
