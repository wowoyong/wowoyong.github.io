## 핵심 요약

컴파일러는 사람이 작성한 고급 언어 코드를 기계가 실행할 수 있는 형태로 바꾸는 번역기이며, 어휘 분석, 구문 분석, 의미 분석, 최적화, 코드 생성이 핵심 단계입니다. 학부 수준에서는 각 단계가 어떤 입력을 받아 어떤 결과를 내놓는지, 또 어느 단계에서 어떤 오류를 잡는지를 연결해 이해하는 것이 중요합니다. 실제 개발에서는 빌드 시간, 경고 품질, 실행 속도, 이식성이 모두 컴파일러 설계와 직결되므로, 단순한 번역기를 넘어 개발 생산성을 좌우하는 핵심 도구로 봐야 합니다. 예를 들어 타입 검사와 데이터 흐름 분석은 버그를 미리 막아 주고, 최적화와 레지스터 할당은 같은 프로그램도 훨씬 빠르게 실행되도록 만듭니다. 최근에는 MLIR 같은 다단계 중간 표현, 이기종 가속기 대상 컴파일, 기계학습 기반 최적화, WebAssembly 표준 확장이 중요한 흐름으로 자리 잡고 있습니다. 따라서 컴파일러를 공부할 때는 언어 이론뿐 아니라 성능, 도구화, 안전성, 최신 실행 환경까지 함께 살펴보는 것이 효과적입니다.

## 소스 코드에서 목적 코드까지의 번역 단계

- 컴파일러는 보통 앞단, 중간단, 뒷단으로 나뉘며, 앞단은 문법 이해, 중간단은 표현 변환과 최적화, 뒷단은 실제 기계 명령 생성에 집중합니다.
- 입력은 소스 코드이지만, 중간에는 토큰, 구문 트리, 중간 표현 같은 여러 형태를 거치므로 한 번에 번역된다고 생각하면 구조를 놓치기 쉽습니다.
- 예를 들어 `a+b*c` 같은 식은 먼저 기호 단위로 나뉘고, 이후 곱셈이 덧셈보다 먼저라는 우선순위가 반영된 트리로 바뀝니다.
- 이처럼 단계를 나누면 오류 위치를 더 분명하게 파악할 수 있는데, 변수 이름 오타는 의미 분석에서, 괄호 누락은 구문 분석에서 주로 발견됩니다.
- 실무에서는 같은 앞단을 공유하고 뒷단만 바꿔 여러 기계를 지원하기도 하며, 그래서 이식성이 높은 도구 모음이 만들어집니다.
- 단일 언어 전용 컴파일러보다 공통 기반 구조를 쓰는 방식이 유지보수에 유리한데, 여러 언어가 같은 최적화기를 재사용할 수 있기 때문입니다.
- 학습할 때는 각 단계의 목적을 외우기보다 입력과 출력이 무엇인지 표처럼 정리하면 전체 흐름이 훨씬 선명해집니다.

## 어휘 분석과 구문 분석이 맡는 서로 다른 역할

- 어휘 분석은 문자 흐름을 의미 있는 최소 단위로 나누는 과정이고, 구문 분석은 그 단위들이 문법에 맞게 배열됐는지 확인하는 과정입니다.
- 예를 들어 `while`, 식별자, 숫자, 연산자, 괄호는 어휘 분석 결과인 토큰이 되고, 반복문 구조가 올바른지는 구문 분석이 판단합니다.
- 공백과 주석은 대개 실행 의미가 없어 제거되지만, 들여쓰기가 문법인 언어에서는 공백도 중요한 입력이 될 수 있다는 점이 비교 포인트입니다.
- 구문 분석은 보통 트리 구조를 만드는데, 이 트리가 이후 의미 분석과 최적화의 기준이 되므로 단순한 문법 검사로 끝나지 않습니다.
- 좌재귀 제거, 우선순위 처리 같은 문법 변형은 이론 문제처럼 보이지만 실제로는 파서 구현 가능성과 오류 메시지 품질에 직접 영향을 줍니다.
- 좋은 오류 복구 전략이 있으면 괄호 하나가 틀려도 이후 오류가 연쇄적으로 늘어나는 것을 줄여 개발자가 원인을 더 빨리 찾을 수 있습니다.
- 실전에서는 어휘 분석기는 빠르고 단순할수록 좋지만, 구문 분석기는 사람이 이해하기 쉬운 진단을 내놓는 능력도 매우 중요합니다.

## 의미 분석과 타입 검사가 버그를 줄이는 방식

- 의미 분석은 문법상 맞는 문장이 실제 언어 규칙에도 맞는지 확인하는 단계이며, 선언 전 사용, 타입 불일치, 반환값 규칙 위반 등을 잡아냅니다.
- 예를 들어 정수 변수에 문자열을 대입하거나, 반환형이 없는 함수에서 값을 돌려주려 하면 구문은 맞아도 의미적으로는 오류입니다.
- 타입 검사는 연산 가능 여부를 결정하는데, 같은 `+`라도 정수 덧셈인지 문자열 이어 붙이기인지에 따라 처리 방식이 달라집니다.
- 심벌 테이블은 변수와 함수의 이름, 범위, 타입을 저장하는 핵심 구조로, 중첩 블록에서 같은 이름이 어떻게 해석되는지 결정합니다.
- 정적 검사 언어는 실행 전에 많은 오류를 잡는 대신 제약이 강하고, 동적 검사 언어는 유연하지만 실행 중 오류 가능성이 커진다는 비교가 중요합니다.
- 실무에서는 경고 수준이 높을수록 의미 분석의 가치가 커지는데, 사용되지 않는 변수나 도달할 수 없는 코드도 품질 문제로 다뤄지기 때문입니다.
- 학부 과정에서는 타입 추론, 범위 규칙, 오버로딩 해석을 함께 보면 언어 설계와 컴파일러 구현이 어떻게 연결되는지 잘 드러납니다.

## 중간 표현이 최적화의 중심이 되는 이유

- 중간 표현은 원시 언어와 기계어 사이의 공통 표현으로, 여러 최적화를 언어와 기계에 덜 종속적으로 적용할 수 있게 해 줍니다.
- 대표적인 예로 세 주소 코드나 정적 단일 대입 형식이 있는데, 후자는 변수 값의 흐름이 선명해 데이터 흐름 분석에 특히 유리합니다.
- 예를 들어 같은 값을 여러 번 계산하면 공통 부분식 제거를 통해 한 번만 계산하도록 바꿀 수 있고, 결과는 같지만 실행 비용은 줄어듭니다.
- 상수 전파는 `x=3`이 확실할 때 뒤의 계산에 3을 직접 넣는 방식이며, 이어서 죽은 코드 제거로까지 연결되는 경우가 많습니다.
- 반복문 최적화는 성능 체감이 큰데, 배열 합 계산처럼 많이 반복되는 부분에서 불필요한 메모리 접근을 줄이면 큰 차이가 납니다.
- 다만 최적화가 언제나 좋은 것만은 아니어서, 디버깅이 어려워지거나 컴파일 시간이 늘어나는 대가가 생길 수 있습니다.
- 실무 관점에서 보면 중간 표현은 단순한 내부 자료구조가 아니라 성능, 이식성, 분석 가능성을 함께 좌우하는 설계의 중심입니다.

## 코드 생성과 레지스터 할당이 성능을 좌우하는 이유

- 코드 생성은 중간 표현을 실제 기계 명령으로 바꾸는 단계이며, 대상 구조의 명령 집합과 호출 규약을 정확히 반영해야 합니다.
- 같은 덧셈이라도 어떤 명령을 쓰고 어느 레지스터에 둘지에 따라 속도와 코드 크기가 달라지므로 단순한 치환 문제가 아닙니다.
- 레지스터 할당은 많은 임시 값을 적은 수의 레지스터에 배치하는 문제인데, 부족하면 메모리에 임시 저장하는 넘침이 발생해 성능이 떨어집니다.
- 예를 들어 반복문 안에서 자주 쓰는 값을 레지스터에 남기면 빠르지만, 계속 메모리로 내보내면 같은 계산도 훨씬 느려질 수 있습니다.
- 명령 선택과 명령 스케줄링은 대상 구조의 특성을 활용하는 부분으로, 같은 기능이라도 더 짧��나 더 병렬적인 명령열을 만들 수 있습니다.
- 이식성 관점에서는 앞단이 같아도 뒷단이 다르면 전혀 다른 기계에서 실행 가능한 코드가 나오므로 뒷단의 품질이 제품 범위를 결정합니다.
- 시험에서는 이 부분이 추상적으로 느껴지기 쉽지만, 실제로는 실행 속도와 전력 사용량에 직접 연결되는 가장 현실적인 단계입니다.

## 인터프리터와 가상기계와 비교해 보는 컴파일러 활용

- 인터프리터는 보통 코드를 읽으면서 바로 실행하고, 컴파일러는 미리 번역한 뒤 실행한다는 차이가 있어 실행 시점의 비용 구조가 다릅니다.
- 가상기계 방식은 중간 바이트코드를 만든 뒤 실행 환경이 다시 해석하거나 즉시 기계어로 바꾸므로, 순수 컴파일과 순수 해석의 중간 형태로 볼 수 있습니다.
- 예를 들어 학습용 언어는 구현이 쉬운 해석 방식을 택하기 쉽고, 성능이 중요한 시스템 프로그램은 보통 미리 컴파일하는 방식이 유리합니다.
- 즉시 번역은 실행 중 실제 경로를 보고 최적화할 수 있어 장점이 있지만, 시작 시간이 길어질 수 있다는 점에서 미리 번역과 대비됩니다.
- 오류가 드러나는 시점도 다릅니다. 정적 컴파일은 실행 전에 많은 오류를 잡고, 해석 중심 환경은 특정 경로를 실행할 때까지 오류가 드러나지 않을 수 있습니다.
- 실무에서는 둘 중 하나만 쓰는 것이 아니라, 정적 컴파일과 즉시 번역, 바이트코드 배포를 함께 섞는 혼합 전략이 흔합니다.
- 이 비교를 이해하면 컴파일러를 단순한 과목 주제가 아니라 언어 실행 모델 전체를 설계하는 도구로 볼 수 있습니다.

## 최신 흐름

- 다단계 중간 표현인 MLIR 계열이 확산되면서 하나의 중간 표현만으로는 다루기 어려운 인공지능, 질의 처리, 가속기 대상 컴파일을 더 유연하게 연결하려는 흐름이 강해졌습니다.
- 이기종 하드웨어 대응이 핵심 과제가 되면서 중앙처리장치뿐 아니라 그래픽 처리 장치, 인공지능 가속기, 공간 데이터 흐름 가속기까지 고려한 컴파일 파이프라인이 늘고 있습니다.
- 기계학습 기반 최적화는 연구 단계를 넘어 실제 LLVM 계열 생산 환경에서 코드 크기 축소와 성능 향상 문제에 적용되는 사례가 꾸준히 보고되고 있습니다.
- 빌드 시간 자체를 줄이는 일이 중요한 경쟁력이 되면서 병렬 앞단, 더 빠른 연결기, 불필요한 재빌드 축소 같은 방향이 적극적으로 추진되고 있습니다.
- Rust 진영에서는 2025년 성능 조사 결과를 바탕으로 빠른 연결기 기본 적용, 병렬 앞단 안정화, 대체 코드 생성기 활용 같은 실용적 개선이 중요한 과제로 정리됐습니다.
- WebAssembly는 2025년 3.0 표준 완료로 더 큰 주소 공간과 여러 메모리 같은 기능을 갖추며, 웹 밖 실행 환경까지 겨냥하는 컴파일 대상 플랫폼으로 한층 강해졌습니다.
- WebAssembly 생태계에서는 구성 요소 모델과 형식 명세 도구 SpecTec 도입이 진행되며, 언어 표준의 엄밀성과 도구 자동 생성 가능성이 함께 커지고 있습니다.
- 컴파일러 진단도 단순한 오류 출력에서 벗어나 기계가 읽을 수 있는 형식, 더 나은 수정 힌트, 시각적으로 구분되는 비교 정보 제공 쪽으로 발전하고 있습니다.
- 대형 코드베이스를 위한 증분 최적화와 증분 연결의 중요성이 커지면서, 작은 수정에 대해 전체를 다시 최적화하지 않는 방향이 실무 표준으로 자리 잡고 있습니다.
- 도메인 특화 언어와 도메인 특화 컴파일러가 늘어나면서 범용 컴파일러 하나로 모든 문제를 풀기보다, 공통 기반 위에 분야별 변환 규칙을 얹는 구조가 점점 일반화되고 있습니다.
