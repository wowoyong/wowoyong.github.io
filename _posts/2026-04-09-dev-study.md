## 핵심 요약

소프트웨어공학은 프로그램을 그냥 만드는 법이 아니라, 예측 가능하게 설계하고 시험하고 유지보수하는 방법을 다루는 학문이다. 핵심 개념은 요구사항, 설계, 구현, 시험, 형상관리, 유지보수이며, 이들을 따로 보지 않고 전체 수명주기로 연결해서 이해해야 한다. 실제 적용에서는 기능 하나를 빨리 만드는 것보다 변경 비용을 낮추고 협업 충돌을 줄이는 구조를 만드는 일이 더 중요하다. 예를 들어 좋은 모듈 분해와 자동 시험은 초반에는 시간이 들지만, 기능 추가와 오류 수정 속도를 크게 높여 준다. 최근 흐름으로는 인공지능 코드 보조도구의 확산, 내부 개발자 플랫폼 강화, 공급망 보안과 소프트웨어 자재명세서의 중요성 증가가 두드러진다.

## 요구사항을 기능 목록이 아니라 계약으로 보는 관점

- 요구사항은 사용자가 원하는 말을 적는 문서가 아니라, 무엇을 만들고 무엇은 만들지 않을지를 정하는 기준이다. 예를 들어 로그인 기능에 소셜 로그인까지 포함되는지 먼저 못 박아야 범위 팽창을 막을 수 있다.
- 기능 요구사항은 시스템이 해야 할 일을, 비기능 요구사항은 얼마나 빠르고 안전하고 안정적이어야 하는지를 설명한다. 같은 게시판이라도 응답시간 1초 이내 조건이 붙으면 설계 선택이 달라진다.
- 모호한 문장은 개발 위험을 키운다. 사용자 친화적으로 보여야 한다보다 첫 화면에서 주요 버튼이 한 번의 화면 이동 안에 보여야 한다처럼 검증 가능한 표현이 실무에서 훨씬 유리하다.
- 이해관계자가 다르면 같은 요구도 다르게 해석한다. 학생은 사용 편의성을, 운영자는 통계 기능을, 보안 담당자는 접근 통제를 우선하므로 초기에 충돌을 드러내야 한다.
- 사용 사례는 요구사항을 이야기 흐름으로 바꾸어 준다. 예를 들어 수강신청 시스템에서 학생이 과목을 검색하고 신청하고 결과를 확인하는 순서를 적으면 빠진 예외가 잘 보인다.
- 우선순위 결정은 전부 구현하겠다는 약속보다 현실적이다. 반드시 필요한 기능과 나중에 넣어도 되는 기능을 나누면 제한된 일정에서도 핵심 가치를 먼저 전달할 수 있다.
- 요구사항 변경은 실패 신호가 아니라 자연스러운 현상이다. 다만 변경 이력과 영향 범위를 함께 관리해야 하며, 그렇지 않으면 화면 하나 바꿨는데 시험 항목이 연쇄적으로 깨지는 일이 생긴다.

## 좋은 설계는 재사용보다 변경 비용을 낮추는 데 초점이 있다

- 설계의 목적은 코드를 예쁘게 만드는 것이 아니라, 바뀔 가능성이 큰 부분을 분리해 수정 비용을 줄이는 데 있다. 결제 방식이 자주 바뀌는 서비스라면 결제 모듈을 독립시키는 식이다.
- 응집도는 한 모듈이 한 가지 책임에 얼마나 집중하는지를, 결합도는 모듈끼리 얼마나 얽혀 있는지를 본다. 응집도는 높고 결합도는 낮을수록 수정이 쉬운 편이다.
- 계층 구조는 화면, 업무 로직, 저장소 접근을 나누어 생각하게 해 준다. 예를 들어 화면 코드에 데이터베이스 질의를 직접 넣으면 시험도 어렵고 변경 영향도 커진다.
- 추상화는 세부 구현을 가리는 도구다. 같은 저장 기능이라도 파일 저장과 원격 저장을 같은 인터페이스 뒤에 두면 상위 로직은 구현 차이를 덜 신경 써도 된다.
- 설계 패턴은 외워서 끼워 넣는 공식이 아니다. 객체 생성이 복잡할 때 생성 책임을 분리하거나, 상태 변화 알림이 많을 때 관찰자 방식을 쓰는 식으로 문제와 연결해서 써야 한다.
- 과한 일반화는 오히려 비용을 늘린다. 학생 과제 수준의 작은 프로그램에서 미래 확장을 지나치게 가정하면, 기능보다 뼈대 코드가 더 많아져 읽기 어려워질 수 있다.
- 좋은 설계는 문서보다 코드 구조에서 드러난다. 함수 이름, 모듈 경계, 의존 방향이 자연스럽다면 새 팀원이 읽을 때 설명 시간이 줄어드는 것이 실무적 장점이다.

## 구현 품질은 개인 실력보다 팀 규칙에서 더 안정적으로 나온다

- 코딩 표준은 취향 통일이 아니라 협업 비용 절감 장치다. 들여쓰기와 이름 규칙이 맞으면 코드 읽는 속도가 빨라지고 리뷰가 의미 있는 문제에 집중된다.
- 함수는 짧다고 무조건 좋은 것이 아니라, 한 번 읽고 역할이 보이면 좋은 것이다. 예를 들어 입력 검증, 계산, 저장을 한 함수에 몰아넣으면 오류 위치를 찾기 어렵다.
- 중복 코드는 복사해서 빨리 끝내기 쉬우나, 정책이 바뀌었을 때 여러 파일을 동시에 고쳐야 한다. 할인 규칙을 세 군데에 복붙하면 나중에 한 군데만 수정해 버그가 난다.
- 예외 처리는 실패를 숨기는 작업이 아니라, 실패를 통제 가능한 상태로 바꾸는 작업이다. 단순히 오류를 무시하기보다 사용자에게 재시도 방법을 보여 주는 편이 낫다.
- 코드 리뷰는 틀린 사람을 찾는 절차가 아니라 위험을 줄이는 절차다. 경계값 누락, 동시 수정 충돌, 이름 오해 같은 문제는 작성자보다 검토자가 더 잘 발견하는 경우가 많다.
- 문서는 코드를 대체하지 못하지만, 의도 설명에는 필요하다. 특히 왜 이런 구조를 택했는지 짧게 남겨 두면 나중에 같은 논쟁을 반복하지 않아도 된다.
- 개발 속도와 코드 품질은 항상 반대가 아니다. 초기에 입력 검증과 공통 도구를 정리해 두면 이후 기능 추가가 빨라져 전체 일정은 오히려 안정되는 경우가 많다.

## 시험은 오류를 찾는 일만이 아니라 설계 약점을 드러내는 도구다

- 단위 시험은 작은 함수나 모듈을 독립적으로 검증하는 방식이다. 합계 계산 함수처럼 입력과 출력이 분명한 부분에서 특히 효과가 크다.
- 통합 시험은 모듈 연결이 제대로 되는지 본다. 로그인은 성공하지만 세션 저장이 안 되는 문제처럼, 각 부분은 멀쩡해 보여도 연결에서 오류가 날 수 있다.
- 시스템 시험은 사용자가 실제로 겪는 흐름을 확인한다. 회원가입, 로그인, 글 작성, 로그아웃까지 이어 보는 식이라 화면과 서버의 상호작용을 함께 점검할 수 있다.
- 회귀 시험은 기존 기능이 새 변경으로 깨지지 않았는지 확인한다. 공지사항 기능을 고치다가 댓글 기능이 망가지는 일을 막는 데 실무적으로 매우 중요하다.
- 경계값 시험은 정상 입력보다 오류를 더 잘 잡는다. 점수가 0점과 100점일 때, 혹은 수강 인원이 정원과 같을 때처럼 끝 경계를 꼭 확인해야 한다.
- 시험이 어려운 코드는 대개 설계도 복잡하다. 외부 입력과 전역 상태에 강하게 묶여 있으면 가짜 데이터를 넣기 힘들기 때문에, 시험 어려움 자체가 설계 경고 신호가 된다.
- 시험이 많다고 품질이 자동으로 높아지지는 않는다. 자주 실패하는 취약 흐름과 비즈니스 핵심 규칙을 우선 덮는 시험이, 형식적으로 숫자만 채운 시험보다 훨씬 가치 있다.

## 형상관리와 배포 자동화는 협업 실수를 구조적으로 줄인다

- 형상관리는 파일 저장이 아니라 변화 이력을 관리하는 활동이다. 누가 언제 왜 수정했는지 남기면 오류가 생겼을 때 원인을 추적하기 쉬워진다.
- 가지 분기는 기능 실험을 안전하게 격리하는 데 유용하다. 새 검색 기능을 따로 개발하면 배포 중인 안정 버전을 건드리지 않고도 작업할 수 있다.
- 병합 충돌은 도구 문제가 아니라 협업 신호다. 같은 파일을 여러 사람이 동시에 크게 바꾸고 있다면 모듈 경계나 작업 분할이 잘못되었을 가능성이 있다.
- 지속적 통합은 코드를 자주 합치고 즉시 검사하는 방식이다. 며칠씩 따로 작업한 뒤 한꺼번에 합치면 작은 문제도 큰 충돌로 번지기 쉽다.
- 자동 배포는 사람 손을 덜 쓰기 위해서만이 아니라, 같은 절차를 반복 가능하게 만들기 위해 필요하다. 수동 배포는 한 단계 빠뜨려도 원인 파악이 어렵다.
- 배포 전 검사에는 빌드 성공, 시험 통과, 정적 점검, 비밀정보 노출 확인 등이 포함될 수 있다. 이런 문지기 절차가 있으면 급하게 올린 코드의 사고 확률이 내려간다.
- 좋은 배포 체계는 실패를 전제로 설계한다. 새 버전이 문제를 일으키면 즉시 이전 버전으로 되돌리거나 일부 사용자에게만 먼저 적용하는 방식이 실무에서 많이 쓰인다.

## 유지보수성과 품질 지표를 함께 봐야 장기 비용이 보인다

- 소프트웨어 비용의 큰 부분은 처음 개발보다 이후 유지보수에서 나온다. 기능 추가 요청과 오류 수정이 계속 들어오므로, 처음 구조가 나쁘면 시간이 갈수록 손해가 커진다.
- 가독성은 주관적인 미감이 아니라 유지보수 속도와 직결된다. 변수 이름이 역할을 설명하지 못하면 디버깅 때 문맥을 다시 추정해야 해서 시간이 많이 든다.
- 복잡도 지표는 위험 신호로 쓸 수 있다. 조건문과 분기 수가 지나치게 많은 함수는 작은 수정에도 부작용이 생기기 쉬워 우선 정리 대상이 된다.
- 결함 밀도는 코드 양 대비 오류 수를 보는 방식이지만, 숫자 하나만으로 품질을 단정하면 안 된다. 핵심 기능에서 난 한 건의 결함이 주변 기능 여러 건보다 더 치명적일 수 있다.
- 변경 용이성은 새로운 요구를 넣을 때 영향을 받는 파일 수로 체감할 수 있다. 화면 문구 하나 바꾸는데 여러 계층을 다 건드려야 한다면 설계 개선이 필요하다.
- 기술 부채는 지금 편하게 가기 위해 미래 비용을 미루는 선택이다. 시험 없이 빠르게 붙인 기능은 초반에는 이익처럼 보여도, 다음 수정 때 이자를 크게 치르게 된다.
- 품질 관리는 지표를 모으는 일에서 끝나면 안 된다. 예를 들어 결함이 특정 모듈에 반복되면 담당자 탓보다 구조, 시험 범위, 의사소통 방식까지 함께 봐야 개선이 된다.

## 최신 흐름

- 인공지능 코드 보조도구는 선택 기능이 아니라 기본 개발 환경의 일부로 자리 잡고 있다. 다만 초안 생성은 빨라져도 정확성 검토 책임은 여전히 사람에게 남는다.
- 자동 완성 중심 도구에서 한 단계 나아가, 기능 구현과 오류 수정과 시험 작성까지 맡는 자율형 코딩 에이전트 사용이 늘고 있다. 그래서 개발자는 코드 작성자이면서 작업 지시자 역할도 함께 하게 된다.
- 생산성 평가는 코드 줄 수보다 실제 전달 속도와 품질을 함께 보는 방향으로 바뀌고 있다. 인공지능 도입이 무조건 성과를 올린다고 보기보다, 결함과 재작업 증가까지 함께 측정하는 흐름이 강하다.
- 내부 개발자 플랫폼과 플랫폼 공학이 빠르게 확산되고 있다. 공통 배포, 권한, 관측, 개발 환경 구성을 표준화해 개발자가 제품 문제에 더 집중하도록 만드는 방식이다.
- 플랫폼 안에 생성형 인공지능 기능을 넣는 시도가 늘고 있다. 예를 들어 내부 문서 검색, 배포 절차 안내, 변경 영향 요약을 한곳에서 제공해 팀별 편차를 줄이려는 움직임이 보인다.
- 공급망 보안이 소프트웨어공학의 핵심 주제로 올라왔다. 직접 작성한 코드보다 외부 패키지와 빌드 경로에서 위험이 생기기 쉬워, 소프트웨어 자재명세서 관리 중요성이 커지고 있다.
- 비밀정보 유출 방지와 자동 보안 점검이 배포 과정의 기본 문턱이 되고 있다. 특히 인공지능이 만든 코드에서 키 값 하드코딩이나 취약한 예제가 섞일 수 있어 사전 검사가 더 중요해졌다.
- 형식이 더 분명한 언어와 정적 분석 도구의 선호가 높아지고 있다. 큰 협업 환경에서는 자유로운 표현보다 오류를 일찍 잡아 주는 장치가 유지보수 비용을 낮추기 때문이다.
- 개발자 경험이 복지 개념이 아니라 성과 변수로 다뤄지고 있다. 환경 구축이 느리고 문서가 흩어져 있으면 실제 구현 시간보다 대기와 탐색 시간이 더 커진다는 점이 경영 지표로 연결되고 있다.
- 인공지능을 잘 쓰는 것과 맹신하는 것은 구분된다는 인식이 강해지고 있다. 최근에는 빠른 초안 생성보다 검토 기준, 시험 습관, 책임 경계가 있는 팀이 더 안정적인 결과를 낸다는 해석이 많다.
