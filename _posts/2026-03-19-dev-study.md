## 핵심 요약

자료구조는 데이터를 어떤 형태로 저장하고 탐색하며 수정할지 정하는 방법으로, 프로그램의 실행 속도와 메모리 사용량을 크게 좌우한다. 배열, 연결 리스트, 스택, 큐, 해시, 트리, 그래프 같은 구조는 각각 강점과 약점이 뚜렷하므로 문제 조건에 맞춰 선택해야 한다. 실제 개발에서는 단순히 이론적 시간 복잡도만 보지 않고, 캐시 친화성, 구현 난도, 동시 접근, 라이브러리 지원까지 함께 고려한다. 예를 들어 검색이 많으면 해시나 균형 트리를, 순서 유지가 중요하면 연결 구조나 덱을, 우선순위 처리가 중요하면 힙을 택하는 식으로 판단한다. 최근에는 대규모 데이터, 병렬 처리, 메모리 효율, 비휘발성 메모리 환경에 맞춘 자료구조 연구와 실무 활용이 늘고 있다. 따라서 학습할 때는 정의 암기보다 연산 비용 비교, 실제 사용 사례, 최신 흐름까지 함께 이해하는 것이 중요하다.

## 배열과 연결 리스트의 저장 방식 차이 이해하기

- 배열은 연속된 메모리에 원소를 저장하므로 인덱스로 바로 접근할 수 있고, 예를 들어 학생 점수표처럼 순번 조회가 잦을 때 유리하다.
- 연결 리스트는 원소마다 다음 위치 정보를 따로 저장하므로 중간 삽입과 삭제가 쉬우며, 재생목록처럼 순서를 자주 바꾸는 상황에 더 맞는다.
- 배열의 중간 삽입은 뒤 원소를 밀어야 해서 비용이 커지지만, 연결 리스트는 위치만 알면 링크만 바꾸면 되어 구조적으로 단순하다.
- 반대로 연결 리스트는 특정 위치를 찾으려면 처음부터 따라가야 하므로, 임의 접근 성능은 배열보다 불리하다.
- 배열은 캐시에 잘 맞아 실제 실행 속도가 빠른 경우가 많아서, 이론상 비슷한 복잡도여도 실무에서는 더 자주 선택된다.
- 연결 리스트는 각 원소가 포인터를 추가로 가져 메모리 낭비가 생길 수 있고, 작은 데이터에서는 오히려 비효율적일 수 있다.
- 시험에서는 삽입·삭제만 보고 연결 리스트를 고르기 쉽지만, 실무에서는 탐색 빈도와 메모리 지역성까지 함께 판단해야 한다.

## 스택과 큐로 흐름 제어 모델 만들기

- 스택은 가장 나중에 들어온 데이터가 먼저 나가는 구조로, 함수 호출 기록이나 괄호 검사처럼 되돌아가기 흐름에 적합하다.
- 큐는 먼저 들어온 데이터가 먼저 나가는 구조로, 프린터 대기열이나 작업 요청 처리처럼 순서를 보장해야 할 때 쓰인다.
- 스택의 대표 연산은 삽입과 삭제가 한쪽 끝에서만 일어나 구현이 단순하며, 깊이 우선 탐색에서도 자연스럽게 쓰인다.
- 큐는 너비 우선 탐색, 운영체제의 작업 대기, 네트워크 패킷 처리처럼 먼저 온 일을 먼저 처리해야 할 때 직관적이다.
- 배열 기반 스택은 빠르지만 크기 제한 문제가 있고, 연결 기반 스택은 확장성이 좋지만 포인터 관리 부담이 있다.
- 원형 큐는 배열의 앞부분을 다시 활용해 공간 낭비를 줄이며, 단순 큐보다 실무적 구현 가치가 높다.
- 덱은 양쪽 끝 삽입과 삭제가 모두 가능해 스택과 큐의 성격을 함께 가지며, 슬라이딩 윈도우 문제에서 특히 자주 등장한다.

## 해시 테이블의 평균 성능과 충돌 처리 판단하기

- 해시 테이블은 키를 해시 함수로 변환해 저장 위치를 빠르게 찾는 구조로, 사전 검색이나 회원 번호 조회에 매우 자주 쓰인다.
- 평균적으로 삽입, 탐색, 삭제가 빠르지만, 서로 다른 키가 같은 위치로 가는 충돌이 많아지면 성능이 급격히 떨어질 수 있다.
- 체이닝은 같은 버킷에 연결 구조를 두는 방식이라 구현이 유연하고 삭제가 쉬우며, 표준 라이브러리에서도 흔히 보인다.
- 개방 주소법은 빈 칸을 찾아 이동하며 저장하므로 메모리 접근이 연속적일 수 있지만, 삭제 처리와 군집화 문제가 더 까다롭다.
- 좋은 해시 함수는 키를 고르게 퍼뜨려야 하며, 단순 나머지 연산만으로는 데이터 패턴에 따라 편향이 생길 수 있다.
- 순서를 보장하지 않는 해시는 빠른 조회에 강하지만, 범위 검색이나 정렬된 출력이 필요하면 균형 트리보다 불리하다.
- 실무에서는 단순히 평균 상수 시간만 믿지 않고, 충돌 분포, 재해시 비용, 메모리 사용량을 함께 검토해야 안정적이다.

## 트리 구조에서 계층 표현과 탐색 효율 잡기

- 트리는 부모와 자식 관계로 데이터를 계층적으로 표현하며, 파일 시스템이나 조직도처럼 위계가 있는 정보를 다루기 좋다.
- 이진 탐색 트리는 왼쪽은 작은 값, 오른쪽은 큰 값을 두어 탐색을 빠르게 하지만, 한쪽으로 치우치면 연결 리스트처럼 느려질 수 있다.
- 균형 트리는 높이를 일정하게 유지해 최악의 경우도 줄이므로, 데이터 삽입 순서가 불규칙한 실제 환경에서 더 안정적이다.
- 힙은 부모가 자식보다 크거나 작은 조건만 유지하는 특수 트리로, 최댓값이나 최솟값을 반복적으로 꺼낼 때 효과적이다.
- 트리 순회는 전위, 중위, 후위, 레벨 순회로 나뉘며, 예를 들어 수식 트리는 후위 순회가 계산 순서와 잘 맞는다.
- 이진 탐색 트리는 정렬된 출력이 쉽지만, 단순 최대값 추출만 반복한다면 힙이 더 적합해 목적별 선택이 중요하다.
- 실무에서는 검색, 정렬, 우선순위, 계층 표현 중 무엇이 핵심인지 먼저 정해야 알맞은 트리 계열을 고를 수 있다.

## 그래프로 관계 모델링하고 탐색 전략 고르기

- 그래프는 정점과 간선으로 관계를 표현하며, 지도 경로, 사회 연결망, 선수 과목 관계처럼 복잡한 연결을 다루는 데 적합하다.
- 인접 행렬은 구현이 단순하고 두 정점 연결 여부를 바로 확인할 수 있지만, 정점 수가 많고 간선이 적으면 메모리 낭비가 크다.
- 인접 리스트는 간선이 적은 희소 그래프에서 효율적이며, 실제 서비스 데이터처럼 연결이 제한적인 경우에 자주 쓰인다.
- 너비 우선 탐색은 가까운 정점부터 차례로 보므로 최단 간선 수 문제에 강하고, 깊이 우선 탐색은 구조 파악과 백트래킹에 유리하다.
- 위상 정렬은 선행 관계가 있는 작업 순서를 정할 때 사용되며, 강의 수강 순서나 빌드 순서 계산에 실용적이다.
- 최단 경로 문제에서는 단순 그래프인지, 가중치가 있는지, 음수 간선이 있는지에 따라 알고리즘 선택이 달라진다.
- 그래프 문제는 자료구조 선택이 곧 성능 차이로 이어지므로, 표현 방식과 탐색 전략을 함께 설계해야 한다.

## 시간 복잡도보다 중요한 실제 선택 기준 읽기

- 같은 시간 복잡도라도 상수항과 메모리 접근 패턴이 다르면 체감 속도가 크게 달라져, 배열이 연결 리스트보다 빠른 일이 흔하다.
- 삽입과 삭제가 많다는 이유만으로 연결 구조를 고르면 안 ���고, 탐색 빈도와 데이터 크기를 함께 봐야 올바른 선택이 된다.
- 메모리 사용량은 서버 비용과 직접 연결되므로, 포인터가 많은 구조는 대규모 환경에서 부담이 커질 수 있다.
- 동시성이 중요한 환경에서는 잠금 비용, 충돌 가능성, 읽기와 쓰기 비율까지 고려해야 자료구조 선택이 현실적이다.
- 표준 라이브러리 구현은 검증되어 있어 직접 구현보다 안전한 경우가 많으며, 면접식 정답과 실무 선택이 다를 수 있다.
- 정렬 상태 유지가 중요하면 균형 트리나 건너뛰기 리스트가, 단순 키 조회가 많으면 해시가 더 실용적일 수 있다.
- 결국 자료구조 선택은 문제의 연산 비율, 데이터 규모, 하드웨어 특성, 유지보수 난도를 함께 보는 공학적 판단이다.

## 최신 흐름

- 대규모 서비스에서는 캐시 친화적인 배열 기반 구조가 다시 주목받으며, 이론보다 실제 메모리 계층 성능을 더 중시하는 경향이 강해지고 있다.
- 병렬 처리 환경에 맞춘 잠금 최소화 자료구조가 활발히 쓰이며, 다중 코어 서버에서는 단일 스레드 기준 최적화만으로 부족하다.
- 읽기 중심 시스템에서는 불변 자료구조가 안정성과 예측 가능성을 높여 함수형 스타일과 함께 활용 범위가 넓어지고 있다.
- 비휘발성 메모리와 같은 새로운 저장 장치 특성을 반영한 영속 자료구조 연구가 늘어나며, 장애 복구 관점의 설계가 중요해지고 있다.
- 대규모 검색과 추천에서는 단순 트리보다 압축 인덱스, 비트맵, 확률적 자료구조를 함께 쓰는 혼합 설계가 일반화되고 있다.
- 근사 정답을 빠르게 얻는 블룸 필터, 카운트 구조 같은 확률적 자료구조가 대용량 분산 시스템에서 더 자주 채택되고 있다.
- 그래프 데이터가 커지면서 메모리 내 전부 적재보다 분산 그래프 저장과 부분 탐색 최적화가 중요한 실무 주제가 되고 있다.
- 생성형 인공지능과 검색 결합 시스템에서는 벡터 인덱스 구조가 핵심 기반으로 떠오르며, 최근접 탐색 성능이 중요한 설계 요소가 되었다.
- 압축과 연산을 함께 고려하는 자료구조가 주목받아, 저장 공간 절감과 조회 성능을 동시에 맞추려는 방향이 강화되고 있다.
- 교육과 면접에서도 단순 구현 암기보다 실제 시스템에서 왜 그 구조를 선택하는지 설명하는 능력이 더 중요해지는 흐름이다.
