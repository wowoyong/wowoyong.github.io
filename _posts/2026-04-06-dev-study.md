## 핵심 요약

컴파일러는 사람이 작성한 고급 언어 코드를 기계가 이해할 수 있는 형태로 바꾸는 번역기이자, 성능과 안정성을 좌우하는 핵심 도구다. 학부 수준에서는 어휘 분석, 구문 분석, 의미 분석, 중간 표현, 최적화, 코드 생성으로 이어지는 전체 흐름을 하나의 맥락으로 이해하는 것이 중요하다. 실제 개발에서는 오류 메시지의 품질, 빌드 시간, 실행 성능, 이식성이 모두 컴파일러 설계와 직접 맞닿아 있다. 최근에는 다단계 중간 표현, 인공지능 가속기용 컴파일, 웹어셈블리, 관찰 가능성과 보안 검증 강화가 중요한 흐름으로 자리 잡고 있다. 따라서 컴파일러는 단순한 번역기를 넘어, 언어 설계와 하드웨어 활용을 잇는 실무형 기반 기술로 봐야 한다.

## 번역기 역할과 인터프리터와의 차이

- 컴파일러는 보통 프로그램 전체를 먼저 번역해 실행 가능한 형태를 만들고, 인터프리터는 실행 시점에 문장을 하나씩 해석하며 처리한다는 차이가 있다.
- 같은 계산기 프로그램이라도 컴파일 방식은 처음 빌드하는 데 시간이 들지만 반복 실행이 빠르고, 인터프리터 방식은 바로 시험해 보기 쉽다는 점에서 사용성이 다르다.
- 컴파일 결과는 기계어일 수도 있고 바이트코드나 중간 코드일 수도 있어서, 항상 완전한 실행 파일만 만들어 내는 것은 아니다.
- 정적 검사에 강한 언어는 컴파일 단계에서 자료형 오류를 많이 잡아 주는 반면, 동적 언어는 실행 중에 오류가 더 늦게 드러나는 경우가 많다.
- 실무에서는 성능이 중요한 서버나 임베디드 장치에서 컴파일 방식의 장점이 크고, 교육용 도구나 스크립트 자동화에서는 즉시 실행성이 더 중요할 수 있다.
- 컴파일러는 단순한 변환기를 넘어 경고 제공, 최적화, 디버그 정보 생성까지 맡기 때문에 개발 경험 전반에 영향을 준다.
- 결국 두 방식의 차이는 번역 시점만이 아니라, 오류를 발견하는 시점, 배포 방식, 성능 특성까지 아우르는 개발 방식의 차이로 이해해야 한다.

## 어휘 분석과 구문 분석이 코드를 구조로 바꾸는 방식

- 어휘 분석은 문자 흐름을 토큰으로 나누는 단계로, 예를 들어 변수 선언문 하나를 키워드, 식별자, 연산자, 상수, 구분자로 구분한다.
- 이 단계에서는 공백과 주석을 대체로 의미 없는 요소로 처리하지만, 들여쓰기가 문법인 언어에서는 예외가 생길 수 있다.
- 구문 분석은 토큰의 나열이 문법 규칙에 맞는지 확인하고, 트리 형태의 구조를 만들어 낸다.
- 예를 들어 덧셈과 곱셈이 함께 있는 식은 읽는 순서와 다르게 곱셈이 먼저 묶여야 하므로, 트리 구조가 연산 우선순위를 정확히 담아야 한다.
- 어휘 분석 오류는 잘못된 문자나 닫히지 않은 문자열처럼 비교적 단순한 경우가 많고, 구문 분석 오류는 괄호 누락이나 문장 구조 손상처럼 더 큰 문맥 문제를 다룬다.
- 실무적으로 보면 좋은 파서는 단순히 실패하는 데 그치지 않고, 어느 위치에서 무엇이 기대되었는지 알려 줘야 디버깅 비용을 줄일 수 있다.
- 즉, 이 두 단계는 코드를 그저 읽는 과정이 아니라, 프로그램의 형식을 계산 가능한 구조로 다시 구성하는 과정이라고 볼 수 있다.

## 의미 분석에서 자료형과 범위를 검증하는 이유

- 의미 분석은 문법상으로는 맞지만 실제 의미가 타당한지 검사하는 단계로, 선언되지 않은 변수 사용이나 자료형 불일치를 찾아낸다.
- 예를 들어 정수와 문자열을 더하는 식은 문장 형태는 맞아 보여도 언어 규칙에 따라 허용되지 않을 수 있다.
- 이 단계에서는 식별자 표를 사용해 변수, 함수, 매개변수의 이름과 범위를 관리하고, 블록 안팎에서 같은 이름이 어떻게 해석되는지 결정한다.
- 함수를 호출할 때 인자의 개수와 자료형이 선언과 맞는지도 여기서 검사하므로, 단순한 문법보다 더 깊은 규칙을 다룬다.
- 자동 형변환을 허용하는 언어는 편리하지만 의도하지 않은 변환이 생길 수 있고, 엄격한 언어는 더 안전하지만 코드가 길어질 수 있다.
- 실제 개발에서는 의미 분석의 품질이 좋을수록 실행 전에 더 많은 버그를 막을 수 있어 테스트 부담도 줄어든다.
- 따라서 의미 분석은 문법 뒤에 숨어 있는 프로그램의 논리적 일관성을 확인하는 안전장치로 이해하면 된다.

## 중간 표현이 최적화와 이식성을 가능하게 하는 구조

- 중간 표현은 원본 언어와 기계어 사이에 놓인 중간 단계의 표현으로, 여러 언어와 여러 하드웨어를 이어 주는 공통 기반 역할을 한다.
- 같은 덧셈 연산도 추상 구문 트리, 세 주소 코드, 정적 단일 대입 형태처럼 여러 방식으로 표현할 수 있으며, 목적에 따라 장단점이 다르다.
- 중간 표현을 사용하면 앞단은 언어 해석에 집중하고 뒷단은 기계별 코드 생성에 집중할 수 있어 구조를 모듈화하기 쉽다.
- 예를 들어 하나의 앞단으로 여러 중앙 처리 장치용 코드를 만들 수 있으므로, 언어 구현 비용을 크게 줄일 수 있다.
- 최적화는 보통 중간 표현 위에서 수행되는데, 기계어보다 추상적이어서 분석하기 쉽고 원본 코드보다 구조가 정돈되어 있기 때문이다.
- 여러 단계의 중간 표현을 쓰는 설계는 고수준 최적화와 저수준 최적화를 나누어 수행할 수 있어 최근 더욱 중요해지고 있다.
- 실무에서는 중간 표현의 품질이 곧 컴파일러의 확장성으로 이어지므로, 새로운 하드웨어나 특정 분야 지원의 핵심 기반이 된다.

## 최적화 기법을 성능과 빌드 시간의 균형으로 읽기

- 최적화는 무조건 빠르게 만드는 마법이 아니라, 실행 시간, 메모리 사용량, 코드 크기, 컴파일 시간 사이에서 균형을 잡는 선택이다.
- 상수 접기처럼 단순한 계산 결과를 미리 구해 두는 기법은 구조가 단순하면서도 효과가 커서 기본 최적화로 자주 쓰인다.
- 죽은 코드 제거는 결과에 영향을 주지 않는 계산을 없애는 방식인데, 디버그용으로 남겨 둔 코드가 실제 배포본에서는 사라질 수 있다는 점도 이해해야 한다.
- 공통 부분식 제거는 같은 계산을 반복하지 않게 해 성능을 높이지만, 메모리 사용이 늘거나 레지스터 부담이 커질 수 있다.
- 반복문 최적화는 효과가 크지만 잘못 적용하면 코드 크기가 커지거나 분기 예측에 불리해질 수 있다.
- 인라인 확장은 함수 호출 비용을 줄이지만, 작은 함수에는 유리해도 큰 함수까지 지나치게 펼치면 명령어 캐시에 손해가 날 수 있다.
- 실무에서는 가장 높은 최적화 단계가 항상 정답은 아니며, 개발 중에는 빌드 시간을 줄이고 배포할 때만 강한 최적화를 적용하는 경우가 많다.

## 코드 생성과 레지��터 할당이 하드웨어 성능을 좌우하는 이유

- 코드 생성 단계는 중간 표현을 실제 기계 명령으로 바꾸는 과정이며, 같은 연산이라도 어떤 명령을 선택하느냐에 따라 성능이 달라진다.
- 레지스터는 매우 빠르지만 수가 적기 때문에, 어떤 값을 레지스터에 남기고 어떤 값을 메모리로 내보낼지 정하는 레지스터 할당이 중요하다.
- 레지스터가 부족하면 스필이 발생해 메모리 접근이 늘어나고, 계산 자체보다 데이터 이동이 병목이 되는 경우가 많다.
- 명령 선택은 단순한 번역이 아니라 대상 구조의 강점을 활용하는 과정으로, 예를 들어 같은 곱셈도 특수 명령이 있으면 더 적은 단계로 처리할 수 있다.
- 명령어 스케줄링은 서로 독립적인 연산의 순서를 조정해 파이프라인 활용도를 높이는 기법으로, 중앙 처리 장치 구조를 이해해야 효과를 낼 수 있다.
- 같은 소스 코드라도 서로 다른 구조의 기계에서 성능 차이가 크게 나는 이유는 바로 이 뒷단 설계에 있다.
- 실무적으로는 뒷단의 품질이 언어 자체의 평가에 영향을 줄 만큼 중요하므로, 빠른 언어라는 평가는 종종 컴파일러 품질과 함께 읽어야 한다.

## 최신 흐름

- 다단계 중간 표현이 널리 퍼지면서, 고수준 수학 연산부터 저수준 기계 명령까지 여러 추상화 단계를 나누어 최적화하는 설계가 주류가 되고 있다.
- 인공지능 가속기용 컴파일의 중요성이 커지면서, 그래프 최적화와 연산 배치가 중앙 처리 장치 중심 시대보다 훨씬 큰 비중을 차지하고 있다.
- 웹어셈블리 생태계가 커지면서 브라우저 밖 실행 환경까지 포함한 배포 경로가 넓어졌고, 컴파일러는 웹과 서버를 함께 고려하는 방향으로 발전하고 있다.
- 실행 중 컴파일과 사전 컴파일을 섞는 혼합 전략이 늘어나면서, 시작 속도와 장기적인 성능을 함께 맞추는 방식이 점점 보편화되고 있다.
- 관찰 가능성 강화도 중요한 흐름이어서, 디버그 정보와 성능 분석 정보, 오류 위치 추적을 더 잘 남기는 컴파일러가 실무에서 선호된다.
- 보안 측면에서는 메모리 안전성 검사, 제어 흐름 보호, 정의되지 않은 동작 탐지 같은 검증 기능이 컴파일 단계에 더 깊이 통합되고 있다.
- 도메인 특화 언어와 도메인 특화 컴파일이 늘어나면서, 범용 언어 하나로 모든 문제를 해결하기보다 특정 계산 영역에 맞춘 최적화가 더 강조되고 있다.
- 커널 관찰과 네트워크 처리 같은 분야에서는 eBPF 계열 도구와 연결된 컴파일 경로가 실전 시스템 도구로 널리 쓰이고 있다.
- 벡터화와 병렬화 자동화도 계속 고도화되어, 같은 코드라도 자료 병렬 하드웨어를 얼마나 잘 활용하느냐가 큰 성능 차이를 만든다.
- 2026년 시점의 컴파일러는 단순한 언어 구현 기술을 넘어, 인공지능 시스템, 웹 실행 환경, 고성능 계산, 안전한 시스템 소프트웨어를 잇는 공통 기반으로 자리 잡고 있다.
