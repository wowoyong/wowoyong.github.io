## 핵심 요약

트리는 계층적 관계를 표현하는 비선형 자료구조로, 노드와 간선으로 구성되며 사이클이 없는 방향 그래프의 특수한 형태입니다. 이진 탐색 트리(BST)는 왼쪽 자식이 부모보다 작고 오른쪽이 크다는 성질로 O(log n) 탐색을 가능하게 하지만, 편향된 입력에서는 O(n)으로 성능이 저하됩니다. 이를 해결하기 위해 AVL 트리와 레드-블랙 트리 같은 자가 균형 트리가 등장하였으며, 실제 데이터베이스 인덱스와 언어 런타임 내부에서 널리 쓰입니다. 힙은 트리 기반의 우선순위 큐 구현에 최적화되어 있으며, 배열로 효율적으로 표현할 수 있다는 점이 큰 장점입니다. 최근에는 함수형 언어의 불변 트리, 병렬 처리를 위한 세그먼트 트리, 기계 학습의 결정 트리 앙상블까지 트리의 응용 범위가 빠르게 넓어지고 있습니다. 트리를 깊이 이해하면 파일 시스템, 검색 엔진, 컴파일러의 동작 원리까지 폭넓게 파악할 수 있습니다.

## 트리의 구성 요소와 핵심 용어 정의

- 루트: 부모가 없는 최상위 노드. 트리에는 루트가 정확히 1개이며, 그래프에서 연결 여부를 확인할 때 루트로부터 도달 가능한지를 기준으로 삼음
- 간선: 노드 사이를 잇는 연결. n개의 노드가 있으면 간선은 항상 n-1개 — 사이클이 없음을 증명하는 핵심 성질
- 깊이: 루트에서 특정 노드까지의 간선 수. 루트의 깊이는 0이며, 성능 분석 시 최대 깊이(높이)가 시간 복잡도를 결정
- 높이: 트리 전체의 최대 깊이. 완전 이진 트리의 높이는 ⌊log₂n⌋로 균형 유지의 기준값이 됨
- 차수: 노드가 가진 자식 수. 이진 트리는 차수가 최대 2이며, B-트리는 차수가 m 이상인 다진 트리
- 잎 노드: 자식이 없는 노드. 파일 시스템에서 실제 파일, 수식 트리에서 피연산자에 해당
- 형제 노드: 같은 부모를 가진 노드. 트리 순회 시 형제 방문 순서가 결과 순서를 결정
- 부분 트리: 임의의 노드와 그 모든 자손으로 구성되는 트리. 재귀 알고리즘의 분할 정복 기반이 됨

## 이진 트리의 종류 비교: 완전·포화·균형·편향

- 포화 이진 트리: 모든 내부 노드가 자식 2개, 모든 잎이 같은 깊이. 높이 h이면 노드 수 2^(h+1)-1 — 이론적으로 가장 이상적인 형태
- 완전 이진 트리: 마지막 레벨을 제외하면 포화, 마지막 레벨은 왼쪽부터 채움. 힙을 배열로 표현할 때 이 성질이 인덱스 계산을 단순하게 만듦
- 균형 이진 트리: 임의 노드에서 왼쪽·오른쪽 부분 트리 높이 차가 1 이하. 탐색·삽입·삭제 모두 O(log n) 보장
- 편향 트리: 모든 노드가 한 방향으로만 자식을 가짐. 연결 리스트와 동일한 구조로 BST에 정렬된 값을 순서대로 삽입하면 발생, 탐색이 O(n)으로 저하
- 완전 이진 트리 vs 포화 이진 트리: 포화는 모든 자리가 채워지지만 완전은 마지막 줄을 왼쪽부터 채우는 유연한 형태 — 힙 구현에는 완전 이진 트리 사용
- 이진 트리의 배열 표현: i번 인덱스 노드의 왼쪽 자식은 2i, 오른쪽은 2i+1, 부모는 ⌊i/2⌋. 완전 이진 트리일 때만 공간 낭비 없이 적용 가능
- 쓰레드 이진 트리: 순회 시 스택·재귀 없이 다음 노드를 O(1)에 찾기 위해 빈 포인터를 순서 노드로 연결한 형태 — 임베디드 환경에서 스택 절약
- 이진 트리 노드 수 성질: 잎 노드 수 = 차수 2인 노드 수 + 1. 이 관계는 트리 구조 검증에 활용됨

## 트리 순회 방식과 실제 활용 맥락

- 전위 순회(뿌리→왼→오른): 부모를 먼저 방문. 트리 복사, 디렉터리 구조 출력 시 유용 — 부모를 먼저 알아야 자식을 배치할 수 있기 때문
- 중위 순회(왼→뿌리→오른): BST에서 중위 순회 결과는 오름차순 정렬된 수열. 이 성질을 이용해 BST의 정렬 여부를 O(n)에 검증 가능
- 후위 순회(왼→오른→뿌리): 자식을 먼저 처리해야 하는 경우 사용. 디렉터리 용량 계산, 컴파일러 수식 평가 시 활용
- 레벨 순서 순회(너비 우선): 큐를 사용해 같은 깊이의 노드를 먼저 방문. 최단 경로 탐색, 트리의 너비 계산에 활용
- 수식 트리: 연산자를 내부 노드, 피연산자를 잎 노드로 구성. 후위 순회 결과가 역폴란드 표기법(RPN)이 되어 스택 기반 계산기 구현에 직접 적용
- 전위 순회 vs 후위 순회 선택 기준: 부모 정보가 먼저 필요하면 전위, 자식 결과가 합산되어야 부모 처리가 가능하면 후위
- 모리스 순회(Morris traversal): 추가 공간 O(1)로 중위 순회 구현. 임베디드·메모리 제한 환경에서 유용하며 쓰레드 이진 트리 개념을 일시적으로 사용
- 반복 순회 vs 재귀 순회: 재귀는 코드가 간결하지만 깊은 트리에서 스택 오버플로 위험. 실무에서는 명시적 스택으로 반복 구현이 안전

## 이진 탐색 트리(BST)의 동작 원리와 한계

- BST 성질: 왼쪽 부분 트리의 모든 값 < 루트 < 오른쪽 부분 트리의 모든 값. 이 성질이 재귀적으로 모든 노드에 적용됨
- 탐색 시간 복잡도: 균형 상태에서 O(log n), 편향 상태에서 O(n). 랜덤 입력 기대값은 O(log n)이지만 최악을 보장하지 않음
- 삽입: 탐색과 동일한 경로로 내려가다 빈 자리에 삽입. 항상 잎 노드로 추가되므로 기존 구조를 변경하지 않음
- 삭제의 세 가지 경우: (1) 잎 노드 — 그냥 제거, (2) 자식 1개 — 부모와 자식을 직접 연결, (3) 자식 2개 — 중위 후계자(오른쪽 부분 트리의 최솟값) 또는 중위 선행자로 대체
- BST의 한계: 1, 2, 3, 4, 5 순서로 삽입하면 오른쪽으로 편향된 연결 리스트와 동일한 구조 생성. 실제로 정렬된 로그 파일을 BST에 순서대로 삽입하면 이 문제가 발생
- BST vs 이진 탐색 배열: BST는 삽입/삭제가 O(log n)이지만 배열 이진 탐색은 탐색만 O(log n), 삽입/삭제는 O(n). 동적 데이터에는 BST가 유리
- BST 검증 방법: 중위 순회로 오름차순 수열이 나오는지 확인하거나, 각 노드가 허용 범위(최솟값, 최댓값) 내에 있는지 재귀로 확인
- 중복 처리 전략: 중복 허용 시 왼쪽 또는 오른쪽 어느 방향으로 일관되게 처리할지 규칙을 정해야 함 — 구현마다 다르므로 주의

## 자가 균형 트리: AVL 트리와 레드-블랙 트리 비교

- AVL 트리: 모든 노드에서 왼쪽·오른쪽 높이 차(균형 인수)를 -1, 0, +1로 유지. 삽입/삭제 후 균형 인수 위반 노드를 찾아 회전으로 복구
- 회전 종류: 단순 좌회전, 단순 우회전, 좌-우 이중 회전, 우-좌 이중 회전 — 총 4가지 패턴. 어떤 경우에 어떤 회전을 적용하는지가 핵심 학습 포인트
- 레드-블랙 트리: 각 노드에 빨��색/검은색 색상을 부여하고 5가지 색상 규칙을 유지. AVL보다 느슨한 균형이지만 회전 횟수가 적어 삽입/삭제가 빠름
- 레드-블랙 5가지 규칙 핵심: (1) 루트는 검은색, (2) 빨간 노드의 자식은 검은색, (3) 임의 노드에서 잎까지 경로의 검은 노드 수가 동일. 이로 인해 최장 경로가 최단 경로의 2배 이하
- AVL vs 레드-블랙 선택 기준: 읽기가 많으면 AVL(더 엄격한 균형 → 탐색 빠름), 쓰기가 많으면 레드-블랙(회전 적음 → 삽입/삭제 빠름). 리눅스 커널 CFS 스케줄러는 레드-블랙 트리 사용
- 자바 TreeMap, C++ std::map은 레드-블랙 트리 구현. 반면 AVL 트리는 데이터베이스 인덱스 등 읽기 편향 작업에 쓰임
- 균형 유지 비용: 삽입/삭제 후 회전은 O(1) ~ O(log n) 회전. 균형 유지 덕분에 전체 연산이 O(log n) 최악을 보장
- 스플레이 트리: 최근 접근 노드를 루트 근처로 이동시키는 자가 조정 트리. 접근 지역성이 높은 패턴에서 실질적으로 빠른 성능 제공

## 힙과 우선순위 큐 구현 원리

- 최대 힙: 부모 ≥ 자식인 완전 이진 트리. 루트가 항상 최댓값이므로 최댓값 조회가 O(1) — 작업 스케줄러에서 우선순위 높은 작업 즉시 조회에 활용
- 최소 힙: 부모 ≤ 자식. 다익스트라 알고리즘, 허프만 코딩 등 최솟값을 반복 추출하는 문제에 사용
- 배열 표현: 1-인덱스 기준으로 부모 i, 왼쪽 자식 2i, 오른쪽 자식 2i+1. 포인터 없이 배열만으로 완전 이진 트리를 표현하여 캐시 효율이 높음
- 상향 삽입: 배열 끝에 추가 후 부모와 비교하며 올라감. O(log n) — 완전 이진 트리의 높이만큼 이동
- 최댓값 추출: 루트 제거 후 마지막 노드를 루트에 배치, 두 자식 중 큰 값과 교환하며 내려감. O(log n)
- 힙 구성: n개의 원소를 하나씩 삽입하면 O(n log n)이지만, 모든 내부 노드에 대해 하향 정렬을 적용하면 O(n)에 힙 구성 가능 — 이 최적화가 힙 정렬 효율의 핵심
- 힙 정렬: 최대 힙 구성 후 루트(최댓값)를 배열 끝과 교환, 힙 크기를 줄이고 하향 정렬 반복. O(n log n), 제자리 정렬, 불안정 정렬
- 이진 힙 vs 다진 힙: 다진 힙은 자식이 여러 개. 삽입은 빠르지만 추출이 느려짐. 삽입이 잦은 다익스트라 구현에서 4진 힙이 이진 힙보다 실측 성능이 좋은 경우 있음

## 트리의 실무 적용: 파일 시스템·인덱스·파서

- 파일 시스템: 디렉터리 구조가 트리 형태. 리눅스 ext4의 디렉터리 내 파일 탐색은 해시 트리(HTree)로 O(log n) — 단순 선형 탐색 대비 대형 디렉터리에서 압도적 속도
- B-트리와 B+ 트리: 데이터베이스 인덱스의 표준. 디스크 블록 크기에 맞춰 노드를 크게 키워 입출력 횟수를 줄임. B+ 트리는 잎 노드를 연결 리스트로 연결해 범위 탐색 최적화
- 추상 구문 트리(AST): 컴파일러가 소스 코드를 파싱한 결과. 변수 선언, 함수 호출, 연산자 등이 노드. 코드 분석 도구, 변환 도구 모두 AST를 조작하여 동작
- 트라이: 문자열을 접두사 단위로 분해하여 저장하는 트리. 자동 완성, 사전 검색에서 O(m) 탐색(m: 문자열 길이) — 해시 맵 대비 접두사 탐색에 유리
- 세그먼트 트리: 배열 구간의 합·최댓값·최솟값을 O(log n)에 쿼리하고 갱신. 경쟁 프로그래밍과 실시간 집계 시스템에서 핵심 자료구조
- 결정 트리: 기계 학습에서 분류·회귀에 사용. 노드마다 조건 분기, 잎 노드가 최종 예측값. 랜덤 포레스트는 다수의 결정 트리를 앙상블
- 도메인 이름 시스템(DNS): 계층적 도메인 구조가 트리. 루트 → 최상위 도메인(.com) → 2단계 도메인 순으로 위임 탐색 — 트리 순회가 네트워크에 투영된 사례

## 최신 흐름

- 영속 트리: 함수형 언어(하스켈, 클로저)에서 이전 버전을 보존하면서 수정 가능한 트리. 경로 복사 방식으로 구현하며 불변 데이터 구조의 핵심
- 병행 레드-블랙 트리: 멀티코어 환경에서 잠금 없는 방식으로 트리를 수정하는 연구가 활발. 자바의 ConcurrentSkipListMap은 트리 대신 스킵 리스트로 유사 성능을 비교 기술로 제공
- 가중치 균형 트리: 서브트리 크기 비율로 균형을 정의. 함수형 집합·맵 라이브러리(하스켈 Data.Set 등)에서 AVL·레드-블랙보다 구현이 단순하여 선호
- 학습 기반 인덱스: 신경망이 B-트리 인덱스를 대체할 수 있는지 연구. 구글의 2018년 논문 이후 데이터베이스 커뮤니티에서 활발히 논의 중
- 압축 트라이: PATRICIA 트리, 기수 트리 등. 리눅스 커널의 경로 이름 조회와 라우팅 테이블(LPM)에 기수 트리를 사용
- 증분 파싱: 편집기(Neovim, 비주얼 스튜디오 코드)가 코드 수정 시 AST 전체를 재구성하지 않고 변경된 부분만 갱신. Tree-sitter 라이브러리가 이를 구현
- 양자 컴퓨팅 트리 탐색: 그로버 알고리즘 응용으로 트리 탐색을 √n 수준으로 가속하는 이론 연구. 아직 실용 단계는 아니지만 알고리즘 이론에서 주목받는 방향
- 벡터 데이터베이스의 KD 트리·볼 트리: 고차원 벡터 유사도 검색(근사 최근접 이웃, ANN)에 트리 계열 인덱스 사용. Faiss, Annoy 같은 라이브러리가 이를 구현
- 트리 기반 병렬 집계: 아파치 스파크의 맵-리듀스 단계가 트리 형태의 집계로 진화. 트리 집계 방식으로 드라이버 병목을 줄임
- 결정 트리 해석 가능성(XAI): 딥러닝 모델의 블랙박스 문제 대안으로 결정 트리 기반 모델이 재조명. SHAP·LIME 설명 방법도 내부적으로 트리 구조를 활용
