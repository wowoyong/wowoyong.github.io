## 핵심 요약

디자인 패턴은 소프트웨어 설계에서 반복적으로 등장하는 문제에 대한 검증된 재사용 가능 해결책으로, 1994년 GoF(Gang of Four)가 23가지 패턴을 체계화하면서 대중화되었다. 패턴은 생성, 구조, 행동 세 범주로 나뉘며 각각 객체 생성 방식, 클래스 조합 구조, 객체 간 협력과 책임 분배 문제를 해결한다. 실무에서는 스프링 프레임워크의 싱글톤·팩토리·프록시 패턴처럼 프레임워크 내부에 패턴이 깊이 내재되어 있어 이를 이해하면 코드 해석력이 크게 높아진다. 패턴은 SOLID 원칙과 상호 보완적으로 작동하며, 올바르게 적용하면 자연스럽게 좋은 객체지향 설계 원칙을 따르게 된다. 최근에는 마이크로서비스 아키텍처와 함수형 프로그래밍의 부상으로 서킷 브레이커, 사가, 이벤트 소싱 같은 분산 시스템 패턴이 GoF 패턴만큼 중요한 설계 언어로 자리잡고 있다.

## 디자인 패턴의 정의와 역사적 탄생 배경

- 디자인 패턴이란 소프트웨어 설계의 반복 문제에 대한 검증된 해결책이다 — 코드 자체가 아니라 '설계 템플릿'으로 이해해야 한다.
- 1994년 GoF(에리히 감마, 리처드 헬름, 랄프 존슨, 존 블리시디스)가 '디자인 패턴' 책에서 23개 패턴을 체계화했다.
- 개념의 뿌리는 건축가 크리스토퍼 알렉산더의 '패턴 언어'다 — 건축 설계의 반복 솔루션 아이디어를 소프트웨어에 차용한 것.
- 패턴은 문제(Problem), 상황(Context), 해결책(Solution), 결과(Consequences) 네 가지 요소로 기술된다.
- 패턴을 공유하면 팀 내 공통 언어가 생긴다 — '옵서버 패턴 쓰자'는 한 문장이 수십 줄의 설명을 대체한다.
- 패턴은 특정 언어에 종속되지 않지만, 자바·코틀린·파이썬 등 객체지향 언어에서 가장 자연스럽게 표현된다.
- 모든 상황에 패턴을 억지로 적용하는 것은 '패턴 남용'이다 — 단순한 상황은 단순하게 두는 것이 더 좋은 설계다.

## 생성 패턴 - 객체를 어떻게, 얼마나 만들 것인가

- 싱글톤(Singleton): 클래스 인스턴스를 전역에서 오직 하나만 보장한다 — 데이터베이스 연결 풀, 로그 관리자에 흔히 쓰인다.
- 팩토리 메서드(Factory Method): 객체 생성을 서브클래스에 위임하여 어떤 클래스를 만들지 결정권을 분리한다.
- 추상 팩토리(Abstract Factory): 관련 객체 패밀리를 함께 생성하는 인터페이스를 제공한다 — 운영체제별 UI 컴포넌트 묶음 생성에 적합.
- 빌더(Builder): 복잡한 객체를 단계별로 조립한다 — `Person.builder().name('홍길동').age(30).build()` 형태가 대표적.
- 프로토타입(Prototype): 기존 객체를 복제하여 새 객체를 생성한다 — 깊은 복사가 필요한 게임 캐릭터 생성 등에 활용된다.
- 생성 패턴의 핵심 가치는 '무엇을 생성하는지'와 '어떻게 생성하는지'를 분리해 코드 유연성을 높이는 것이다.
- 자바스크립트는 프로토타입 기반 언어라 GoF 생성 패턴이 다른 언어와 다르게 표현될 수 있다 — 클래스 문법은 문법적 설탕에 불과하다.

## 구조 패턴 - 클래스와 객체를 조합해 기능 확장하기

- 어댑터(Adapter): 호환되지 않는 인터페이스를 연결한다 — 구형 결제 API를 새 시스템에 연결할 때 중간 어댑터 클래스를 삽입하는 것.
- 데코레이터(Decorator): 기존 객체에 새 기능을 동적으로 추가한다 — 자바 I/O 스트림의 BufferedReader→InputStreamReader→FileInputStream 구조가 대표 예.
- 프록시(Proxy): 실제 객체에 대한 접근을 제어하는 대리 객체 — 스프링 AOP의 트랜잭션 처리가 프록시 패턴으로 구현된다.
- 퍼사드(Facade): 복잡한 서브시스템에 단순화된 인터페이스를 제공한다 — 홈시어터의 복잡한 설정을 `homeTheater.watchMovie()` 하나로 감싸는 것.
- 컴포지트(Composite): 트리 구조로 개별 객체와 복합 객체를 동일하게 다룬다 — 파일 시스템에서 파일과 폴더를 같은 인터페이스로 처리하는 것.
- 브리지(Bridge): 추상화와 구현을 분리하여 독립적으로 변경 가능하게 한다 — 리모컨(추상)과 TV 기종(구현)을 분리해 각각 확장하는 예.
- 플라이웨이트(Flyweight): 유사한 객체를 공유하여 메모리를 절약한다 — 텍스트 편집기에서 같은 글자 객체를 재사용하는 것이 전형적 예.
- 구조 패턴의 공통 목표는 상속 없이 인터페이스 유연성과 기능 확장성을 달성하는 것이다.

## 행동 패턴 - 객체 간 협력과 책임을 어떻게 나눌 것인가

- 옵서버(Observer): 한 객체의 상태 변화를 구독한 모든 객체에 자동 통지 — 이벤트 리스너, 리액트의 상태 관리가 이 원리를 따른다.
- 전략(Strategy): 알고리즘을 캡슐화하고 교환 가능하게 만든다 — 결제 방식(신용카드/카카오페이/페이팔)을 실행 시점에 교체하는 것이 대표 예.
- 커맨드(Command): 요청을 객체로 캡슐화하여 취소, 재실행, 큐잉을 지원한다 — 텍스트 편집기의 실행 취소 기능이 이 패턴으로 구현된다.
- 이터레이터(Iterator): 컬렉션 내부 구조를 노출하지 않고 순차 접근을 제공한다 — 자바의 for-each 루프가 내부적으로 이터레이터를 사용한다.
- 템플릿 메서드(Template Method): 알고리즘 뼈대를 상위 클래스에 정의하고 세부 단계를 서브클래스에서 구현한다 — 시험 절차(준비→풀기→제출)는 고정, 내용만 다름.
- 상태(State): 객체 내부 상태에 따라 행동이 달라지도록 한다 — 신호등이 빨강·노랑·초록 상태에 따라 다르게 동작하는 것.
- 책임 연쇄(Chain of Responsibility): 요청을 처리할 수 있는 객체를 체인으로 연결하여 차례로 시도한다 — HTTP 미들웨어 파이프라인이 이 구조다.
- 중재자(Mediator): 객체 간 복잡한 통신을 중앙화하여 결합도를 낮춘다 — 채팅방 서버가 모든 사용자 메시지를 중재하는 방식.

## SOLID 원칙과 디자인 패턴의 상호 연결 구조

- SOLID는 다섯 가지 객체지향 설계 원칙이다: 단일책임(SRP), 개방폐쇄(OCP), 리스코프치환(LSP), 인터페이스분리(ISP), 의존역전(DIP).
- 전략 패턴은 OCP(개방폐쇄 원칙)를 구현한 대표 사례 — 기존 코드 수정 없이 새 전략 클래스를 추가하여 기능을 확장한다.
- 의존성 주입(DI)은 DIP(의존역전 원칙)의 실현이며, 스프링 프레임워크 전체 구조의 핵심 메커니즘이다.
- 리스코프치환 원칙을 위반하면 상속 기반 패턴(템플릿 메서드)이 예상치 못한 버그를 유발한다 — 서브타입은 부모 타입을 완전히 대체할 수 있어야 한다.
- 인터페이스분리 원칙은 퍼사드·어댑터 패��과 결합하여 레거시 시스템 리팩토링 시 자주 활용된다.
- 패턴을 SOLID와 연계하면 '왜 이 패턴을 써야 하는가'를 설계 원칙으로 정당화할 수 있어 팀 내 설득이 쉬워진다.
- 반대로 SOLID만 따지면 구체적 구현이 불명확해지는데, 패턴이 그 구체적인 청사진을 제시한다.

## 실무 적용 전략과 피해야 할 안티패턴

- 안티패턴이란 겉으로 합리적으로 보이지만 실제로 문제를 더 악화시키는 반복적 실수 패턴이다.
- 갓 클래스(God Class): 하나의 클래스가 너무 많은 책임을 지는 안티패턴 — 단일책임 원칙(SRP) 위반의 전형으로, 1000줄을 넘는 서비스 클래스가 대표적.
- 스파게티 코드: 논리 흐름이 얽혀 추적이 불가능한 코드 — 패턴 없이 조건문과 콜백이 무한 중첩된 경우.
- 조기 최적화: 아직 병목이 확인되지 않았는데 복잡한 캐싱·패턴을 미리 도입하는 실수 — '추측하지 말고 측정하라'가 원칙.
- 패턴 과적용: 단순 CRUD에 팩토리·빌더·전략을 모두 적용하면 오히려 코드 이해가 어려워지는 역효과가 난다.
- 실무 코드 리뷰 시 '이 패턴이 없으면 어떤 문제가 생기는가?'를 먼저 묻는 것이 과도한 설계를 막는 좋은 습관이다.
- 스프링, 장고, NestJS 같은 프레임워크가 이미 많은 패턴을 내장하므로, 직접 구현 전에 프레임워크 기능을 먼저 확인해야 한다.
- 마틴 파울러의 '리팩토링' 책에 나오는 '코드 냄새(Code Smell)' 목록과 GoF 패턴을 함께 공부하면 언제 패턴을 도입할지 감이 잡힌다.

## 최신 흐름

- 마이크로서비스 아키텍처 확산으로 서킷 브레이커, 사가(Saga), API 게이트웨이 같은 분산 시스템 패턴이 GoF 패턴만큼 필수 지식이 되었다.
- 함수형 프로그래밍의 부상으로 불변성, 순수 함수, 모나드 같은 함수형 패턴이 객체지향 패턴을 보완하거나 대체하는 흐름이 생겼다.
- 이벤트 소싱과 CQRS(명령조회 책임 분리) 패턴이 대규모 트래픽 서비스에서 주류 아키텍처 패턴으로 자리잡았다.
- 쿠버네티스 생태계와 함께 사이드카(Sidecar), 앰배서더(Ambassador) 같은 컨테이너 기반 인프라 패턴이 성장하고 있다.
- 인공지능 애플리케이션 개발에서 에이전트 루프, 도구 활용, 검색 증강 생성(RAG) 같은 거대 언어 모델 특화 아키텍처 패턴이 새롭게 정의되고 있다.
- 저수준 GoF 패턴보다 고수준 아키텍처 패턴(헥사고날, 클린 아키텍처, 도메인 주도 설계)으로 설계 논의의 무게 중심이 이동하고 있다.
- 코틀린, 스칼라, 러스트처럼 다중 패러다임 언어가 인기를 얻으면서 전통적 GoF 패턴의 언어 중립적 해석이 재정의되고 있다.
- 리액티브 프로그래밍(RxJava, 프로젝트 리액터)의 확산으로 옵서버 패턴이 스트림 기반 비동기 처리 패턴으로 진화했다.
- 깃옵스(GitOps)와 코드형 인프라(IaC) 패턴이 소프트웨어 공학 영역으로 편입되면서 배포 파이프라인 설계에도 패턴 개념이 적용된다.
- 인공지능 코드 생성 도구가 보편화되면서 패턴을 직접 구현하기보다 '어떤 패턴을 요청할 것인가'를 이해하는 역량이 중요해졌다.
