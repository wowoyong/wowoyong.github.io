## 핵심 요약

컴퓨터 보안은 정보를 안전하게 보호하기 위해 기밀성, 무결성, 가용성을 함께 관리하는 분야다. 학부 과정에서는 인증과 인가, 암호 기술, 취약점 분석, 운영체제와 네트워크 방어, 안전한 개발 절차를 연결해서 이해하는 것이 중요하다. 실제 적용에서는 비밀번호만 강하게 만드는 것보다 권한을 최소화하고 입력을 검증하며 로그를 남기고 빠르게 패치하는 체계가 더 큰 효과를 낸다. 예를 들어 웹 서비스는 해시로 비밀번호를 보호하고, 서버는 접근 권한을 분리하며, 배포 과정에서는 의존성 취약점을 함께 점검해야 한다. 2026년 기준으로는 패스키 확산, 양자내성암호 전환 준비, 메모리 안전 언어 채택, 공급망 투명성 강화처럼 보안을 처음부터 설계에 넣는 흐름이 두드러진다.

## 기밀성·무결성·가용성으로 보안 목표 세우기

- 기밀성은 허가되지 않은 사람이 정보를 보지 못하게 하는 성질이며, 학생 성적 파일에 접근 권한을 제한하는 사례로 이해하면 쉽다.
- 무결성은 정보가 허가 없이 바뀌지 않는 성질이며, 송금 금액이 전송 중 10000원에서 90000원으로 바뀌지 않도록 막는 것이 대표 예시다.
- 가용성은 필요할 때 시스템을 사용할 수 있는 성질이며, 분산 서비스 거부 공격은 기밀성보다 가용성을 먼저 무너뜨리는 공격이라는 점에서 구분된다.
- 세 목표는 서로 충돌할 수 있는데, 지나치게 엄격한 인증은 안전하지만 사용성을 낮춰 실제 운영에서는 우회 행동을 부를 수 있다.
- 보안 목표는 데이터마다 다르게 잡아야 하며, 공개 공지사항은 기밀성보다 무결성이 더 중요하고 의료 정보는 세 목표가 모두 중요하다.
- 실무에서는 세 목표를 수치화해 우선순위를 정하는데, 결제 시스템은 몇 분의 장애도 큰 손실이므로 가용성 요구 수준이 특히 높다.
- 시험 문제 유출 사례를 보면 기밀성 실패이고, 성적 조작은 무결성 실패이며, 수강신청 서버 마비는 가용성 실패라서 같은 보안 사고도 분류가 다르다.

## 인증·인가·책임추적으로 접근 통제 이해하기

- 인증은 누구인지 확인하는 과정이고 인가는 무엇을 할 수 있는지 정하는 과정이라서, 로그인 성공 뒤에도 관리자 메뉴가 막혀 있을 수 있다.
- 비밀번호는 가장 흔한 인증 수단이지만 재사용과 피싱에 약하므로, 일회용 코드나 생체 정보와 결합한 다중 요소 인증이 더 안전하다.
- 권한은 최소 권한 원칙으로 부여해야 하며, 조교 계정이 성적 열람만 가능하고 수정은 못 하게 하는 구성이 전형적인 예다.
- 역할 기반 접근 통제는 사용자마다 권한을 일일이 주는 방식보다 관리가 쉬워서, 학생·교수·관리자처럼 역할이 명확한 환경에 잘 맞는다.
- 책임추적은 누가 언제 무엇을 했는지 남기는 것으로, 로그가 없으면 사고 후 원인 파악이 어려워 실무에서는 보안만큼 감사에도 중요하다.
- 세션 탈취는 인증 이후를 노리는 공격이라서, 로그인만 안전하다고 끝이 아니며 세션 만료와 재인증 정책이 함께 필요하다.
- 공유 계정을 쓰면 편해 보여도 책임추적이 깨지므로, 실무에서는 계정 공유 대신 개인 계정과 권한 위임 기능을 쓰는 편이 낫다.

## 대칭키·공개키·해시를 비교하며 암호 기술 익히기

- 대칭키 암호는 암호화와 복호화에 같은 열쇠를 써서 빠르지만, 열쇠를 안전하게 나누는 문제가 항상 따라온다.
- 공개키 암호는 공개키와 비밀키를 나눠 써서 키 배포가 쉽지만 계산량이 커서, 실제 통신에서는 대칭키와 함께 섞어 쓰는 경우가 많다.
- 해시는 원문을 고정 길이 값으로 바꾸는 함수이며, 비밀번호 저장에 평문 대신 해시를 쓰는 것이 기본이지만 단순 해시만으로는 부족하다.
- 전자서명은 공개키 구조를 이용해 보낸 사람과 변조 여부를 확인하는 방법이라서, 단순 암호화와 달리 무결성과 부인 방지에 초점이 있다.
- 솔트는 같은 비밀번호가 같은 해시로 저장되는 문제를 줄여 주며, 여러 사용자가 1234를 써도 저장 값이 달라지게 만든다.
- 암호화는 내용을 숨기기 위한 것이고 해시는 원래대로 되돌리기 어렵게 만드는 것이므로, 주민등록번호 저장에는 두 방식을 목적에 맞게 구분해야 한다.
- 실무에서는 직접 암호 알고리즘을 구현하기보다 검증된 라이브러리를 쓰는 것이 중요하며, 직접 구현은 작은 실수로도 큰 취약점이 생긴다.

## 입력 검증 실패에서 시작되는 대표 공격 파악하기

- 버퍼 오버플로는 프로그램이 허용한 메모리 범위를 넘겨 쓰는 문제로, 낮은 수준 언어에서 특히 자주 등장해 시스템 권한 탈취로 이어질 수 있다.
- 에스큐엘 주입은 입력값을 질의문 일부로 해석하게 만드는 공격이라서, 문자열 연결보다 준비된 문장을 쓰는 방식이 훨씬 안전하다.
- 크로스 사이트 스크립팅은 악성 스크립트를 사용자 브라우저에서 실행시키는 공격이며, 게시판 댓글처럼 입력이 화면에 다시 출력되는 곳에서 자주 발생한다.
- 명령어 주입은 운영체제 명령을 조합하는 코드에서 생기기 쉬우며, 파일 이름 하나만 잘못 받아도 서버 명령 실행으로 번질 수 있다.
- 경쟁 상태 취약점은 실행 순서 차이를 노리는 문제라서, 같은 파일을 동시에 수정하는 상황에서 권한 우회가 생길 수 있다.
- 정수 오버플로는 값의 범위를 넘겨 계산 결과가 뒤집히는 현상으로, 단순 계산 오류처럼 보여도 메모리 할당 크기 착오로 이어질 수 있다.
- 실무적 해석으로 보면 많은 공격은 복잡한 수학보다 입력 검증 실패에서 시작하므로, 개발 초기에 경계 조건을 테스트하는 습관이 큰 차이를 만든다.

## 운영체제·네트워크 계층에서 방어면 넓히기

- 운영체제 보안은 프로세스 격리와 권한 분리를 핵심으로 하며, 한 프로그램이 뚫려도 다른 프로그램 전체로 피해가 번지지 않게 만든다.
- 파일 권한은 읽기·쓰기·실행을 나눠 관리하므로, 웹 서버가 모든 시스템 파일을 읽지 못하게 제한하는 구성이 기본 방어가 된다.
- 방화벽은 허용된 통신만 통과시키는 장치로, 사무실 문지기처럼 동작하지만 내부 공격까지 모두 막아 주지는 못한다.
- 전송 구간 암호화는 도청을 줄여 주며, 공개 와이파이에서 로그인할 때 암호화된 연결과 아닌 연결의 차이는 매우 크다.
- 침입 탐지와 침입 방지는 각각 경보 중심과 차단 중심이라는 차이가 있어, 운영 환경에서는 오탐 비용을 고려해 함께 조정한다.
- 서비스 거부 공격 방어는 단일 서버 성능만 높여서는 부족하고, 부하 분산과 캐시, 트래픽 필터링을 함께 써야 효과가 난다.
- 가상화와 컨테이너는 격리 단위를 제공하지만 만능은 아니어서, 공유 커널 구조를 쓰는 환경에서는 권한 설정과 이미지 관리가 더 중요하다.

## 안전한 개발 생명주기와 사고 대응 연결하기

- 안전한 개발 생명주기는 요구사항 단계부터 보안을 넣는 방식이며, 배포 직전에만 점검하는 것보다 수정 비용이 훨씬 낮다.
- 위협 모델링은 무엇이 중요한 자산이고 누가 어떤 경로로 공격할지를 미리 그려 보는 작업이라서, 설계 단계의 빠진 통제를 찾는 데 유용하다.
- 정적 분석은 코드를 실행하지 않고 문제를 찾고 동적 분석은 실행 중 이상을 찾으므로, 둘은 경쟁 관계보다 상호 보완 관계에 가깝다.
- 패치 관리는 취약점이 공개된 뒤 빠르게 대응하는 절차로, 같은 취약점도 시험 환경 없이 급히 반영하면 장애라는 다른 위험을 만들 수 있다.
- 백업은 랜섬웨어 대응의 마지막 안전장치인데, 원본과 분리된 백업이 없으면 감염 후 복구 비용이 급격히 커진다.
- 사고 대응은 탐지, 분석, 격리, 복구, 재발 방지로 이어지며, 로그와 시간 기록이 정확할수록 원인 분석이 쉬워진다.
- 실무에서는 보안팀만 잘해서 해결되지 않고 개발팀, 운영팀, 관리자까지 절차를 공유해야 하므로 훈련과 모의 대응이 중요하다.

## 최신 흐름

- 패스키 확산이 빨라지며 비밀번호 의존도를 줄이는 흐름이 강해졌고, 2025년에는 대형 서비스 다수가 기본 로그인 수단으로 채택 범위를 넓혔다.
- 양자내성암호 전환 준비가 본격화되었으며, 미국 표준 기관이 2024년에 첫 표준을 확정한 뒤 2025년에도 추가 표준화 작업을 이어 가고 있다.
- 메모리 안전 언어 채택이 보안 전략의 핵심으로 올라왔고, 2025년 미국 보안 기관들은 버퍼 오버플로 같은 취약점을 줄이는 실질적 방법으로 이를 강조했다.
- 보안을 나중에 덧붙이는 대신 처음부터 안전하게 설계하는 접근이 더 강해졌으며, 제조사 책임을 높이는 정책과 지침이 계속 늘고 있다.
- 소프트웨어 공급망 투명성을 높이기 위한 소프트웨어 자재 명세서 활용이 확대되었고, 2025년에는 국제 공조 차원의 공통 방향도 제시되었다.
- 제로 트러스트는 더 이상 구호가 아니라 운영 원칙으로 자리 잡아, 내부망이라도 자동 신뢰하지 않고 요청마다 검증하는 구조가 표준에 가까워지고 있다.
- 인공지능 시스템 자체를 보호하는 보안이 커졌으며, 학습 데이터 보호, 모델 배포 보안, 위협 정보 공유 절차를 다루는 공식 지침이 2024년과 2025년에 이어졌다.
- 다중 요소 인증도 아무 방식이나 쓰는 시대에서 피싱 저항성이 높은 방식으로 옮겨 가고 있어, 문자 메시지보다 기기 기반 인증이 더 선호된다.
- 오픈소스 핵심 구성요소의 위험 관리는 단순 라이선스 확인을 넘어 취약점, 유지관리 상태, 의존성 깊이까지 보는 방향으로 발전하고 있다.
- 결국 최근 흐름의 공통점은 사람의 실수를 사후 탐지로만 막기보다, 언어·도구·배포 체계 자체를 바꿔 취약점이 생기기 어렵게 만드는 데 있다.
