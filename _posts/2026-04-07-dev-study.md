## 핵심 요약

자료구조는 데이터를 어떤 형태로 저장하고 꺼낼지 정하는 규칙이며, 같은 문제라도 어떤 구조를 선택하느냐에 따라 속도와 메모리 사용량이 크게 달라집니다. 학부 수준에서는 배열, 연결 리스트, 스택, 큐, 트리, 해시 표를 중심으로 각 구조의 삽입, 삭제, 탐색 비용을 비교하는 것이 핵심입니다. 실제 적용에서는 단순히 이론적 시간복잡도만 보는 것이 아니라 캐시 친화성, 구현 난이도, 데이터 변경 빈도까지 함께 판단해야 합니다. 예를 들어 탐색이 많으면 해시 표나 균형 트리가 유리하고, 순차 접근이 많으면 배열이 더 단순하고 빠를 수 있습니다. 최근 흐름에서는 대규모 메모리 처리, 병렬 환경, 압축 저장, 인공지능 학습용 데이터 처리에 맞춘 자료구조 설계가 더 중요해지고 있습니다. 따라서 자료구조 학습은 시험용 암기보다 문제의 접근 패턴을 읽고 적절한 저장 모델을 고르는 훈련으로 이해하는 것이 좋습니다.

## 문제의 접근 패턴으로 자료구조를 고르는 기준

- 자료구조 선택의 첫 기준은 무엇을 가장 자주 하는지 보는 것이며, 검색 중심인지 삽입 중심인지에 따라 적합한 구조가 달라집니다.
- 순차적으로 데이터를 훑는 일이 많다면 배열이 단순하고 빠르며, 예를 들어 성적 목록 전체 출력은 배열이 잘 맞습니다.
- 중간 삽입과 삭제가 매우 잦다면 연결 리스트가 후보가 되지만, 임의 위치 접근은 느려 배열보다 불리합니다.
- 최근에 넣은 값을 먼저 꺼내야 하면 스택이 자연스럽고, 함수 호출 기록이나 괄호 검사에서 자주 쓰입니다.
- 먼저 들어온 값을 먼저 처리해야 하면 큐가 적절하며, 프린터 작업 대기열이 대표적인 예시입니다.
- 키로 빠르게 찾는 작업이 많다면 해시 표가 강력하지만, 정렬 상태를 유지해야 한다면 트리가 더 적합합니다.
- 실무에서는 이론적 성능만 보지 않고 구현 복잡도와 디버깅 비용도 같이 보며, 작은 데이터에서는 단순한 배열이 오히려 더 낫습니다.

## 배열과 연결 리스트의 저장 방식 차이 읽기

- 배열은 메모리에 연속적으로 저장되는 구조라서 인덱스로 바로 접근할 수 있고, 그래서 특정 위치 읽기가 매우 빠릅니다.
- 연결 리스트는 각 원소가 다음 원소의 위치를 함께 저장하므로 메모리가 흩어져 있어 순차적으로 따라가야 합니다.
- 배열은 중간에 값을 끼워 넣을 때 뒤 원소들을 밀어야 하므로 비용이 크지만, 연결 리스트는 연결만 바꾸면 되어 상대적으로 유리합니다.
- 반대로 배열은 캐시에 잘 맞아 실제 실행 속도가 좋은 경우가 많고, 연결 리스트는 포인터 추적 때문에 체감 성능이 떨어질 수 있습니다.
- 학생들이 자주 헷갈리는 점은 삽입이 빠르다는 말이 '위치를 이미 알고 있을 때' 기준이라는 것이며, 위치 탐색 비용은 별도입니다.
- 예를 들어 재생 목록 맨 앞 삭제가 반복되면 연결 리스트가 편하지만, 천만 개 중 500번째 값 읽기는 배열이 압도적으로 낫습니다.
- 실무 해석으로 보면 읽기 중심 데이터는 배열 계열, 구조 변경 중심 데이터는 연결 구조를 먼저 검토하는 식으로 접근합니다.

## 스택과 큐로 흐름 제어 문제 풀기

- 스택은 가장 나중에 들어간 데이터가 먼저 나오는 구조로, 접시를 쌓았다가 위에서 꺼내는 상황에 비유할 수 있습니다.
- 큐는 가장 먼저 들어간 데이터가 먼저 나오는 구조로, 줄 서기와 비슷하며 작업 순서를 보존할 때 적합합니다.
- 스택은 함수 호출, 되돌리기 기능, 수식 계산에 쓰이며, 특히 괄호 짝 검사에서는 여는 괄호를 쌓아 두는 방식이 전형적입니다.
- 큐는 너비 우선 탐색, 운영 요청 처리, 메시지 대기열 같은 흐름 제어 문제에서 자주 등장합니다.
- 스택과 큐는 둘 다 삽입과 삭제 위치가 제한된다는 점에서 배열보다 규칙이 강하지만, 그래서 오히려 문제 모델링이 쉬워집니다.
- 구현은 배열이나 연결 리스트 둘 다 가능하지만, 원형 배열 큐는 앞부분 빈칸을 재활용할 수 있어 메모리 낭비를 줄입니다.
- 실제 코딩 시험에서는 '최근 기록'이면 스택, '도착 순서 처리'이면 큐라는 신호를 먼저 읽는 습관이 중요합니다.

## 트리와 힙에서 계층 구조를 이해하기

- 트리는 부모와 자식 관계로 데이터를 표현하는 구조이며, 폴더 구조나 조직도처럼 계층을 나타낼 때 자연스럽습니다.
- 이진 탐색 트리는 왼쪽은 작은 값, 오른쪽은 큰 값 규칙을 이용해 탐색 범위를 줄이며, 정렬된 탐색에 유리합니다.
- 하지만 한쪽으로 치우치면 연결 리스트처럼 느려질 수 있어, 균형 유지 여부가 실제 성능을 좌우합니다.
- 힙은 완전 이진 트리 기반으로 최댓값이나 최솟값을 빠르게 꺼내는 데 특화되어 있으며, 우선순위 큐 구현에 널리 쓰입니다.
- 이진 탐색 트리와 힙은 둘 다 트리지만 목적이 다르며, 전자는 빠른 검색, 후자는 우선순위 처리에 초점이 있습니다.
- 예를 들어 응급실 환자 우선 처리나 작업 스케줄링은 힙이 잘 맞고, 범위 검색은 탐색 트리가 더 적절합니다.
- 실무적으로는 '정렬된 상태로 찾기'와 '가장 큰 값 하나를 반복 추출하기'를 구분해야 올바른 구조를 고를 수 있습니다.

## 해시 표의 평균 성능과 충돌 처리 해석

- 해시 표는 키를 해시 함수로 계산한 위치에 저장해 평균적으로 매우 빠른 검색을 제공하는 구조입니다.
- 예를 들어 학번으로 학생 정보를 찾는 문제에서 배열보다 메모리를 덜 낭비하면서도 빠른 접근이 가능합니다.
- 핵심 위험은 서로 다른 키가 같은 위치로 가는 충돌이며, 이를 연결 방식이나 개방 주소 방식으로 처리합니다.
- 연결 방식은 한 칸에 여러 원소를 묶어 두고, 개방 주소 방식은 빈 칸을 찾아 이동시키는 방식이라 삭제 처리에서 차이가 큽니다.
- 해시 표는 평균적으로 빠르지만 최악의 경우 느려질 수 있으므로, '항상 빠르다'고 외우면 오해가 생깁니다.
- 정렬 순서를 유지하지 않기 때문에 범위 검색이나 사전식 출력은 불리하며, 이런 경우 균형 트리와 비교해야 합니다.
- 실무에서는 해시 함수 품질, 적재율, 재해시 비용까지 고려해야 하며 대량 데이터에서는 이 설정이 체감 성능에 직접 영향을 줍니다.

## 시간복잡도 표를 그대로 믿으면 틀리는 이유

- 시간복잡도는 입력 크기가 커질 때 연산량이 어떻게 늘어나는지 보여 주는 지표이지만, 실제 실행 시간과 완전히 같지는 않습니다.
- 예를 들어 배열 탐색과 연결 리스트 탐색은 둘 다 선형 시간이지만, 배열은 캐시 효율 덕분에 실제로 더 빠른 경우가 많습니다.
- 빅오 표기법은 상수항과 하드웨어 차이를 숨기므로, 작은 입력에서는 복잡도가 더 좋아도 실제로 이득이 없을 수 있습니다.
- 최선, 평균, 최악 경우를 구분해야 하며, 해시 표는 평균은 빠르지만 충돌이 심하면 최악 성능이 나빠질 수 있습니다.
- 암화 분석은 여러 연산을 묶어 평균 비용을 보는 방식으로, 동적 배열의 크기 확장 비용을 해석할 때 자주 등장합니다.
- 비교 기준도 중요해서 탐색 한 번의 속도만 볼지, 삽입과 삭제까지 포함한 전체 작업량을 볼지에 따라 결론이 달라집니다.
- 실무적 해석으로는 복잡도 표를 출발점으로 삼되, 데이터 크기와 접근 패턴을 기준으로 실험까지 해야 올바른 선택이 됩니다.

## 시험과 구현에서 자주 틀리는 함정 정리

- 배열 삽입이 느리다고 할 때는 보통 중간 삽입을 뜻하며, 맨 끝 추가는 동적 배열에서 평균적으로 빠를 수 있습니다.
- 연결 리스트는 삭제가 빠르지만 삭제할 노드를 찾는 과정까지 포함하면 전체 비용이 커질 수 있다는 점을 자주 놓칩니다.
- 스택과 큐 문제에서 자료를 꺼내는 순서를 반대로 이해하면 구현은 맞아 보여도 결과가 완전히 달라집니다.
- 트리 순회에서는 전위, 중위, 후위의 방문 시점을 혼동하기 쉬우며, 식 계산과 파일 삭제 같은 예시로 연결해 기억하면 덜 틀립니다.
- 해시 표는 충돌이 없다고 가정하면 안 되며, 시험에서도 충돌 처리 방식에 따라 결과 배열이 달라질 수 있습니다.
- 힙은 전체 정렬 구조가 아니라 부모와 자식 사이의 우선순위만 보장하므로, 임의 원소 탐색에는 적합하지 않습니다.
- 구현 실수로는 인덱스 범위 오류, 빈 구조에서 삭제 시도, 머리 노드 갱신 누락이 많아서 작은 예제로 손으로 추적해 보는 연습이 필요합니다.

## 최신 흐름

- 대규모 데이터 처리에서는 단순 연결 리스트보다 캐시 친화적인 연속 저장 구조가 더 선호되며, 실제 성능 차이가 이론보다 크게 나타납니다.
- 인공지능 학습 데이터 파이프라인에서는 배치 단위 접근이 많아 배열 기반 구조와 압축 표현의 중요성이 커지고 있습니다.
- 병렬 처리 환경에서는 여러 스레드가 동시에 접근해도 안전한 동시성 큐와 락 경감 구조가 점점 더 많이 연구되고 활용됩니다.
- 그래프와 행렬 데이터를 다룰 때는 희소 표현을 이용해 메모리를 줄이는 방식이 널리 쓰이며, 추천 시스템과 과학 계산에서 특히 중요합니다.
- 저장 장치 속도 격차를 줄이기 위해 메모리 안 구조와 디스크 친화 구조를 함께 고려하는 설계가 강조되고 있습니다.
- 대규모 서비스에서는 해시 표도 단일 기계 기준이 아니라 분산 해시 관점으로 확장해 위치 결정과 부하 분산을 함께 봅니다.
- 압축과 검색을 동시에 고려한 비트 벡터, 블룸 필터 같은 확률 기반 구조가 빠른 전처리와 필터링에 자주 도입됩니다.
- 실시간 분석 시스템에서는 삽입과 삭제가 연속적으로 일어나므로 고정 구조보다 동적 확장과 메모리 재사용 전략이 중요해지고 있습니다.
- 하드웨어 구조를 반영한 자료구조 설계가 늘어나면서 단순한 점근적 분석보다 캐시, 분기 예측, 메모리 정렬 효과를 함께 평가합니다.
- 교육 측면에서도 단순 암기보다 실제 측정 실험을 통해 배열, 트리, 해시 표의 성능을 비교하는 방식이 더 강조되는 흐름입니다.
