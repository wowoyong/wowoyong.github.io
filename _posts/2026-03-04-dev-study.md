## 핵심 요약

컴파일러는 사람이 작성한 고급 언어 코드를 기계가 실행할 수 있는 형태로 바꾸는 번역 체계이며, 어휘 분석부터 코드 생성까지 이어지는 단계별 처리 흐름을 이해하는 것이 핵심입니다. 학부 수준에서는 각 단계의 입력과 출력, 자료구조의 활용 방식, 오류 처리 전략을 서로 연결해 이해하는 것이 중요합니다. 실제로는 빠른 빌드 시간, 이해하기 쉬운 오류 메시지, 높은 최적화 품질이 개발 생산성과 프로그램 성능에 직접 영향을 줍니다. 최근에는 안전한 언어 구현, 웹어셈블리 대상 코드 생성, 여러 단계의 중간 표현을 활용한 최적화가 실무와 연구에서 함께 중요해지고 있습니다. 따라서 컴파일러 학습은 단순히 번역기를 이해하는 데 그치지 않고, 프로그래밍 언어 설계와 성능 최적화, 개발 도구 제작 역량까지 함께 기르는 기초 과목이라고 볼 수 있습니다.

## 번역기 전체 흐름과 단계별 입력·출력

- 컴파일러는 소스 코드를 여러 단계로 나누어 처리하며, 한 번에 모두 바꾸는 방식보다 단계별 분리가 디버깅과 유지보수에 더 유리합니다.
- 어휘 분석은 문자 흐름을 의미 있는 최소 단위로 나누는 과정으로, 예를 들어 `if(x)`를 예약어, 괄호, 식별자로 구분해 다음 단계의 부담을 줄입니다.
- 구문 분석은 문법 규칙에 맞는 구조를 트리 형태로 만들며, 같은 토큰열이라도 괄호 위치에 따라 연산 우선순위가 달라진다는 점을 분명하게 보여 줍니다.
- 의미 분석은 타입, 선언 여부, 범위를 검사하는 단계로, 문법상 맞는 코드라도 선언되지 않은 변수를 사용하면 이 단계에서 오류가 발생합니다.
- 중간 표현 생성은 기계어 직전의 공통 표현을 만드는 과정으로, 특정 하드웨어에 덜 얽매여 여러 대상 코드로 확장하기 쉽습니다.
- 최적화는 같은 결과를 더 적은 비용으로 내도록 바꾸는 작업이며, 불필요한 계산을 없애는 방식으로 실행 시간과 코드 크기를 함께 줄일 수 있습니다.
- 코드 생성은 최종 기계 명령으로 바꾸는 단계이며, 앞선 단계의 품질이 낮으면 이 단계에서 아무리 잘 만들어도 성능 한계가 남습니다.

## 어휘 분석기 설계와 토큰 분리 기준

- 어휘 분석기는 정규식 기반 규칙으로 토큰을 나누고, 공백과 주석을 제거해 이후 단계가 핵심 구조만 보도록 만듭니다.
- 식별자, 숫자, 연산자, 예약어를 구분하는 기준은 언어 설계와 직결되며, 예를 들어 `while`은 이름처럼 보여도 예약어로 고정됩니다.
- 가장 긴 일치를 우선하는 규칙을 사용하면 `==`를 `=` 두 개로 잘못 나누는 문제를 막을 수 있어 실제 구현에서 매우 중요합니다.
- 토큰에 위치 정보(줄, 칸)를 함께 저장하면 오류 메시지를 정확하게 표시할 수 있어 사용자 경험이 크게 좋아집니다.
- 초기 단계에서 심벌 테이블과 연결하면 예약어를 빠르게 판별할 수 있으며, 단순 반복 비교보다 해시 기반 조회가 더 효율적입니다.
- 숫자 리터럴을 처리할 때는 정수와 실수를 구분해야 하며, `12.34`와 `12`는 이후 타입 추론 결과도 달라집니다.
- 어휘 단계의 오류는 가능한 한 빨리 잡는 편이 좋지만, 한 글자 문제로 전체 분석을 멈추지 않고 복구할 수 있어야 도구의 완성도가 높아집니다.

## 구문 트리 구성과 문법 충돌 해석

- 구문 분석기는 토큰열이 문법에 맞는지 확인하고 트리를 만들며, 이 트리는 이후 의미 분석의 중심 자료가 됩니다.
- 왼쪽에서 오른쪽으로 읽는 하향식 방식은 이해와 구현이 쉬워 학부 실습에서 자주 쓰이며, 손으로 추적하기도 편합니다.
- 상향식 방식은 더 넓은 문법을 다루기에 좋지만 표 기반 해석이 복잡하므로, 처음 배울 때는 상태 전이를 꼼꼼히 살펴야 합니다.
- 연산자 우선순위와 결합 방향은 `a-b-c` 같은 식의 해석에 직접 영향을 주므로, 같은 문법이라도 결과 트리가 달라질 수 있습니다.
- 왼쪽 재귀 제거는 하향식 분석기 구현에서 자주 필요하며, 이를 처리하지 않으면 재귀 호출이 끝나지 않는 문제가 생깁니다.
- 구문 오류 복구는 세미콜론 같은 동기화 지점을 활용해 다음 문장부터 계속 읽도록 만들 수 있어, 실무용 컴파일러에서 필수적인 기능입니다.
- 추상 구문 트리는 불필요한 문법 기호를 덜어 낸 구조이므로, 실제 최적화와 코드 생성에서는 원시 구문 트리보다 더 실용적입니다.

## 의미 분석에서 타입·범위·심벌 처리

- 의미 분석은 문법적으로 맞는 코드가 실제로도 타당한지 검사하는 단계이며, 선언 전에 사용하는 경우 같은 문제를 잡아냅니다.
- 심벌 테이블은 변수, 함수, 타입 정보를 저장하는 사전 역할을 하며, 블록이 깊어질수록 범위 관리가 핵심이 됩니다.
- 정적 타입 검사는 실행 전에 많은 오류를 줄여 주며, 예를 들어 정수 변수에 문자열을 더하려 하면 이 단계에서 막힙니다.
- 함수 호출 검사는 인자의 개수와 타입을 비교하며, 오버로딩이 있는 언어에서는 같은 이름이라도 서명까지 함께 확인해야 합니다.
- 형 변환은 자동 변환과 명시적 변환을 구분해야 하며, 편의성을 높이면서도 예상하지 못한 값 손실은 줄여야 합니다.
- 제어 흐름 검사를 통해 반환 누락이나 도달할 수 없는 코드 등을 찾을 수 있으며, 이는 논리 오류를 조기에 드러내는 실무적 장점이 있습니다.
- 의미 분석이 탄탄하면 이후 최적화도 더 안전해지며, 타입 정보가 있어야 어떤 연산을 바꾸어도 결과가 유지되는지 판단하기 쉽습니다.

## 중간 표현과 최적화 전략의 연결

- 중간 표현은 소스 언어와 대상 기계 사이를 잇는 공통 다리이며, 여러 언어와 여러 기계를 연결할 때 재사용성이 높습니다.
- 세 주소 코드 같은 단순한 표현은 사람이 읽기 쉬워 교육용으로 적합하고, 복잡한 최적화 원리를 단계별로 확인하기에도 편합니다.
- 정적 단일 대입 형태는 변수의 흐름을 분명하게 보여 주어 상수 전파와 죽은 코드 제거 같은 최적화에 특히 유리합니다.
- 상수 접기는 `2+3`을 미리 `5`로 바꾸는 방식으로, 실행 중 계산을 줄이기 때문에 작은 규칙이라도 전체 성능에 누적 효과를 냅니다.
- 공통 부분식 제거는 같은 계산을 반복하지 않도록 해 주며, 루프 내부에서 효과가 커서 실제 실행 시간 차이가 크게 날 수 있습니다.
- 루프 최적화는 자주 실행되는 구간을 다루므로 중요도가 높고, 루프 밖으로 뺄 수 있는 계산을 이동시키는 기법이 대표적입니다.
- 최적화는 무조건 많이 한다고 좋은 것이 아니며, 컴파일 시간 증가와 디버깅의 어려움까지 함께 따져야 실무적인 판단이 됩니다.

## 목적 코드 생성과 실행 성능의 현실적 균형

- 코드 생성기는 중간 표현을 실제 기계 명령으로 바꾸며, 같은 알고리즘이라도 명령 선택에 따라 속도와 크기가 달라집니다.
- 레지스터 할당은 자주 쓰는 값을 빠른 저장 공간에 두는 과정으로, 공간이 부족하면 메모리 접근이 늘어나 성능이 크게 떨어집니다.
- 명령 선택 단계에서는 하나의 연산을 여러 방식으로 구현할 수 있으며, 대상 구조의 특수 명령을 잘 활용하면 코드 길이를 줄일 수 있습니다.
- 명령 스케줄링은 서로 독립적인 명령의 순서를 조정해 대기 시간을 줄이는 기법으로, 파이프라인 구조를 가진 처리기에서 특히 중요합니다.
- 디버깅하기 쉬운 코드와 최고 성능의 코드는 자주 충돌하며, 지나친 최적화는 소스 줄과 실행 위치의 대응을 어렵게 만듭니다.
- 이식성을 중시하면 공통 기능 위주로 생성하고, 최고 성능을 원하면 특정 구조에 맞춘 별도 경로를 두는 방식이 자주 쓰입니다.
- 실무에서는 절대적인 최고 성능보다 예측 가능한 빌드 시간과 안정성도 중요하므로, 목적 코드 생성은 균형 잡힌 설계가 핵심입니다.

## 최신 흐름

- 여러 단계의 중간 표현 활용이 늘어나면서, 고수준 최적화와 저수준 최적화를 하나의 표현에 모두 담기보다 단계별로 나누어 처리하는 흐름이 강해졌습니다.
- 웹어셈블리 대상 컴파일이 확대되면서 브라우저뿐 아니라 서버, 플러그인, 경량 실행 환경까지 컴파일러의 적용 범위가 넓어지고 있습니다.
- 메모리 안전성을 중시하는 언어 구현이 주목받으면서, 소유권 검사나 엄격한 타입 검사를 컴파일 단계에서 강화하는 사례가 많아졌습니다.
- 빠른 빌드 시간을 위한 경량 백엔드 수요가 커지면서, 최고 수준의 최적화보다 개발 반복 속도를 우선하는 설계가 실무에서 중요해졌습니다.
- 증분 컴파일이 보편화되면서 일부 파일만 다시 분석하고 생성하는 방식이 대형 프로젝트의 생산성을 좌우하는 핵심 기능이 되었습니다.
- 오류 메시지의 품질 경쟁이 치열해지면서, 단순한 문법 오류 표시를 넘어 원인 추정과 수정 방향 제시까지 포함하는 컴파일러가 늘고 있습니다.
- 언어 서버와 컴파일러 프런트엔드의 결합이 강화되면서, 편집기의 자동 완성과 실시간 진단이 같은 분석 정보를 공유하는 구조가 일반화되고 있습니다.
- 이기종 구조 지원의 중요성이 커지면서, 중앙처리장치뿐 아니라 그래픽 처리 장치와 특수 가속기를 겨냥한 중간 표현 확장이 활발해지고 있습니다.
- 최적화에서도 코드 크기 감소가 다시 중요해지면서, 모바일·엣지 환경에서는 최고 속도보다 배포 크기와 시작 지연 감소가 우선되는 경우가 많습니다.
- 생성형 인공지능 코딩 도구가 널리 퍼지면서, 컴파일러는 잘못 만들어진 코드의 의미 오류를 빠르게 잡아 내고 안전장치 역할을 하는 방향으로 더욱 주목받고 있습니다.
