## 핵심 요약

컴퓨터 보안은 기밀성, 무결성, 가용성을 지키기 위해 시스템과 데이터를 보호하는 학문이다. 학부 과정에서는 암호화, 인증, 접근통제, 네트워크 방어, 취약점 분석, 악성코드 대응, 안전한 개발 절차를 함께 이해해야 전체 그림이 잡힌다. 실제 적용에서는 이론만 아는 것보다 어떤 공격이 어떤 방어를 우회하는지, 그리고 비용과 편의성 사이에서 어떻게 균형을 잡는지가 중요하다. 예를 들어 강한 인증을 도입해도 세션 탈취나 잘못된 권한 설정이 있으면 침해를 막지 못하므로 계층형 방어가 필요하다. 2026년 초 기준으로는 신원 중심 보안, 비밀번호 없는 로그인 확산, 공급망 보안 강화, 양자내성 암호 전환 준비, 인공지능 기반 탐지와 공격 대응이 중요한 흐름으로 자리 잡고 있다.

## 기밀성·무결성·가용성으로 보는 보안 목표

- 기밀성은 허가된 사람만 정보를 보게 하는 성질로, 학생 성적 파일을 암호화해 외부 유출을 막는 상황이 대표적이다.
- 무결성은 데이터가 허가 없이 바뀌지 않았음을 뜻하며, 송금 금액이 전송 중 변조되지 않았는지 확인하는 전자서명이 좋은 예시다.
- 가용성은 필요할 때 서비스를 쓸 수 있는 성질로, 분산 서비스 거부 공격이 가용성을 직접 노리는 전형적인 공격이다.
- 세 목표는 자주 충돌하는데, 매우 강한 검증 절차는 보안을 높이지만 접속 지연을 늘려 가용성을 떨어뜨릴 수 있다.
- 실무에서는 세 목표를 동시에 최대화하기보다 자산의 중요도에 따라 우선순위를 다르게 둔다. 예를 들어 병원 시스템은 가용성이 특히 중요하다.
- 위협은 자연재해, 사용자 실수, 내부자 남용, 외부 공격처럼 다양하므로 해킹만 막는다고 보안이 끝나지 않는다.
- 보안 정책은 기술의 상위 개념이다. 같은 암호화 기술을 써도 백업 정책이나 권한 정책이 부실하면 실제 보호 수준은 낮다.

## 암호화와 전자서명이 실제 서비스에서 쓰이는 방식

- 대칭키 암호는 같은 열쇠로 암호화와 복호화를 처리해 빠르며, 대용량 파일 보호에 적합하다는 점에서 공개키 방식과 구별된다.
- 공개키 암호는 공개키와 비밀키를 나눠 쓰므로 키 배포가 쉬워 인터넷 통신에 유리하지만 계산량이 더 크다.
- 현실의 웹 서비스는 보통 공개키 방식으로 안전하게 비밀키를 교환한 뒤, 실제 데이터 전송은 빠른 대칭키 방식으로 처리한다.
- 해시 함수는 원문을 짧은 고정 길이 값으로 바꾸며, 비밀번호를 그대로 저장하지 않고 해시값으로 저장하는 이유가 여기에 있다.
- 전자서명은 '누가 보냈는지'와 '내용이 안 바뀌었는지'를 함께 확인하게 해 주므로 단순 암호화와 목적이 다르다.
- 인증서와 공개키 기반 구조는 공개키가 정말 해당 서버의 것인지 보증하는 체계로, 없으면 중간자 공격에 취약해진다.
- 실무적으로 중요한 점은 알고리즘 이름보다 키 관리다. 강한 암호를 써도 비밀키를 깃 저장소에 올리면 보안은 무너진다.

## 인증·인가·접근통제를 구분해서 이해하기

- 인증은 '누구인지 확인'하는 과정이고, 인가는 '무엇을 할 수 있는지 결정'하는 과정이므로 로그인 성공과 관리자 권한 부여는 별개다.
- 비밀번호만 쓰는 방식은 구현이 쉽지만 피싱과 재사용 공격에 약해, 추가 인증 수단을 결합한 다중요소 인증이 널리 쓰인다.
- 문자 메시지 기반 인증은 예전보다 약한 방식으로 평가되며, 피싱 저항성이 높은 보안키나 기기 기반 인증이 더 안전한 대안으로 본다.
- 세션 관리는 로그인 뒤의 신뢰를 유지하는 장치인데, 세션 토큰이 탈취되면 비밀번호를 몰라도 사용자처럼 행동할 수 있다.
- 역할 기반 접근통제는 직무별 권한 관리가 쉬워 기업에서 많이 쓰이고, 속성 기반 접근통제는 위치·시간·기기 상태까지 반영해 더 세밀하다.
- 최소 권한 원칙은 필요한 권한만 주는 방식으로, 개발 서버 계정이 운영 데이터베이스 삭제 권한까지 갖지 않게 하는 것이 대표적이다.
- 실무 해석으로 보면 로그인 화면 보안보다 권한 분리와 권한 검토가 더 큰 사고를 막는 경우가 많다. 내부 계정 오남용이 실제로 자주 문제 된다.

## 네트워크 공격을 패킷 흐름 관점에서 파악하기

- 도청 공격은 전송 중 데이터를 몰래 읽는 것이고, 변조 공격은 내용을 바꾸는 것이므로 둘 다 막으려면 암호화와 무결성 검증이 함께 필요하다.
- 중간자 공격은 통신 사이에 끼어들어 내용을 가로채거나 바꾸는 방식으로, 서버 인증서 검증을 무시하면 쉽게 당할 수 있다.
- 재전송 공격은 정상 메시지를 다시 보내 부정 동작을 유도하는 기법이며, 일회용 값이나 시간 정보를 넣어 방지한다.
- 분산 서비스 거부 공격은 서버를 마비시키는 공격으로, 기밀성보다 가용성을 노린다는 점에서 비밀번호 탈취와 성격이 다르다.
- 가짜 무선 접속 지점을 만드는 공격은 사용자가 정상 망이라고 착각하게 하므로, 공용 와이파이 환경에서는 자동 접속을 조심해야 한다.
- 방화벽은 흐름을 통제하고, 침입탐지는 이상 징후를 찾고, 침입방지는 탐지 뒤 차단까지 수행한다는 점에서 역할 차이가 있다.
- 현업에서는 네트워크 경계 장비만 믿지 않고 종단 장치, 신원, 응용 계층 로그까지 함께 분석해야 침해 원인을 정확히 찾는다.

## 버퍼 오버플로·주입 공격으로 배우는 취약점 원리

- 버퍼 오버플로는 할당된 메모리 범위를 넘겨 쓰는 취약점으로, 오래된 저수준 언어 프로그램에서 특히 자주 등장했다.
- 주입 공격은 프로그램이 데이터를 코드처럼 처리할 때 발생하며, 데이터베이스 질의문 조작이 대표적 예시다.
- 교차 사이트 스크립트 공격은 웹페이지에 악성 스크립트를 심어 다른 사용자의 세션이나 입력을 탈취하게 만든다.
- 교차 사이트 요청 위조는 사용자가 로그인된 상태를 악용해 원치 않는 요청을 보내게 하며, 스크립트 삽입과는 공격 경로가 다르다.
- 경로 조작 취약점은 파일 경로 입력을 악용해 의도하지 않은 파일에 접근하는 문제로, 백업 파일 노출 사고와 연결되기 쉽다.
- 메모리 안전 언어, 입력 검증, 출력 인코딩, 준비된 질의문, 권한 분리는 각기 다른 취약점에 대응하므로 한 가지 기법으로 모두 해결되지는 않는다.
- 실무에서는 취약점 개수보다 공격 가능 경로가 중요하다. 낮은 심각도 취약점도 관리자 권한 획득과 연결되면 위험도가 크게 올라간다.

## 악성코드와 침해 대응을 단계별로 이해하기

- 바이러스는 다른 파일에 붙어 퍼지고, 웜은 스스로 전파되며, 트로이목마는 정상 프로그램처럼 속��다는 점에서 전파 방식이 다르다.
- 랜섬웨어는 파일 암호화만이 아니라 데이터 유출 뒤 협박까지 결합하는 경우가 많아, 단순 복구만으로 끝나지 않는다.
- 정보탈취 악성코드는 비밀번호, 세션 쿠키, 지갑 정보 같은 자격 증명을 훔쳐 이후 침입의 출발점이 된다.
- 공격자는 정찰, 초기 침입, 권한 상승, 내부 이동, 데이터 유출 같은 단계를 밟는 경우가 많아, 대응도 단계별 관찰이 중요하다.
- 침해 대응은 탐지, 분석, 격리, 제거, 복구, 재발 방지 순서로 진행하는 것이 일반적이며, 격리를 늦추면 피해가 확산된다.
- 로그 보존과 시간 동기화는 포렌식의 기본이다. 기록 시각이 뒤섞이면 공격 순서를 재구성하기 매우 어렵다.
- 실무적으로는 백신 하나보다 백업, 계정 보호, 네트워크 분리, 복구 훈련이 실제 피해 규모를 줄이는 데 더 직접적이다.

## 안전한 개발과 운영에서 보안이 앞당겨지는 이유

- 보안을 개발 마지막 점검으로만 두면 수정 비용이 커지므로, 설계 단계에서 위협 모델링을 수행하는 편이 훨씬 효율적이다.
- 안전한 기본값은 사용자가 따로 설정하지 않아도 비교적 안전한 상태로 시작하게 만드는 원칙으로, 기본 비밀번호 제공은 나쁜 예다.
- 코드 정적 분석은 실행 전 문제를 찾고, 동적 분석은 실제 실행 중 이상을 찾으므로 둘은 경쟁 관계보다 보완 관계에 가깝다.
- 의존 라이브러리 점검이 중요한 이유는 우리 코드가 안전해도 외부 패키지 취약점으로 전체 서비스가 뚫릴 수 있기 때문이다.
- 비밀정보 관리는 환경변수, 비밀 저장소, 접근 기록과 함께 다뤄야 하며, 설정 파일에 키를 하드코딩하는 방식은 운영 사고로 이어지기 쉽다.
- 보안 업데이트는 기능 추가보다 덜 눈에 띄지만, 알려진 취약점을 막는 가장 비용 대비 효과가 큰 방법 중 하나다.
- 실무 해석으로 보면 좋은 보안 문화는 개발자 개인의 주의력에 기대지 않고, 점검 절차와 자동화 도구로 실수를 줄이는 방향에 가깝다.

## 최신 흐름

- 신원이 새로운 경계로 여겨지면서 네트워크 안쪽이라는 이유만으로 신뢰하지 않고, 사용자·기기·세션 상태를 계속 확인하는 방식이 강화되고 있다.
- 비밀번호 없는 로그인과 피싱 저항성 인증이 빠르게 확산되고 있으며, 보안키와 기기 기반 인증이 문자 메시지 인증보다 우선되는 흐름이다.
- 양자내성 암호 전환 준비가 본격화되면서, 조직들은 지금 쓰는 암호 알고리즘과 키 사용 위치를 먼저 목록화하는 작업부터 시작하고 있다.
- 공급망 보안이 핵심 의제로 올라와 외부 패키지, 빌드 서버, 서명 키, 협력사 계정까지 공격 표면으로 관리하는 추세가 강해졌다.
- 안전한 설계와 안전한 기본값이 제품 경쟁력으로 취급되며, 취약한 기본 설정을 고객 책임으로 돌리는 관행이 점점 줄고 있다.
- 인공지능은 공격 자동화에도 쓰이지만 방어 측에서는 이상 징후 탐지, 경보 우선순위화, 조사 보조에 적극 활용되는 방향으로 발전하고 있다.
- 클라우드 보안은 서버 자체보다 권한 설정, 저장소 공개, 서비스 간 연결, 토큰 관리 같은 구성 오류 중심으로 사고가 나는 비중이 커지고 있다.
- 데이터 탈취 뒤 협박하는 이중 갈취형 공격이 계속 늘어나 백업만으로는 충분하지 않고, 유출 탐지와 법적 대응 준비까지 함께 요구된다.
- 외부 노출 자산 관리가 중요해져 조직이 직접 기억하지 못하는 하위 도메인, 시험 서버, 오래된 응용 프로그램까지 지속적으로 찾는 운영이 늘고 있다.
- 규제와 표준의 영향력이 커지면서 보안은 선택 기능이 아니라 조달, 감사, 서비스 출시 조건으로 묶이는 방향으로 이동하고 있다.
