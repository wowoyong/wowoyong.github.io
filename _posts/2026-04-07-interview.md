## 오늘의 면접 주제
> OS 기본기인 메모리 관리 질문은 백엔드와 시스템 면접에서 매우 자주 나오기 때문입니다.

## 핵심 한 줄 요약
가상 메모리는 각 프로세스에 연속된 메모리 공간처럼 보이게 하면서, 실제 물리 메모리를 페이지 단위로 효율적으로 나눠 쓰는 방식입니다.

## 면접 질문 & 모범 답변

### Q1. 가상 메모리는 무엇인가요?
**기본 답변**: 가상 메모리는 프로세스가 실제 RAM보다 더 크고 연속된 메모리를 쓰는 것처럼 보이게 하는 추상화입니다. CPU가 보는 주소는 virtual address이고, MMU가 이를 physical address로 변환합니다. 덕분에 프로세스 격리와 메모리 효율이 좋아집니다.  
**심화 포인트**: 핵심은 "큰 메모리를 흉내 낸다"가 아니라, 보호와 재배치가 쉬워진다는 점입니다.

### Q2. 페이징은 어떻게 동작하나요?
**기본 답변**: 가상 메모리와 물리 메모리를 같은 크기의 page와 frame으로 나눕니다. page table은 "이 가상 페이지가 어느 물리 프레임에 있는지"를 기록합니다. 주소 변환은 page number와 offset을 나눠서 처리합니다.  
**심화 포인트**: offset은 그대로 유지됩니다. 그래서 변환은 "페이지 위치만 바꾸는 작업"에 가깝습니다.

### Q3. TLB는 왜 필요한가요?
**기본 답변**: page table을 매번 메모리에서 찾으면 주소 한 번 읽을 때마다 추가 메모리 접근이 생깁니다. TLB는 최근 주소 변환 결과를 저장하는 작은 캐시입니다. hit가 나면 page table을 다시 보지 않아도 되어 성능이 크게 좋아집니다.  
**심화 포인트**: 메모리 접근이 느린 이유는 연산 때문이 아니라, 메모리를 한 번 더 읽어야 하기 때문입니다.

### Q4. page fault가 발생하면 무슨 일이 일어나나요?
**기본 답변**: 필요한 페이지가 RAM에 없으면 page fault가 발생합니다. OS는 디스크나 swap 영역에서 해당 페이지를 읽어와 빈 frame에 적재합니다. 빈 공간이 없으면 기존 페이지 하나를 내보낸 뒤 다시 실행합니다.  
**심화 포인트**: page fault는 예외이지만, 잘못이 아닐 수도 있습니다. demand paging에서는 정상 동작의 일부입니다.

### Q5. swapping과 paging은 어떻게 다른가요?
**기본 답변**: paging은 페이지 단위로 필요한 조각만 올리고 내립니다. swapping은 전통적으로 프로세스 전체를 통째로 내보내는 개념에 가깝습니다. 실무에서는 보통 page 단위 교체가 중심입니다.  
**심화 포인트**: 면접에서는 두 용어를 섞어 쓰는 경우가 많지만, granularity 차이를 구분하면 좋습니다.

### Q6. 페이지 교체 알고리즘은 왜 중요한가요?
**기본 답변**: 메모리가 가득 찼을 때 어떤 페이지를 내보낼지 결정해야 하기 때문입니다. 이 선택이 잘못되면 page fault가 급격히 늘고 시스템이 느려집니다. 대표적으로 FIFO, LRU, Clock이 자주 나옵니다.  
**심화 포인트**: 좋은 알고리즘은 "앞으로 안 쓸 페이지"를 맞히는 것입니다. 하지만 미래를 모르니 과거 접근 패턴으로 추정합니다.

### Q7. LRU가 자주 언급되는 이유는 무엇인가요?
**기본 답변**: 최근에 사용한 페이지는 다시 사용할 가능성이 높다는 locality 가정을 쓰기 때문입니다. 실제 프로그램은 반복문, 함수 호출, 배열 순회처럼 가까운 데이터를 다시 보는 경우가 많습니다. 그래서 LRU는 직관적이고 설명하기 좋습니다.  
**심화 포인트**: 완전한 LRU는 구현 비용이 큽니다. 그래서 실제 OS는 Clock 같은 근사 알고리즘을 더 자주 씁니다.

### Q8. thrashing은 무엇이고 왜 위험한가요?
**기본 답변**: 프로세스 실행보다 페이지 교체에 더 많은 시간을 쓰는 상태입니다. RAM이 부족한데 working set을 다 담지 못하면 발생합니다. CPU 사용률은 낮은데 디스크 I/O만 많아지는 이상한 상황이 생깁니다.  
**심화 포인트**: 단순히 메모리가 적어서가 아니라, "현재 자주 쓰는 페이지 집합"을 유지하지 못해서 생깁니다.

## 헷갈리기 쉬운 포인트
TLB miss는 변환 캐시 실패입니다. page fault는 아예 해당 페이지가 RAM에 없는 상태입니다.  
virtual memory와 swap space는 같은 말이 아닙니다. swap은 가상 메모리를 구현하는 한 방법입니다.  
LRU가 이론적으로 좋아도 항상 실제 OS의 최종 선택은 아닙니다. 구현 비용과 오버헤드도 같이 봐야 합니다.

## 면접관 시각
면접관은 정의를 외웠는지보다, 주소 변환 흐름을 자연스럽게 설명하는지 봅니다. TLB, page fault, page replacement를 하나의 흐름으로 연결하면 이해도가 높아 보입니다. 실무 관점에서는 "왜 느려지는가"를 locality, I/O, working set으로 설명할 수 있는지를 중요하게 봅니다.
